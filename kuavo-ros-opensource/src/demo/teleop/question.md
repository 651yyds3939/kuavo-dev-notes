# Teleop 动捕遥操作问题记录

本文档按时间线整理 `teleop_math_solver.py` 与 `teleop_geometric_ik_fusion.py` 开发调试过程中遇到的问题、原因分析与最终解法。  
当前推荐使用 **`teleop_math_solver.py`（3D 球坐标版）**；IK 融合版仍不建议作为主方案。

---

## 阶段一：Math 版 vs IK 版 —— 为什么 IK 不稳定？

### 现象
- `teleop_math_solver.py`（3D math）大部分动作可用
- `teleop_geometric_ik_fusion.py`（IK 版）抖动大、易失败、跟手差

### 原因
| 问题 | 说明 |
|------|------|
| 坐标系未对齐 | MediaPipe 世界坐标（髋部原点、Y 向下、Z 朝相机）与机器人 `base_link`（X 前 Y 左 Z 上）不一致，直接把手腕位移当 IK 目标会偏 |
| 肩膀锚点错误 | IK 版使用 `SHOULDER_R/L = [0.15, ±0.22, 0.12]`，URDF 实际约为 `[-0.0175, ±0.2927, 0.4245]` |
| 仅 6 关节 vs 7DOF+IK | Math 版直接映射关节角；IK 版要求精确 3D 末端位姿，对动捕噪声极敏感 |
| 50Hz 同步阻塞 IK 服务 | 每帧 `wait_for_service` 等待求解，延迟抖动大 |
| 缺官方管线 | Quest3 官方有肩宽自适应、臂长缩放、卡尔曼滤波等，手机单目 + 简易映射无法替代 |

### 结论
IK 不稳定主要是**前端坐标映射 + 调用方式**问题，不是 Kuavo IK 服务本身不可用。动捕遥操作优先用 **关节空间直接映射（math 版）**。

## 阶段二：左右手镜像反了

### 现象
动左手，机器人动右手。

### 原因
在 `cv2.flip` **翻转后的画面**上做 MediaPipe 检测，3D 世界坐标 X 轴被镜像，左右映射错乱。

### 解法
```python
# 检测用原始画面
results = self.pose.process(cv2.cvtColor(frame, cv2.COLOR_BGR2RGB))
# 仅显示时镜像
cv2.imshow(..., cv2.flip(frame, 1))
```
- 11/13/15 → 机器人左臂，12/14/16 → 右臂（与 MediaPipe 解剖标号一致）
- 若仍反：启动参数 `_swap_arms:=true`

## 阶段三：加入低通滤波

### 现象
手臂控制抖动、不跟手。

### 原因
MediaPipe 每帧有噪声，50Hz 直接发关节角会高频抖动。

### 解法
指数滑动平均（EMA）：
```
输出 = α × 当前角度 + (1-α) × 上一帧角度
```
- 默认 `_filter_alpha:=0.35`
- α 越小越平滑但延迟越大；α 越大越跟手但越抖

### 低通滤波有什么用？
滤掉动捕帧间的高频噪声，让关节指令更平滑，机器人动作不会「发抖」。代价是略有延迟。

## 阶段四：胸前横放 —— 第一次尝试（失败）

### 现象
人手横在胸前，机器人小臂竖着或方向完全不对。

### 原因
1. 大臂横放时 `atan2(z, y)` 奇异，旧代码把 pitch **锁死在上一帧**
2. 只控 pitch / roll / elbow，**肩 yaw（关节 3）恒为 0**，肘只能在竖直平面弯
3. 7 自由度手臂 + 3 个骨骼点，无法完全表达「胸前横放 + 小臂水平」

### 错误尝试（已废弃）
- 完整 7DOF + 躯干坐标系重写 → 左右全反，比原版更差
- 强行 roll 顶到 120° → 手臂向外侧举，不是过胸
- 肩 yaw = 90° → 肘弯平面变竖直，小臂上举

---

## 阶段五：回退原版 + 最小改动

### 决策
用户反馈重写版「一塌糊涂」，**完整回退**到调通的 3D 球坐标逻辑，只保留：
1. 检测用原图、显示才镜像
2. 低通滤波

### 原则
胸前问题做小范围补丁，**不再大改整体架构**。

## 阶段六：肘弯不到 90° 以内？URDF 限制？

### 现象
感觉大臂小臂夹角弯不到 90° 以内（不能握拢）。

### 原因（非 URDF 限制）
```xml
<!-- zarm_l4_joint 肘关节 -->
<limit lower="-2.618" upper="0.0" />  <!-- -150° ~ 0° -->
```
URDF 允许弯到 **150°**，不是 90° 卡死。

实际原因：
- 缺肩 yaw，肘弯平面不对，**看起来**像只能弯 90°
- MediaPipe 锐角弯曲常**低估**
- 低通滤波有延迟

### 解法
- 胸前模式补充肩 yaw
- 可选 `_elbow_gain:=1.2` 放大深度弯曲（>60° 部分）

## 阶段七：胸前模式检测与映射（逐步修正）

### 7.1 检测从未触发
**日志**：`p:-90  y:0  [无左胸]`  
**原因**：旧条件 `yz_norm < 0.08` 太严，大臂仍有 z 分量  
**解法**：改为「前臂近水平（x-z 主导）+ 肘弯 > ~40°」

### 7.2 yaw = 90 导致小臂竖直
**日志**：`p:0  r:34  y:90  e:-107  [左胸]`  
**原因**：把前臂水平角误映射到肩 yaw=90°，肘弯平面变竖直  
**解法**：胸前模式 **yaw 改由「前臂与大臂水平朝向差」** 计算，并限制在 ±57° 内；后证实 yaw=0 + 中等 roll 组合更合适

### 7.3 roll = 120 导致方向错误
**日志**：`p:0  r:120  y:0  e:-97  [左胸]`  
**原因**：roll 顶满 120° 是夸父**向外侧举**的极限，不是横过胸前  
**解法**：roll 用原版 `atan2(x, yz)`（约 30°~50°），配合 pitch + yaw，不再强行放大 roll

### 7.4 最终胸前模式逻辑（当前）
```
检测：前臂 x-z 平面占比高 + 肘弯 > 40° + 滞后防抖（6 帧进 / 12 帧出）
映射：
  pitch = atan2(z, hypot(x,y))  限制在 ±25°~±30°
  roll  = atan2(x, yz)          中等值，不过 120°
  yaw   = fore_az - upper_az    让肘弯平面躺到胸前
  wrist_yaw = 由前臂水平朝向微调（仅胸前）
```

## 阶段八：左右腕姿态不对称 + 模式切换抖动

### 现象
1. 双手自然下垂：左手正常，右手掌朝向略偏
2. 胸前 ↔ 普通模式切换时 `[左胸]` 标签闪烁、关节跳变

### 原因
1. 曾给右臂 **roll 取反** → 侧平举时 roll 反向，手臂**折进身体**（已撤销）
2. 退出胸前模式后 yaw/腕关节不再更新，卡在旧值
3. 检测阈值单一，边界帧来回切换

### 解法
| 问题 | 解法 |
|------|------|
| 右手下垂姿态 | 仅在 `is_arm_hanging()` 时微调 **腕 roll**（增益 0.35），侧平举/胸前不干预 |
| 右臂 roll 取反 | **已删除**，左右统一 `roll = atan2(x, yz)` |
| 模式切换抖 | 滞后防抖 + 双阈值；**始终滤波** yaw/腕关节（2/4/5/9/11/12），普通模式目标为 0 平滑回落 |
| 侧平举误判胸前 | 手臂伸直 + 大臂朝两侧 → 强制 `detect_chest_cross = False` |

---

## 阶段九：右手侧平举折进身体（回归 bug）

### 现象
右手横举，机器人手臂插进胸口。终端可能出现 `[左胸]`（左臂误判）。

### 原因
阶段八为修下垂姿态加的 **`roll = -roll`（右臂取反）**，侧平举时正 roll 变负 roll，往体内折。

### 解法
1. **彻底去掉**右臂 roll 取反
2. 腕 roll 修正**仅在下垂时**生效
3. 侧平举排除胸前检测

→ 用户反馈：**「这一版终于接近完美了」**

|------|----------|------|

```bash

```

- **纯手腕拧转**（手臂不动只转腕）：Pose 只有肩/肘/腕 3 点，观测不足
- **胸前横放 100% 精准**：7DOF 手臂 + 3 点近似，必有误差
- 若要更高精度 → IK 版 + 完整坐标标定，或 VR/深度相机 + Hand 追踪

## 附录：IK 融合版（`teleop_geometric_ik_fusion.py`）专项记录

> 本节为 2026-06 后续调试补充，记录 IK 版的原理、踩坑与最终解法。  
> **上文阶段一至阶段九、文件说明等原有内容保持不变。**

### IK 版设计原理（初版设想）

IK 融合版的目标是在 math 版几何关节 seed 的基础上，调用 Kuavo 内置 IK 服务，用**笛卡尔空间末端位姿**做二次优化，理论上可以：

1. 利用机器人完整运动学链，弥补「3 点动捕 → 7 关节」的欠定问题；
2. 通过 `/ik/two_arm_hand_pose_cmd_srv_muli_refer` 多 seed 提高求解成功率；
3. 在几何解附近做腕部/冗余自由度微调。

初版管线分三块：

```
模块一：几何 seed —— 从肩→肘、肘→腕向量算 pitch/roll/elbow（6 关节）
模块二：绝对位移 —— MediaPipe 腕相对肩位移 → 映射到 base_link 目标点
模块三：IK 缝合 —— 把目标 pos + seed joint_angles 发给 IK，用返回 q_arm 驱动机器人
```

### 阶段十：IK 版「手放下、机器人仍举在胸前」

#### 现象
- 用户双手自然下垂，机器人双手仍弯肘举在胸前；
- 终端日志 `[IK]` 与 `[GEO-fallback]` 交替出现；
- IK 服务侧显示求解**成功**，位置误差仅 ~10–17 mm，但姿态完全不对。

典型日志对比：

| 模式 | 左肘 e | 右肘 e | 说明 |
|------|--------|--------|------|
| `[IK]` | ≈ -62° | ≈ -60° | 肘大幅弯曲，手在胸前 |
| `[GEO-fallback]` | ≈ -30° | ≈ -28° | 更接近自然下垂 |

IK 求解器打印的 Eef cmd 示例（用户手放下时）：

```
左手 pos: [0.021, 0.125, 0.310]
右手 pos: [-0.042, -0.211, 0.197]
```

z ≈ 0.2–0.3 m 对应机器人**胸前高度**，与「手放下」不符。

#### 原因（核心）

**不是 IK 求解器垃圾，而是喂给 IK 的目标位姿错了。**

| 层级 | 问题 | 说明 |
|------|------|------|
| 坐标系 | MediaPipe world ≠ robot base_link | MP：髋部原点、Y 向下、Z 朝相机；机器人：X 前 Y 左 Z 上。两者无标定，不能直接当 IK 目标 |
| 映射公式 | `offset = [-dz, -dx, -dy]` | 拍脑袋的轴置换，无肩宽/臂长/朝向标定，下垂时仍映射到胸前 |
| 肩膀锚点 | `[0.15, ±0.22, 0.12]` | 与 URDF `[-0.0175, ±0.2927, 0.4245]` 偏差大，目标点系统性偏移 |
| 控制逻辑 | IK 成功则**全量覆盖**几何解 | 几何解（正确）被错误 IK 结果（错误）替换 |
| 姿态 | 固定 `quat = [0, -0.707, 0, 0.707]` | 掌心朝下四元数，与动捕无关，进一步约束 IK 往错误解靠 |

本质矛盾：

- **Math 版**：只用**相对向量**（肩→肘、肘→腕）算关节角 → 与绝对坐标系无关 → 准；
- **旧 IK 版**：把 MP **绝对位移**当机器人腕部目标 → IK「成功」追到错误点 → 比 math 版更差。

#### 为什么 IK 显示 success 但动作仍错？

IK 只保证「当前关节角 → 算出的末端位姿」与「你给的目标位姿」接近。  
目标本身错了，IK 越成功，机器人离真人动作越远。这是**前端映射错误**，不是 `/ik/two_arm_hand_pose_cmd_srv` 坏了。

### 阶段十二：IK 版最终解法（当前可用版本）

#### 设计原则

```
动捕遥操：关节空间几何映射为主，IK 为辅（可选腕部微调）
绝不：把 MediaPipe 绝对坐标直接当机器人 IK 目标
```

#### 新管线

```
┌─────────────────────────────────────────────────────────┐
│  MediaPipe（未翻转原图）→ 肩/肘/腕 3D 点                │
└──────────────────────────┬──────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────┐
│  几何求解 solve_arm() —— 与 teleop_math_solver 完全一致   │
│  （普通模式 + 胸前模式 + 滞后防抖 + 腕部下垂微调）         │
└──────────────────────────┬──────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────┐
│  主输出 = geometric_q_arm（度）→ 滤波 → /kuavo_arm_traj   │  ← 默认路径
└──────────────────────────┬──────────────────────────────┘
                           │ 仅当 ~use_ik:=true
                           ▼
┌─────────────────────────────────────────────────────────┐
│  FK(/ik/fk_srv, 几何关节弧度) → 腕部 pos + quat         │
│  IK(muli_refer, pos_cost_weight=0, joint_angles_as_q0)  │
│  合并：默认只采纳腕关节 5/6/12/13，肩肘由几何解保护      │
└─────────────────────────────────────────────────────────┘
```

#### 关键改动对照

| 旧版（错误） | 新版（正确） |
|-------------|-------------|
| MP 腕位移 → `SHOULDER + offset` 作 IK 目标 | **删除** MP 绝对位置映射 |
| IK 成功 → 14 关节全替换 | **几何解为主**；IK 仅可选微调腕部 |
| 固定掌心朝下四元数 | FK 返回与几何关节一致的 `quat_xyzw` |
| `pos_cost_weight=0.01` 追错误位置 | `pos_cost_weight=0.0` 以 seed 为主 |
| 默认 `use_ik=true` | 默认 **`use_ik=false`**，行为等同 math 版 |
| 单 seed `/ik/two_arm_hand_pose_cmd_srv` | 默认 `/ik/two_arm_hand_pose_cmd_srv_muli_refer` |

#### 腕部保护逻辑

```python
WRIST_JOINT_IDX = [5, 6, 12, 13]   # 左腕 yaw/roll/pitch、右腕 yaw/roll
CORE_JOINT_IDX  = [0,1,2,3, 7,8,9,10]  # 肩肘核心，不被 IK 拉偏

# ik_wrist_only=true（默认）时：
# 仅当 |ik_deg[i] - geo_deg[i]| < 40° 才采纳 IK 的腕部值
```

### IK 版 ROS 参数（当前）

```bash
# 默认纯几何，与 math 版一致
python3 teleop_geometric_ik_fusion.py

# 显式参数
_filter_alpha:=0.35      # 低通，同 math 版
_elbow_gain:=1.2          # 肘弯增益，同 math 版
_swap_arms:=false         # 左右反了时 true
_use_ik:=false            # 默认关闭 IK（推荐）
_ik_wrist_only:=true      # 开启 IK 时仅微调腕部

_ik_service:=/ik/two_arm_hand_pose_cmd_srv_muli_refer
_fk_dof:=14               # FK 请求关节维数
```

可选开启腕部 IK：

```bash
python3 teleop_geometric_ik_fusion.py _use_ik:=true _ik_wrist_only:=true
```

终端模式标签：

| 标签 | 含义 |
|------|------|
| `[GEO]` | 纯几何解（默认，应与 math 版一致） |
| `[GEO+WRIST]` | 几何 + FK/IK 腕部微调 |
| `[GEO+IK]` | `ik_wrist_only:=false` 且核心关节偏差 < 20° 时全量采纳 IK |

### Math 版 vs IK 版：何时用哪个？

| 维度 | `teleop_math_solver.py` | `teleop_geometric_ik_fusion.py`（当前） |
|------|-------------------------|----------------------------------------|
| 控制空间 | 关节空间直接映射 | 关节空间为主，可选 IK 腕部 |
| 坐标标定 | 不需要 | **不需要**（已去掉 MP 绝对坐标） |
| 默认行为 | 几何解 | `use_ik=false` 时与 math 版相同 |
| 胸前模式 | 有 | 有（共用同一套 solve_arm） |
| 适用场景 | **手机单目动捕遥操（推荐）** | 与 math 等价 + 可选腕部 IK 实验 |
| 不适用 | 纯手腕拧转 | 同左；IK 无法凭空补全缺失观测 |

结论：

- **手机摄像头 + MediaPipe Pose**：优先 math 版或 IK 版默认配置（`use_ik=false`）；
- **VR/Pico 等已有标定笛卡尔位姿的管线**：才适合 IK 做主路径（参考 `pico-body-tracking-server`）；
- IK 融合版的价值不在「替代 math」，而在「几何解稳定后，可选地让官方 IK 润色腕关节」。

### IK 版调试技巧（补充）

1. **先看模式标签**：若手放下仍举胸，检查是否仍在跑旧脚本（有 `[IK]` 全量覆盖且无 `[GEO]`）。
2. **对比肘角 e**：下垂应对 `e ≈ -20° ~ -40°`；若 `e ≈ -60°` 且标签为 IK，说明仍在追错误笛卡尔目标（旧版行为）。
3. **IK 服务窗口**：若 `Eef cmd` 的 z 仍在 0.2+ 而人手下垂，说明前端映射错，不是 IK 内部 bug。
4. **快速回退**：`_use_ik:=false` 一行参数即可回 math 同级行为。
5. **服务依赖**：`use_ik=true` 时需 `/ik/fk_srv` 与 `muli_refer` 服务在线；纯几何模式无此依赖。

### IK 版已知局限

- MediaPipe Pose 只有肩/肘/腕 3 点，**观测不足以驱动完整 7DOF + 精确末端位姿 IK**；
- 无相机–机器人手眼标定时，**任何 MP 绝对坐标 → base_link 映射都不可靠**；
- IK 每帧阻塞耗时，即使降频到 25 Hz 仍不如纯关节映射轻量；
- `ik_wrist_only` 对整体动作提升有限，胸前模式等仍依赖几何 `solve_arm`。
