### 0.3 两种发车模式决策（经典版 vs OctoMap 版）

为了应对不同的物理桌面环境，我们设计了两种绝不冲突的发车模式。**终端 1、2、4、5、6、7 在两种模式下完全相同，仅在终端 3 和终端 8 产生分化。切记不可混用两者的 launch 文件**。

| 模式 | 终端 3 启动指令 | 终端 8 业务脚本 | 3D 点云建图 | 抓取后收手动作 |
| --- | --- | --- | --- | --- |
| **经典版** (默认，算力极稳) | `move_group.launch` | `moveit_auto_grasp.py` | ❌ 无 | VLA 纯骨骼肩膀外摆 75° |
| **OctoMap 点云版** (高级避障) | `move_group_octomap.launch` | `moveit_octomap_grasp.py` | ✅ 仅抬升后开启 | OMPL+OctoMap 绕行 (失败退回 vla) |

#### 💡 打破迷思：“MoveIt 实际到底用到了哪一步？”

用户常问：经典版和老代码看起来差不多，MoveIt 真在用吗？下面是底层真相：

* **启动阶段**：MoveIt 负责加载 `biped_s49` 模型（此时出现的 URDF collision WARN 可忽略）。
* **抓取前（核心）**：全权调用 MoveIt 的 `/compute_ik` 服务，进行抓握、预瞄、抬升等关键点的高精关节角解算。
* **执行插入**：**坚决不用** OMPL，而是调用 `execute_single_pose` 直发 `/kuavo_arm_target_poses` 底层话题。
* **收手阶段**：经典版**不用**点云，调用 `execute_vla_style_return()`。
* **异常回防**：只有在发生错误，触发 `_safe_return_both_arms()` 时，才会调用 OMPL。
*(判别技巧：日志若结尾为 `肩膀外摆避障`，说明跑的是经典版，全程无点云；若包含 `阶段B` / `OctoMap外移`，则为点云版收手。)*

### 0.5 关键配置字典与 TCP 精密标定表

为了压榨出 100% 的抓取成功率，我们在 `src/demo/vla_grasp/moveit_auto_grasp.py` 内部固化了极限调参的 TCP 参数表（OctoMap 版直接复用该模块参数）。若出现机械偏差，请在此处微调。

| 代码参数名 | 当前绝对值 | 物理含义与调参法则 |
| --- | --- | --- |
| `TCP_OFFSET_X_LEFT` | `-0.013` | 左手深度补偿。若抓取偏前（顶翻水瓶），让此值更负。 |
| `TCP_OFFSET_X_RIGHT` | `-0.022` | 右手深度补偿，实测略多于左手。 |
| `TCP_OFFSET_Y_LEFT` | `0.004` | 左手横向补偿。若夹爪偏左，则**减小**该值。 |
| `TCP_OFFSET_Y_RIGHT` | `0.065` | 右手横向补偿。若夹爪偏右，则**增大**该值。 |
| `SAFE_LOCKED_Z` | `0.385` | 绝对抓握空域高度（米）。 |
| `LIFT_HEIGHT` | `0.22` | 抓紧后首选垂直抬升高度（22cm）。 |
| `LIFT_HEIGHT_FALLBACKS_M` | `(0.22, 0.18, 0.14, 0.10)` | 抬升 IK 无解时按序递减重试（米）。 |

### 0.6 双机重构与双机修改指南（附相机防死锁补丁）

当我们需要同时修改底层的 MoveIt 以及上位机的传感器逻辑时，存在物理空间的隔离，需要利用 Remote SSH 建立双路工作空间。

**跨设备文件修改操作规范**：

| 需要改什么内容？ | 物理存放位置 | 推荐的 IDE 打开方式 |
| --- | --- | --- |
| 抓取 Python 脚本、MoveIt Launch、TCP 参数 | 下位机 NUC | 本机直接打开 `~/kuavo-ros-opensource`，或 `ssh lab@192.168.26.1` |
| 相机底层 Launch、Orbbec 驱动、点云参数 | 上位机 Orin | Remote SSH 穿透至 `leju_kuavo@192.168.26.12`，打开 `~/kuavo_ros_application` |

| 跨机目录本地化编辑 | 下位机 SSHFS | 挂载：`sshfs leju_kuavo@192.168.26.12:~/kuavo_ros_application ~/remote_upper_ws` |

#### 🛡️ 终极防崩溃：上位机相机 Launch 修改摘要（终端 5 相关）

**历史痛点**：`/camera/depth/color/points` 话题虽然存在，但 `hz` 探针总是返回 `no new messages`。第一版排障时，我们强制打开了驱动自带的点云并配置了 relay 转发，结果导致多订阅压力压垮驱动，一抓取相机就报 `SIGSEGV`。

**最终长效稳定解决方案**：
我们必须在上位机的 `~/kuavo_ros_application/src/dynamic_biped/launch/orbbec_sensor_robot_enable.launch` 中进行彻底的外部接管配置：

1. **彻底关闭驱动内点云**：`enable_point_cloud` 强行设为 **`false`**。
2. **给带宽减负**：将 `color_fps` 从 60 降级为 **30**。
3. **外部手搓点云生成**：利用 `depth_image_proc/point_cloud_xyz` Nodelet，挂载名为 `vla_depth_to_pointcloud` 的 publisher，吃进深度图，自己算好点云后，输出到 `/camera/depth/color/points`。
*(完整 XML 改前/改后见 **附录 C**)*。

### 0.7 常见悬疑现象速查表 (FAQ)

如果环境发生变异，请对照以下黄金军规自查：

1. **报 `No /joint_states`？** → 终端 2 没开，或者没运行成功。
2. **报 `compute_ik` 不可用？** → 终端 3 没就绪，没等出 `You can start planning now!`。
3. **`compute_ik` 疯狂 WARN 但依然能抓取？** → 正常现象，TRAC-IK 底层废弃 API 报警，不影响算力，请无视。
4. **脑子里记得有点云收手，但实机跑出来却没有？** → 你跑错脚本了！你跑成了经典版。请换成 OctoMap 版本的终端 3 + 终端 8。
5. **URDF 出现 collision WARN 报警？** → 模型文件确实没写碰撞，启动时报一次是正常的，可忽略。
6. **胳膊不动 / 乱动？** → 检查是否误开了 `kuavo_moveit_bridge.py`（与终端 8 互斥，见 0.8）。
7. **启动终端 8 后长时间无反应？** → 正常：须先等视觉 10 帧采完才加载 MoveIt（见 0.8.2）。

当这套“八大终端矩阵”绿灯全亮时，这台双臂机器人将化身为同时具备极其稳定的底层盲插能力与 3D 点云避障智商的顶级工业怪兽。去敲下回车吧，大捷就在眼前！

### 0.8 系统级禁令与启动铁律（当前稳定版必须遵守）

#### 0.8.1 系统级禁令

| 禁令 | 原因 | 违反后果 |
| --- | --- | --- |
| Python `move_group.compute_cartesian_path()` | Boost.Python 签名 Bug | 进程闪退 `ArgumentError` / `SIGSEGV` |
| MoveIt 版调用 `/ik/two_arm_hand_pose_cmd_srv` | 项目要求纯 MoveIt IK | 架构违背；与 TRAC-IK seed 策略冲突 |
| 与 `kuavo_moveit_bridge.py` 同时运行 | 同抢 `/kuavo_arm_target_poses` | 胳膊不动或乱动 |
| 两种 `move_group` 同时开 | 重复节点 / 话题冲突 | 不可预测 |
| 经典版终端 3 与点云版终端 8 混配 | 传感器配置不一致 | 点云收手失效或误报 |

**允许**：ROS 服务 `/compute_ik`、`/compute_cartesian_path`（C++ 服务，非 Python 绑定）；代码内保留链式 IK / 笛卡尔服务为**备用**，主路径不调用。

#### 0.8.2 启动顺序铁律（与 `auto_grasp_TF2.py` 对齐）

```
wait /joint_states → arm_traj_change_mode(2) → 夹爪张开
→ execute_dual_arm_init_home() # DUAL_ARM_INIT_DEG：肩 20°、肘 -30°
```

**勿**在视觉就绪前加载 MoveIt；**勿**跳过终端 2（`/joint_states` 引渡）。

## 第1章 项目背景与架构重构抉择

具身智能（Embodied AI）的发展从来不是一蹴而就的，它是一场算法与现实物理世界残酷碰撞的持久战。本章将追溯该项目的起源，剖析我们为什么要在拥有了“百发百中”的自研硬编码方案后，依然顶着无数报错和 Bug，毅然决然地投入 MoveIt! 庞大复杂的架构怀抱。

### 1.1 前情提要：硬编码时代的极致与局限

在正式接入 MoveIt! 之前，我们在这台 Kuavo 14-DOF（双臂 14 自由度，单臂 7 轴冗余）的人形机器人上，已经完成了一项堪称“艺术品”的工程壮举——一套完全不依赖任何高级规划框架的**纯视觉识别与硬编码抓取系统**。

#### 1.1.1 纯视觉降维映射（非 MoveIt）的百发百中体验

在早期的 VLA（Vision-Language-Action 雏形）架构中，我们的主脚本 `auto_grasp_TF2.py` / `vla_auto_grasp_daemon.py` 表现出了令人恐怖的稳定性和执行力。
我们是如何做到“百发百中”的？这得益于我们对机械公差与数学极限的病态级压榨：

1. **时域中值滤波（10帧速采）**：面对深度相机红外散斑导致的 XY 坐标疯狂跳动，我们直接写死了一个 10 帧采集环，抛弃平均数，采用 `np.median` 中值滤波，瞬间将噪点剥离，得到了如磐石般稳定的水瓶绝对坐标。
2. **TCP 毫米级分参补偿（`tcp_offsets_for_arm`）**：由于夹爪旋转 90 度后，内部的齿轮结构产生了无法在运动学模型中体现的偏心误差，导致“左手抓偏左，右手抓偏右”。我们引入了丧心病狂的物理补丁：
* `TCP_OFFSET_X_LEFT = -0.013`, `TCP_OFFSET_X_RIGHT = -0.022`
* `TCP_OFFSET_Y_LEFT = 0.004`, `TCP_OFFSET_Y_RIGHT = 0.065`
* `LIFT_HEIGHT_FALLBACKS_M = (0.22, 0.18, 0.14, 0.10)` — 抬升 IK 失败时递减重试

3. **倒推法预瞄与绝对直线盲插**：我们利用 `math.hypot` 计算起点和终点的斜率，在抓取点往后倒推 `PRE_GRASP_DIST = 0.12m` 设立预瞄点。随后，调用极度可靠的原厂 `execute_single_pose` 函数，通过 1kHz 底层样条插值，笔直地把夹爪塞向目标，实现绝杀。
4. **肩关节强行外摆 75° 的 VLA 收手策略**：抓取完成后如何避免砸倒桌上的黑书？我们不依赖任何点云，直接暴力量化：抬升至 `LIFT_HEIGHT = 0.22m` 后，强行下发特定关节指令，让机械臂的肩膀大臂向外侧平行外摆（避开桌面上方领空），然后再曲肘护胸收回。

这套基于纯几何数学与厂家底层 IK 逆解服务（`/ik/two_arm_hand_pose_cmd_srv`）打造的脚本，执行速度极快，抓取精度令人发指。

#### 1.1.2 为什么必须引入 MoveIt!？（动态障碍物与 3D 空间天花板）

既然老版本已经“百发百中”，为什么我们还要自找麻烦去啃 MoveIt! 这块硬骨头？
答案只有四个字：**“泛化能力（Generalization）”**。

老版本代码的本质是一条“定死的数学公式”。它之所以能避开桌上的书本，是因为我们把避障路径（L型走廊/外摆75°）以硬代码的形式“刻”在了程序里。

* **致命缺陷一：静态陷阱**。如果今天放在桌子上的不是一本黑书，而是一个竖立的矿泉水桶，或者是一个高高的杂物架，刚好挡在了我们预设的“安全外摆走廊”上，老版本代码会**像个瞎子一样，全速、毫不留情地一巴掌把杂物拍飞，甚至导致自身机械臂扭曲受损**。
* **致命缺陷二：空间天花板**。纯逆解代码无法感知空间的三维稠密状态。它无法理解“绕过去”、“钻过去”这种人类的高级空间规划。没有基于 OctoMap（八叉树）的实时三维碰撞检测，机器人永远只是一个按设定路径运转的高级自动化机器，而永远触碰不到“具身智能”的真正门槛。

引入 MoveIt!，就是为了打破这座无形的墙。虽然前方的路布满了 C++ 报错、点云假死、时间参数化崩溃等恶心陷阱，但这却是通往未来大模型（LLM/VLM）接管动态复杂场景的必经之路。上限的抬高，值得付出一切代价。

### 1.2 MoveIt! 算力节点的架构部署之辨

在决定引入 MoveIt! 后，我们面临的第一个架构级工程难题就是：**如此庞大、吃算力、且占据核心控制权的 MoveIt! 生态群（包括 MoveGroup、OMPL 规划器、Planning Scene 等），到底该部署在上位机（Orin NX），还是下位机（NUC）？**

这个决定一旦做错，将导致无法挽回的延迟与控制崩溃。

#### 1.2.1 方案 A：部署于上位机（Orin NX）的带宽延时隐患

* **架构设想**：让算力最强的 Orin NX 承担所有重活。上位机既跑 YOLO 视觉，又跑深度相机点云生成，还顺带把 MoveIt! 一起跑了。然后上位机只需把 MoveIt 算好的 `JointTrajectory`（一堆关节角度序列）通过 ROS 局域网发给下位机执行即可。
* **致命隐患**：
1. **状态反馈的灾难性延迟**：MoveIt! 规划路线时，极度依赖极其精准的、由底盘传来的 `1kHz` 级别的 `/joint_states`（全身28轴关节反馈）。如果把 MoveIt 放在上位机，这些海量的高频关节状态数据必须全部通过以太网口塞进局域网。网络稍微有一点抖动（Jitter），MoveIt 就会认为机器人的当前状态（Start State）已经过时，导致规划的起点与物理实体的真实位置产生撕裂。
2. **执行链路的脆弱性**：MoveIt! 与底层的 `FollowJointTrajectory` Action Server 之间的交互是高频双向反馈的。如果中间隔着一条物理网线，一旦发生丢包或阻塞，下位机正在半空中挥舞的数十公斤重的双臂将瞬间失去主脑指令，触发底层断崖式掉落或失控。

#### 1.2.2 方案 B：部署于下位机（NUC）的强实时性优势

* **架构设想**：让 MoveIt! 彻底“下沉”到机器人内部。下位机 NUC 承担全部的运动控制与规划任务。上位机（Orin NX）纯粹被降格为一个“眼睛”，只负责向局域网广播视觉靶点坐标（`/vla/yolo_target`）和 3D 点云（`/camera/depth/color/points`）。
* **工程优势**：
1. **极度贴近硬件**：下位机的 MoveGroup 与底层的 WBC 控制器、IK 原厂解算服务存在于同一台主机的同一块内存池中。它们之间的通讯（无论是 `/joint_states` 还是 Action 调用）延迟几乎为零，实现了微秒级的运动链闭环。
2. **算力切分合理**：OctoMap 碰撞树的更新虽然吃 CPU，但 NUC 的 i7 处理器完全足以应对。而最消耗 GPU 的 YOLOv8 张量推理和点云生成，被完美外包给了拥有独立 NVIDIA Ampere 架构的上位机 Orin NX。

#### 1.2.3 最终决断：MoveGroup 下沉与 TF2 跨机解耦架构

经过对以上风险的极限推演，我们最终拍板了**绝对的“算力解耦、控制下沉”架构**。

1. **上位机（Orin NX 视觉脑）**：
* 专注执行 `gemini_330_series.launch` 和 `depth_image_proc` 节点。
* 专注执行 YOLOv8 实例分割，输出 `/vla/yolo_target`。其核心精髓在于：上位机的 Python 脚本利用 `tf_buffer.lookup_transform`，通过局域网读取下位机的 TF 树，把相机的局部坐标 $P_{camera}$ 乘以机器人的颈部运动矩阵，最终向外广播的是抵消了晃动误差的、基于 `base_link` 的绝对物理坐标。这就把所有的“空间不确定性”直接在上位机抹平了。

2. **下位机（NUC 运动脑）**：
* 稳如泰山地驻留 `move_group.launch`（或 `move_group_octomap.launch`）。
* 运行终极业务主脑 `moveit_octomap_grasp.py`，接收上位机投喂的纯净点云和纯净目标靶点，安心地利用 OMPL 和 TRAC-IK 引擎去打磨那最后一段极其复杂的 3D 避障轨迹与精准插值切入。

正是这一极具前瞻性的架构重构，彻底扫清了我们在引入 MoveIt! 体系初期的基础通讯障碍，为后续我们与 `SIGSEGV`、`Time Parameterization`（时间参数化）等深层地狱级 Bug 进行殊死搏斗，打下了最坚实的地基。

### 1.3 MoveIt! 核心引擎的安装与配置实战（Setup Assistant 全流程）

在确立了“算力下沉至下位机”的分布式架构后，我们的核心任务是在 NUC（Ubuntu 20.04 + ROS Noetic）上从零搭建 MoveIt! 引擎。MoveIt! 并不是一个单一的脚本，而是一个庞大的生态。要让它接管 Kuavo 4 Pro 的 14 个手臂关节，我们必须使用官方的图形化配置工具 **MoveIt! Setup Assistant**（即开发者圈内戏称的“拐杖老头”）来生成专属的配置包。

#### 1.3.1 底层环境整备：MoveIt! 框架与依赖树安装

在开始配置之前，必须确保下位机的 ROS Noetic 环境中安装了 MoveIt! 的全量包以及我们后续必须用到的高级运动学求解器。

**执行终端指令（下位机 SSH 环境下）：**

```bash
# 1. 更新本地软件包索引
sudo apt-get update

# 2. 安装 MoveIt! 核心元程序包（包含所有核心依赖）
sudo apt-get install ros-noetic-moveit -y

# 3. 安装 MoveIt! 视觉工具包（后续 RViz 调试轨迹显示必须）
sudo apt-get install ros-noetic-moveit-visual-tools -y

# 4. 【极度关键】安装 TRAC-IK 运动学求解器插件
# 原因：MoveIt 默认的 KDL 求解器在面对 Kuavo 的 7-DOF 冗余机械臂时表现极差（会导致手肘乱甩），必须用 TRAC-IK 替代。
sudo apt-get install ros-noetic-trac-ik-kinematics-plugin -y

```

#### 1.3.2 唤醒“拐杖老头”：Setup Assistant 初始化与 URDF 导入

环境就绪后，我们需要将机器人的物理模型（URDF）喂给 MoveIt，让它理解机器人的骨骼结构。

**执行终端指令：**

```bash
# 启动 MoveIt! 配置助手图形界面
roslaunch moveit_setup_assistant setup_assistant.launch

```

执行后，屏幕上会弹出一个带有“拄着拐杖的老头” Logo 的图形化界面。

**具体配置操作（Create New MoveIt Configuration Package）：**

1. 点击界面左侧的 **Create New MoveIt Configuration Package**。
2. 在 `URDF File` 选择框中，点击 `Browse`，导航到我们工作空间下的机器人模型文件（即 `~/kuavo-ros-opensource/src/.../urdf/biped_s49.urdf`）。
3. 点击 **Load Files**。此时，如果模型加载成功，界面右侧会渲染出 Kuavo 机器人的 3D 灰色骨架模型。

#### 1.3.3 碰撞矩阵构建（Self-Collisions）：规避物理自残的第一道防线

机器人手臂在挥舞时，如果不加限制，左手极有可能会打到自己的胸口或右手。MoveIt! 通过采样生成自碰撞免检矩阵（Allowed Collision Matrix, ACM）来节省算力。

**具体配置操作：**

1. 点击左侧导航栏的 **Self-Collisions**。
2. 将 `Sampling Density`（采样密度）直接拉到最高（Highest，通常为 10000 次采样）。
3. 点击 **Generate Collision Matrix**。
4. **【踩坑记录与警告】**：在此步骤以及后续启动终端时，控制台曾疯狂刷屏黄字警告：`[WARN] Link zarm_r7_link has visual geometry but no collision geometry`。
* **原因溯源**：原厂提供的 `biped_s49.urdf` 模型文件存在偷工减料，很多连杆（包括手臂、雷达、相机）只写了用于渲染外观的 `<visual>` 标签，而省略了用于物理碰撞计算的 `<collision>` 标签。
* **解决方案**：在 Setup Assistant 阶段我们只能无视此警告继续生成。在后续的实战代码中（第7章），我们通过手动在代码里注入一个虚拟的 `CollisionObject`（桌面安全盒）和设置外扩容差（`AVOID_OUTWARD_Y = 0.10`）来弥补模型碰撞体的缺失。

#### 1.3.4 核心命脉配置（Virtual Joints & Planning Groups）

这是整个配置环节的灵魂，它决定了我们后续在 Python 脚本中通过 `MoveGroupCommander("left_arm")` 能够调动哪些电机。

**A. 配置虚拟关节（Virtual Joints）**

1. 点击左侧 **Virtual Joints** -> `Add Virtual Joint`。
2. `Virtual Joint Name` 填入 `virtual_joint`。
3. `Child Link` 选择机器人的根节点（通常为 `base_link` 或 `pelvis`）。
4. `Parent Frame Name` 填入 `world`，`Joint Type` 选择 `fixed`（固定）。这一步是告诉 MoveIt，我们的机器人躯干是相对大地坐标系静止的（抓取时不考虑底盘移动）。

**B. 配置规划组（Planning Groups）——【重中之重】**
我们需要把机器人的 14 个手臂关节拆分成左右两个独立的大脑。

1. 点击左侧 **Planning Groups** -> `Add Group`。
2. **构建左臂 (`left_arm`)**：
* `Group Name`: 填入 **`left_arm`**。
* `Kinematic Solver`: **【防坑警告】绝对不要选默认的 `kdl_kinematics_plugin`！** 下拉选择我们在 1.3.1 安装的 **`trac_ik_kinematics_plugin/TRAC_IKKinematicsPlugin`**。KDL 面对 7 轴冗余臂时解算极慢且姿态怪异，TRAC-IK 结合了 KDL 和数值优化，成功率呈指数级上升。
* 点击 `Add Kin. Chain`（添加运动学链）。
* `Base Link` 选择 `base_link`。
* `Tip Link` 选择 **`zarm_l7_end_effector`**。**【天坑复盘】**：这里绝对不能选 `zarm_l7_link`！我们曾在这里吃过大亏，导致抓取时 Z 轴产生 17cm 的巨大落差，因为原厂的 TCP（工具中心点）定义在 end_effector 上。

3. **构建右臂 (`right_arm`)**：
* 重复上述步骤，`Group Name` 设为 **`right_arm`**，求解器依然选 **TRAC-IK**。
* `Base Link` 选 `base_link`，`Tip Link` 选 **`zarm_r7_end_effector`**。

#### 1.3.5 预设姿态与末端定义（Robot Poses & End Effectors）

为了让脚本能够实现“一键回防”和“安全退回”，我们需要在界面里提前捏好几个绝对安全的静态姿态（Named Poses）。

**具体配置操作（Robot Poses）：**

1. 点击左侧 **Robot Poses** -> `Add Pose`。
2. 创建左臂护胸姿态：
* `Pose Name`: 命名为 **`left_ready`**。
* `Planning Group`: 选择 `left_arm`。
* 在右侧的关节滑动条中，将左臂的各个关节调整至“曲肘、收拳、紧贴胸前”的安全防撞姿态（对应后续代码中频繁调用的 `arm.set_named_target("left_ready")`）。
* 点击 `Save`。

3. 创建右臂护胸姿态：
* 重复上述步骤，命名为 **`right_ready`**，选择 `right_arm` 组，调整右臂至镜像的护胸待命姿态。

*注：End Effectors（末端执行器夹爪）虽然可以在 Setup Assistant 里配置，但由于乐聚的夹爪底层使用了特定的 Action Service（`controlLejuClaw`），我们选择在 Python 业务流中独立控制夹爪闭合，因此在拐杖老头中跳过夹爪组的配置，以精简 TF 树负担。*

#### 1.3.6 补全与生成（Configuration Files）

所有的配置参数、骨骼映射和求解器设定，最终需要打包成一个独立的 ROS 功能包。

**具体配置操作：**

1. 点击左侧 **ROS Control** -> `Auto Add FollowJointTrajectory Controllers For Each Planning Group`。让 MoveIt 自动生成适配我们规划组的控制器接口模板。
2. 填写作者信息（Author Information）：随意填写。
3. 点击左侧 **Configuration Files**。
4. `Configuration Package Save Path` 填入我们下位机工作空间下的目标路径：`~/kuavo-ros-opensource/src/kuavo_arm_moveit_config`。
5. 点击 **Generate Package**。
6. 等待进度条走完，提示成功后退出“拐杖老头”。

#### 1.3.7 MoveIt! 配置阶段的“隐藏天坑”与后期强力介入

至此，我们的 `kuavo_arm_moveit_config` 功能包已经生成，但由于我们做的是 VLA 视觉抓取任务，默认生成的包有几个致命缺陷，必须手动介入修改源代码：

### 2.1 深入 URDF 骨骼树：从 5 代旧模型到 `biped_s49` 的迁移与碰撞体修补

MoveIt! 对机器人所有的运动学推演（包括逆解、碰撞检测、重心计算）都 100% 依赖于我们在 Setup Assistant 中喂给它的 URDF（Unified Robot Description Format）模型文件。如果 URDF 的连杆定义有丝毫偏差，物理世界中的机器人就会化身为一台“盲目破坏机”。

#### 2.1.1 摒弃残缺模型：载入 `biped_s49.urdf` 与 17cm 致命落差的修复

在早期的拓荒中，由于版本迭代混乱，下位机工作空间中存在多个历史 URDF 模型。当我们加载了错误的旧代模型并在 Setup Assistant 中配置 Planning Group 时，埋下了一个导致后续抓取屡屡“隔空抓物”的惊天大坑。

* **执行终端指令（启动配置助手核查模型）**：

```xml
<group name="right_arm">

```

#### 2.1.2 物理自残隐患：无碰撞几何体（Collision Geometry）对 OMPL 的降维打击

* **现象记录**：在终端启动 MoveIt! 时（执行 `roslaunch kuavo_arm_moveit_config move_group.launch`），控制台疯狂刷屏恶心的黄色警告：
```text
[WARN] [1781602367.901304423]: Link camera has visual geometry but no collision geometry. Collision geometry will be left empty. Fix your URDF file by explicitly specifying collision geometry.
[WARN] [1781602367.910158341]: Link zarm_r7_link has visual geometry but no collision geometry...

```

* **根因剖析**：乐聚原厂的 URDF 建模为了节省资源或赶进度，只赋予了机器人的手臂、头部、雷达等部件 `<visual>`（外观渲染）属性，却没有提供对应的 `<collision>`（物理碰撞）网格（Mesh）。对于依赖体素碰撞树的 MoveIt OMPL 规划器来说，这台机器人就是一个“只有幻影、没有实体的幽灵”。在规划收手退回路径时，极易发生“机械臂穿透自己的大腿或胸膛”的自残级路径解。
* **工程妥协与解决**：短期内利用 Meshlab 等软件重构厂家的 URDF 成本极高。为了防止机械臂因为“不知自己粗细”而砸毁自身或桌面，我们在后续的代码中（`moveit_octomap_grasp.py`）强行在 `PlanningScene` 中注入了一个 `DESK_BOX` 碰撞体，并且在点云避障时，给予了 `AVOID_OUTWARD_Y = 0.10`（向外扩容 10cm）的安全余量，以软件逻辑强行弥补硬件模型的残缺。
**Python 环境强行注入桌面碰撞盒代码实录**：
```python
def setup_desk_collision_box(scene_iface):
    """向 PlanningScene 添加桌面长方体，弥补 URDF 无桌面的缺陷。"""
    box = CollisionObject()
    box.header.frame_id = "base_link"
    box.id = DESK_BOX_ID
    primitive = SolidPrimitive()
    primitive.type = SolidPrimitive.BOX
    primitive.dimensions = list(DESK_BOX_SIZE)
    box.primitives.append(primitive)
    pose = Pose()
    pose.position.x = DESK_BOX_CENTER[0]
    pose.position.y = DESK_BOX_CENTER[1]
    pose.position.z = DESK_BOX_CENTER[2]
    pose.orientation.w = 1.0
    box.primitive_poses.append(pose)
    box.operation = CollisionObject.ADD
    scene_iface.add_object(box)
    rospy.sleep(0.5)
    rospy.loginfo(
        f"📦 已添加桌面碰撞盒 id={DESK_BOX_ID} "
        f"center=({DESK_BOX_CENTER[0]:.2f},{DESK_BOX_CENTER[1]:.2f},{DESK_BOX_CENTER[2]:.2f})"
    )

```

### 2.2 动作管道（Action Pipeline）的“鸠占鹊巢”死锁与物理失联

这是我们在真机联调中遇到的最令人绝望、也是最诡异的通信 Bug。它一度让我们怀疑机器人的硬件驱动是否已经烧毁。

#### 2.2.1 灵异现象：MoveIt 光速宣告 `SUCCEEDED` 但实体机械臂纹丝不动

* **复现步骤（历史）**：我们在终端 3 启动了 `move_group.launch`，然后在早期业务脚本中通过 MoveIt Action 下发 `arm.go()`（**当前稳定版已弃用 Action**，直发 `/kuavo_arm_target_poses`）。
* **诡异日志**：在 `move_group` 的终端中，日志显示极度完美，毫无滞后：
```text
[INFO] [1781446658.454062019]: Solution found in 0.010888 seconds
[INFO] [1781446658.459468303]: SimpleSetup: Path simplification took 0.005340 seconds and changed from 4 to 2 states
[INFO] [1781446658.460339006]: Completed trajectory execution with status SUCCEEDED...
```

* **物理现实**：无论代码里怎么规划，实验室里的机器人实体的胳膊就是像死铁一样，**纹丝不动**，我们仅仅听到夹爪“咔哒”空咬了一下。

#### 2.2.2 抓包查证：原厂控制器的 0 秒空切拦截与摆烂式反馈

为了抓出这只“看不见的黑手”，我们动用了 ROS 诊断三板斧，进行了深度抓包。

* **执行终端指令（诊断节点状态与话题）**：

```python
 mode_client(changeArmCtrlModeRequest(control_mode=2))
```

这个官方的 Action Server 只要监测到处于 `模式 2`，它就进入了“摆烂拦截状态”——一旦 MoveIt 下发了轨迹，这个“骗子服务器”会瞬间把轨迹丢进垃圾桶，不驱动任何物理电机，并在 0.01 秒内立刻通过 ROS Action 回调机制向 MoveIt 回复一个 `SUCCESSFUL`！这就造成了 MoveIt 大脑以为任务完美完成，而底层的真实物理电机根本没有收到任何电流指令的“双重欺骗死锁”。

### 2.3 中间件探索：构建 `kuavo_moveit_bridge` 拦截 Action 流（已废弃的经验遗产）

面对 Action 管道被官方驱动“流氓霸占”的局面，我们最初试图用优雅的软件工程方式来解决，于是诞生了第一代桥接器脚本。

#### 2.3.1 监听剥离：试图截获 `FollowJointTrajectory` 动作流与 14 轴状态映射

* **代码构想**：我们编写了 `src/demo/vla_grasp/kuavo_moveit_bridge.py`，作为一个 ROS 中间件。它的任务是自己建立一个 Action Server，在 MoveIt 和真机之间架起一座桥。
* **指令截获与重组**：它试图把 MoveIt 算出来的包含几十个航点（Waypoints）的 `JointTrajectory` 轨迹数组生生拦截下来。然后提取最后一个点，拼接成乐聚原厂格式：
```python
# 桥接器拦截代码片段伪码
def execute_cb(self, goal):
 points = goal.trajectory.points
 last_point = points[-1] # 提取 MoveIt 的最终目标点

 # 组装原厂消息
 target_msg = armTargetPoses()
 target_msg.times = [2.0]
 target_msg.values = list(last_point.positions)

 self.arm_pub.publish(target_msg)
 self._as.set_succeeded()

```

* **日志特征**：启动该桥接器后，终端曾意气风发地打印出：`🚀 Left & Right 双通道 Action 服务端满血启动，随时开火！`

#### 2.3.2 冲突原罪：单点插值法与 `/kuavo_arm_target_poses` 话题轰炸的双重死锁

* **为什么失败（工程上的必然溃败）**：
1. **话题饥饿与优先级争夺**：由于系统优先级的 Bug，MoveIt 启动时依然优先连上了厂家的那个假 Action Server。导致我们的 Bridge 脚本一直显示“随时开火”，却被饿死在 ROS 总线上，从未截获过 `🔥🔥🔥 截获...` 的实际打印日志。
2. **物理保护崩塌（高频轰炸死锁）**：即使我们后期通过 namespace 隔离强行让 Bridge 截获了点流，由于 MoveIt 下发的轨迹是包含极其细密时间戳的连续流（例如 0.01 秒一个插值点），而原厂底层最稳定接收的通道是**单点大时长插值话题 `/kuavo_arm_target_poses**`（它内部依靠 1kHz WBC 自己做样条插值）。如果我们把 MoveIt 吐出的密集点流，以毫秒级频率疯狂轰炸进这个原生话题，底层控制器的插值器会瞬间队列溢出，触发死机保护。

* **壮士断腕的架构重构**：经过一夜抓包与痛苦调试，我们得出了一个铁血结论——**“只要走 Action 就会被官方劫持”**。于是，我们彻底将 `kuavo_moveit_bridge.py` 扫进垃圾堆，决定在最终的业务主脚本中，直接抛弃 Action 机制，利用 Python 的 `arm_group.plan()` 纯手工剥离出轨迹数组，利用我们自己的逻辑直驱底层话题！

## 第3章 运动学陷阱与 IK 逆解的极限拉扯

> **🔴 踩坑史章节**：本章记录 v1～v5 迭代中与 IK、笛卡尔、OMPL 预瞄相关的**历史惨案与中间方案**。文中出现的原厂 `/ik/two_arm_hand_pose_cmd_srv`、`force_straight=True`、OMPL 降落预瞄等，**均非当前 V6/V7 主路径**——最终收敛见 **5.3** 与 **9.1**。

在利用“非 Action 直驱法”打通了物理连通性之后，真正的恶梦才刚刚开始。机器人的胳膊终于能动了，但它动起来的样子却像个失控的疯子。本章记录了我们在“姿态连贯性”和“直线插值”上与底层算法进行的极限拉扯。

### 3.1 7 自由度冗余臂的“IK 精神分裂症”与巨大钟摆灾难

人类的手臂即使握住同一个杯子，手肘也可以上下自由翻飞，这就是“运动学冗余”（Redundancy）。Kuavo 4 Pro 的单臂拥有 7 个关节，完美继承了这种冗余特性，但也由此引发了毁灭级的控制灾难。

#### 3.1.1 灾难复现：手肘异常翻折、姿态重构大甩尾与水瓶被无情砸飞

* **现象记录**：在执行“预瞄点降落 -> 笔直切入终点锁死水瓶”这最后 12cm 的关键进攻动作时。理论上，夹爪应该像平推抽屉一样，维持当前的四元数姿态，仅在 X 轴和 Y 轴进行纯粹的平移。
* **惊恐瞬间**：实际运行中，机器人的夹爪虽然向前移动了，但它那几十斤重的钢铁大臂，突然在半空中以极快的速度、猛烈地扭转了近 30 度！手肘瞬间从“微抬”状态生硬地翻折成了“朝下”状态。这个巨大的“姿态重构大甩尾”导致夹爪在空间中划出了一道极具破坏力的圆弧钟摆，**直接一巴掌把桌上的美年达水瓶无情地拍飞到了地上**！

#### 3.1.2 根因剖析：MoveIt 通用求解器 vs 原厂定制 IK 的多解性冲突（v2～v4 历史）

> **历史背景**：此问题出现在我们曾**混用** MoveIt OMPL 预瞄 + 原厂 `/ik/two_arm_hand_pose_cmd_srv` 切入的 v2～v4 阶段。当前稳定版已**统一为 MoveIt `/compute_ik` 单大脑**，不再调用原厂 IK 服务。

这就是我在研发日志中定义的“IK 精神分裂症”。由于我们在同一个抓取流程中，混用了两套截然不同的运动学大脑。

* **大脑的割裂**：
* **第 2 步（预瞄点降落，MoveIt 掌权）**：我们将目标坐标交给 MoveIt! 的 OMPL 规划器去寻路。MoveIt 内置的 TRAC-IK/KDL 求解器在计算预瞄点时，在无穷多种 7 轴解里，纯粹基于数学收敛性，随意挑了一个“手肘微抬”的解算姿态。
* **第 3 步（终点切入，原厂 IK 掌权）**：为了追求夹爪夹取的极致精准，我们在代码中调用了乐聚原厂的逆解服务 `/ik/two_arm_hand_pose_cmd_srv`。原厂工程师在这个 C++ 服务里，硬编码加入了“手肘尽量下垂”的拟人化权重，即我们每次请求必传的 `official_seed`：
```python
# 原厂服务请求构造：强制下垂的初始 Seed
ik_req.use_custom_ik_param = True
official_seed = [0.0, 0.0, 0.0, -1.57079633, 0.0, 0.0, 0.0] # 关键的 -90度 肘部约束
ik_req.hand_poses.left_pose.joint_angles = official_seed

```

* **物理必然性**：从第 2 步的“微抬”到第 3 步的“下垂”，虽然空间中只发生了 12cm 的末端直线平移，但在底层的 14 个电机眼里，却接到了大臂需要转动几十度的致命命令。电机拼命执行插值，导致手臂整体像一个大钟摆一样横扫了整个桌面！

### 3.2 MoveIt 笛卡尔直线（Cartesian Path）的底层 C++ API 雪崩

为了解决上述的“精神分裂”，我们试图让单一的大脑（MoveIt）统一接管这最后 12cm 的轨迹。既然要走直线，MoveIt 官方提供了一个看似完美的 API：笛卡尔直线路径规划（Cartesian Path）。

#### 3.2.1 强迫 MoveIt 走直线的尝试：调用 `compute_cartesian_path`

我们在代码中写入了最正统、全网 ROS 教程都在教的调法，期望它能在维持当前四元数的前提下，输出一条 100% 完美的空间直线。

```python
# 设定步长 0.01m，跳跃阈值 0.0
(plan, fraction) = arm_group.compute_cartesian_path([target_pose.pose], 0.01, 0.0)

```

#### 3.2.2 彻底崩溃：`Boost.Python` 签名绑定 Bug 引发的 `SIGSEGV` 闪退

* **报错还原**：只要 Python 解释器执行到这一行代码，终端会瞬间爆出大片红字，并直接导致进程崩溃闪退：
```text
Traceback (most recent call last):
 File "src/demo/vla_grasp/moveit_auto_grasp.py", line 114, in smart_execute
 (plan, fraction) = arm_group.compute_cartesian_path([target_pose.pose], 0.01, 0.0)
Boost.Python.ArgumentError: Python argument types in MoveGroupInterface.compute_cartesian_path(MoveGroupInterface, list, float, float)
did not match C++ signature:
 compute_cartesian_path(moveit::planning_interface::MoveGroupInterfaceWrapper {lvalue}, boost::python::list, double, bool, moveit::py_bindings_tools::ByteString)

```

* **深度抓虫**：这不是我们参数传错了（如你所见，传的是 List 和 Float）。这是 ROS1 Noetic 版本下一个极其臭名昭著的底层 Bug。MoveIt! 的内核是严谨的 C++ 开发的，通过 `Boost.Python` 库封装暴露给 Python 端。由于 C++ 底层期望的是 `std::vector<geometry_msgs::Pose>`，而 Python 传递的是解释型的 `list`，在特定的 GCC 编译环境或特定数据结构下，类型推导出现了内存错位（指针悬空），直接引发底层抛出参数类型不匹配，严重时甚至触发 `SIGSEGV`（内存段错误）。在封闭的真机环境包下，这几乎是一个无解的死局。

#### 3.2.3 壮士断腕：彻底废除 Python 笛卡尔包装类的工程决断

在反复尝试重装库、降级 Python 甚至修改 MoveIt 源码均告失败后，作为系统架构师，必须当机立断：**在最后 12cm 的进攻距离内，彻底将 MoveIt! 踢出核心控制流，绝不使用其包装好的残废 Python API，绝不允许业务流因为库 Bug 而闪退！**

### 3.3 退桥护盾与逆推法（Backward Planning）：纯数学线性贯穿引擎的涅槃

既然不能靠框架，那就靠自己手搓底层数学引擎。我们将老版本（纯硬编码版）中百发百中的核心逻辑强行平移过来，并与 MoveIt 架构完美融合，诞生了这套彻底终结报错的“退桥护盾直通引擎”。

#### 3.3.1 预瞄走廊的建立：利用 `math.hypot` 倒推 12cm 切入向量

为了根治“IK 精神分裂症”，我们要确保预瞄点和抓取点这 12cm 的头尾两端，**必须由同一个 IK 求解器在同一次运算中得出**。

* **数学逆推逻辑**：
我们不再让 MoveIt 瞎猜预瞄点，而是在开局阶段，就算好抓取终点（`q_grasp`）——v3～v4 曾用原厂 IK 服务，**V6 最终版统一改为 MoveIt `/compute_ik`**。随后，利用 Python 的原生数学库，在 XY 平面上计算绝对直线距离，基于预瞄常量 `PRE_GRASP_DIST = 0.12`，强行按比例后退，算出预瞄点的绝对坐标。
```python

ratio = (dist - PRE_GRASP_DIST) / dist
pre_x = shoulder_x + (locked_x - shoulder_x) * ratio
pre_y = shoulder_y + (locked_y - shoulder_y) * ratio

```

```python
q_pre = _solve_pose_ik(ik_client, arm, is_left_arm, pre_pose, q_grasp, f"[{side}手] 退后预瞄 12cm")
if q_pre is None:
    q_pre = q_grasp

```

这样求出的预瞄点关节角（`q_pre`），在物理姿态上与抓取点高度一致，手肘被完全“锁死”，绝不会发生翻折。

## 第4章 时间轴崩溃与物理防撞护盾

> **🔴 踩坑史章节**：本章记录 OMPL + `smart_execute` 早期在时间参数化（TOTG）上的崩溃与修复探索。**当前 V6 经典版抓取主路径不调用 `smart_execute`**；该函数仍保留于源码，供 OctoMap 版阶段 B 收手及异常 OMPL 路径使用。

在解决了前期的“IK精神分裂”与“API雪崩”后，我们的程序从逻辑和空间几何上已经无懈可击。然而，当我们将规划好的完美 3D 避障路径下发给真实的 Kuavo 14-DOF 机器人时，系统却在“时间域（Time Domain）”上发生了极其惨烈的物理崩塌。本章将详细记录这场与时间、惯性以及系统底层抖动进行的残酷搏斗。

### 4.1 OMPL 轨迹的时间参数化（TOTG）退化灾难

MoveIt! 的 OMPL 算法（如 RRTConnect）本质上只负责解决纯粹的空间几何问题（即避开桌子、书本，算出一系列相连的空间点）。为了让这些点变成机器人电机可以执行的运动，MoveIt! 在后台会隐式调用一个名叫 **TOTG (Time-Optimal Trajectory Generation，时间最优轨迹生成)** 或迭代抛物线法的后处理插件，为每一个点赋予一个时间戳（`time_from_start`）。正是这个隐式的后处理插件，差点毁了我们的实体机器人。

#### 4.1.1 惊魂时刻：0.00s 狂暴轨迹导致机械臂重砸桌面

* **灾难重现（现象）**：
在实机运行到“降落预瞄点”阶段时，机械臂本应该像打太极一样柔顺地绕开障碍物。然而，我们亲眼目睹了极其恐怖的一幕：几十公斤重的重型双臂突然像发了疯一样，以肉眼难以看清的速度，在不到半秒的时间内全速挥下！巨大的物理惯性导致电机根本无法刹停，夹爪带着恐怖的动能，重重地砸在实木桌面上，发出了巨大的撞击声，甚至差点引发机器人本体的侧翻。
* **深挖日志（抓虫记录）**：
为了查明这种“自杀式打拳”的原因，我们拍下急停按钮，死死盯住终端打印的日志时间戳。日志显示：
```text
[INFO] [1781600037.944206]: 🧠 MoveIt 避障推演开始: [降落预瞄点]...
[INFO] [1781600038.240384]: 🎉 [降落预瞄点] 避障轨迹完美执行完毕！

```

**发现端倪**：`38.240 - 37.944 = 0.296` 秒！MoveIt 为一条包含 26 个高密度避障航点的复杂 3D 曲线，居然只分配了不到 0.3 秒的执行时间。
* **恶化至极点（0.00s 绝境）**：
当我们试图在代码中加入“时间膨胀系数（Stretch Factor）”去强行拉长这 0.3 秒时，系统直接滑向了更加深渊的 Bug。终端抛出了令人毛骨悚然的日志：
```text
[WARN] 🛡️ 触发时间膨胀！原始狂暴耗时 0.00s -> 强制拉长至温和耗时 0.00s

```
整个轨迹的总时间彻底变成了绝对的 `0.00` 秒。Python 代码在一个微秒内，把几十个点像机枪扫射一样，毫无保留地全部塞给了底层电机的驱动板！

#### 4.1.2 数学溢出根因：极低限速（0.02）与高密避障点的极限重叠

为什么 MoveIt! 这样一个成熟的框架，会吐出 `0.00s` 这种违背物理常识的时间戳？这源于我们在上一阶段埋下的一个“安全陷阱”。

* **安全限速的副作用**：为了防止开局调试时机器人的速度过快，我们在代码初始化部分加入了极端的速度封印：
```python
left_arm.set_max_velocity_scaling_factor(VELOCITY_SCALING)  # 当前仓库 0.35
left_arm.set_max_acceleration_scaling_factor(ACCELERATION_SCALING)

```
* **八叉树点云的极致压缩**：由于我们引入了高频的实时 3D 避障（OctoMap），OMPL 为了在这团密集的点云中找出一条绝对安全的缝隙，解算出了极度密集的航点（例如在 1 厘米的物理空间内塞入了 3 个航点）。
* **TOTG 解算器的数学崩溃**：MoveIt! 后台的 TOTG 算法在处理这些间距只有几毫米的航点，且还要受到 `0.02` 这种极低加速度阈值约束时，由于非线性微分方程的数值跨度过小，**系统内部直接发生了 IEEE 754 浮点数下溢（Underflow）或除零异常**。
* **MoveIt 的摆烂机制**：最致命的是，当发生这种时间解算崩溃时，MoveIt! **并不会报错拒绝，而是依旧亮起绿灯返回 `SUCCEEDED`，但它会把轨迹里所有航点的时间戳（`time_from_start`）全部简单粗暴地填充为 `0.0s`**！导致上层应用被彻底欺骗。

### 4.2 时间剥夺与强制物理节拍器（Time Dilation）

在看透了 TOTG 算法的脆弱性之后，我们制定了铁血的架构级对策：**在代码的执行层，彻底废除和剥夺 MoveIt! 对于时间的决定权。我们自己当上帝，强行重构整条轨迹的时间轴！**

#### 4.2.1 放开 OMPL 速度封印，回归纯几何规划

既然压低 MoveIt 的速度上限会导致其内部数学解算崩溃，那我们的第一步操作就是**彻底解开 MoveIt 内部的速度封印**。

* **代码修改逻辑**：在调用 `arm_group.plan()` 进行规划之前，将速度和加速度系数强行拉回满血状态的 `1.0`。
* **原理**：此时，我们只是把 MoveIt! 当作一个纯粹的“几何计算器（Geometric Path Planner）”。由于参数恢复正常，TOTG 求解器不再发生下溢，它能顺利且极速地吐出完美避开障碍物的那 20~30 个纯空间坐标点，不再引发 0.0s 异常。

#### 4.2.2 强制时钟再分配：将安全时长（3.5s）均匀切片给 N 个航点

得到纯净的空间航点后，我们抛弃了轨迹中附带的全部时间信息，编写了一个名为“物理节拍器（Physical Metronome）”的核心引擎。

* **算法思路**：我们根据每一个动作的危险程度，人工指定一个绝对安全耗时（`total_safe_time`，例如退回护胸用 3.5 秒，切入水瓶用 2.5 秒）。然后，我们数一数 MoveIt 吐出了多少个航点（`num_points`）。我们将总时间均分为 `dt`，强制命令底层的电机，必须精准消耗 `dt` 秒才能从当前点移动到下一个点！
* **引擎代码级重构（片段剖析）**：

通过这套“上帝剥夺”机制，不管 MoveIt 算出来的点有多密集，它都被强行套上了一个绝对均匀的时钟节拍，机器人的机械臂从此变得像打太极拳一样从容不迫，0.0s 砸桌子的历史惨剧被彻底封印。

### 4.3 “一卡一卡”的非实时抖动与终极丝滑方案

在挂载了“强制物理节拍器”之后，机械臂确实不再暴冲了。但随后我们立刻遭遇了机器人运动学中极其经典的“幽灵现象”——机械臂在执行这 3.5 秒的轨迹时，表现得就像播放定格动画一样，浑身抽搐、“一卡一卡（Jitter/Jerk）”地往前探。它失去了老版本代码中那种浑然一体的丝滑感。

#### 4.3.1 操作系统局限：Python 软时钟与 ROS 网卡的 Jitter（网络抖动）

为什么我们在 Python 里写了精确的 `rospy.sleep(dt)`，下发的数据依然会让电机抽搐？

* **非实时操作系统的原罪**：我们的代码跑在 Ubuntu 20.04 上，这是一个通用的分时操作系统（CFS 调度），而不是 RTOS（实时操作系统）。当我们调用 `sleep(0.01)` 时，系统线程调度器无法保证恰好在 10ms 时唤醒线程。实际执行起来，可能这一帧睡了 4ms，下一帧由于系统在处理后台事务，被迫睡了 16ms。
* **TCP/IP 协议栈的传输抖动（Jitter）**：我们的 Python 节点与下位机底层的 C++ 驱动节点虽然在同一台 NUC 上（或者通过局域网），但 ROS 的 Publisher 走的是 TCP 协议栈。数据包会经历封包、网卡排队、拆包的过程，到达电机的时间间隔是极度不均匀的。
* **物理表现**：高频、间距极小但时间戳抖动的数据流灌入底层控制器时，底层的 PID 环刚准备平滑执行上一帧，下一帧突然提前或延后“砸”了过来。导致电机的力矩（Torque）输出发生高频突变（Jerk Spike），反映在几十公斤的钢铁手臂上，就是“一卡一卡”的颤振现象。

#### 4.3.2 方案确立：全量航点泵送交由下位机 1kHz WBC 三次样条插值接管

既然 Python 的软时钟不可靠，我们就必须把“微观上的平滑插值任务”下放给拥有工业级强实时性（Hard Real-Time）的底层硬件去处理。

* **唤醒原厂 WBC 的力量**：乐聚下位机系统内嵌了一个运行在 C++ 实时线程下的全身体扭矩控制器（WBC, Whole Body Control）。这个控制器拥有高达 1000Hz（1毫秒一次）的死循环时钟，极其擅长做三次样条（Cubic Spline）或五次多项式平滑。
* **接口的巧妙利用**：原厂提供的 `/kuavo_arm_target_poses` 话题不仅能接收角度，它内部其实是一个**自带时长（`times=[dt]`）的物理航点接收器**。
* **终极合龙方案（全量泵送）**：
我们不再依赖 Python 的时钟去控制电机的瞬时转速，而是把 MoveIt 吐出的 100% 全量航点（例如 24 个避障点），依次泵入该话题，并且在每一个包里，**明确告诉底层 WBC：“请用你自己 1kHz 的时钟，花费严格的 `dt` 秒（例如 0.14秒），匀速平滑地滑向这个角度！”**
*（注：在此期间我们曾犯过一个错，为了减少抖动，试图只抽稀提取 4 个点发送。结果导致了严重的“偷工减料悖论”——手臂把避障圆弧切成了直线，提前撞飞了水瓶。所以最终的铁律是：**绝对不能偷工减料抽稀，必须 100% 全量航点无缝泵送**！）*

当这一套“Python 算全量避障几何路径 + 强制均分切片时长 + 底层 1kHz WBC 接管样条滑行”的混合引擎彻底合龙后，奇迹发生了：
无论空间避障路径多么复杂扭曲，Kuavo 4 Pro 的重型双臂都展现出了如同太极大师一般丝丝入扣、极度流畅的完美滑行姿态。至此，我们在时间域上的底层战争，宣告全面大捷！

## 第5章 混合架构的诞生：宏观避障与微观盲插

> **🔴 踩坑史 + 🟡 进化史章节**：5.1～5.2 描述 v3～v5 曾尝试的「OMPL 宏观 + 末端盲插」六步状态机。**5.3 才是当前 V6/V7 稳定主流程**，请与 0.8、9、10 对照阅读。

当我们彻底解决了时间轴抖动与 0.00s 崩溃 Bug 后，我们原以为可以靠着 MoveIt! “一招鲜吃遍天”，让它包揽从避障到抓取的全部动作。但现实的物理世界很快给了我们沉重的一击。本章记录了我们在面对“点云遮挡死锁”时，是如何打通任督二脉，设计出这套兼具 MoveIt 避障智商与底层硬编码执行力的工业级混合架构的。

### 5.1 为什么最前沿的具身抓取都需要“接管模式”？

在机器人学中，运动规划分为“无接触自由空间导航”和“富接触（Contact-rich）操作”。MoveIt! 的 OMPL 算法是前者的王者，但在后者面前却是个不折不扣的“怂包”。

#### 5.1.1 OMPL 的“怂”：被水瓶自身点云吓死（前方被完全堵死报错）

* **惊恐复现**：当机械臂在 MoveIt 的引导下，完美地绕开了桌上的书本（阶段A），稳稳停在距离水瓶 12cm 的预瞄点时。我们向 MoveIt 下发了最后一步指令：`arm.set_pose_target(grasp_pose)`，命令它向前推进 12cm 去握住水瓶。
* **日志惨剧**：终端瞬间爆出刺眼的红字：
```text
[WARN] [1781601389.608303551]: Fail: ABORTED: TIMED_OUT
[ERROR] [1781601389.608557]: ❌ 路线推演失败！前方被完全堵死！

```

* **深度根因剖析（OctoMap 的悖论）**：
为什么会这样？因为我们开启了深度相机的点云感知。相机忠实地把眼前那个橘色的美年达水瓶扫描成了成千上万个 3D 点，并喂给了 MoveIt 的 OctoMap（八叉树碰撞树）。
在 OMPL 规划器的眼里，**它根本不知道那是你要抓的“目标”，它只认为那是一团不可逾越的“高密度实体混凝土柱子”！**
当你命令夹爪（同样具有物理体积）去“穿透”这根柱子时，MoveIt 的碰撞检测算法（FCL）会立刻判定为严重的“物理自残穿透行为”，于是它直接拒绝规划，抛出超时与堵死报错。
* **偷工减料悖论的教训**：在我们试图用 OMPL 强行规划，并用粗糙的 `np.linspace(0, num_points - 1, 4)` 进行轨迹抽稀时，还曾导致 MoveIt 为了绕开水瓶的点云边界，强行在半空中画出一个巨大的外扩弧线，直接从侧面把水瓶扇飞的惨剧。

#### 5.1.2 黄金分割方案（v3～v5 历史）：大范围 OMPL + 末端 12cm 盲插

面对这种“目标即障碍”的悖论，工业界的标准做法通常非常繁琐（需要用 YOLO Mask 滤除水瓶点云，或挂载 Allowed Collision Matrix）。但为了追求极致的执行效率，我们确立了一套被称为“黄金分割接管模式”的混合架构。

* **自动驾驶的隐喻**：这就像是特斯拉的 FSD。在复杂的马路上（避开书本、杂物），我们开启自动驾驶（MoveIt! 的 OMPL 算法），它能灵敏地躲避一切障碍；但当车要倒进一个极其狭窄的停车位（雷达会因为离墙太近而疯狂报警死锁）时，老司机果断关掉自动驾驶，切回“手动接管模式”，一脚油门强行倒进去！
* **架构解耦**：
* **宏观域（大于 12cm 的空间转场）**：100% 交给 MoveIt!。利用 OMPL 的高维空间寻路能力，画出优美的太极避障弧线。
* **微观域（最后 12cm 的致命一击）**（历史 v3～v5）：封杀 OMPL，曾用原厂 IK 或 `force_straight` 流式插值。**V6 最终版**改为 MoveIt `/compute_ik` + `execute_single_pose` 单段插入（仍不走 OMPL）。

### 5.2 【历史】v3～v5 六步状态机业务流（OMPL 预瞄时代）

> ⚠️ **以下为历史中间态**，其中步骤 0～2、5～6 的 OMPL/`smart_execute` 描述**不等于当前实机主路径**。现行流程见 **5.3**。

在这套混合架构的思想指导下，我们重构了主脚本 `moveit_auto_grasp.py`（以及点云进阶版 `moveit_octomap_grasp.py`），将其切割为钢铁般严密的“终极六步状态机”。

#### 5.2.1 步骤 0~1：非活动臂冻结与工作臂安全护胸归位

在开局阶段，不仅要锁定目标，更要重置机器人的全局姿态，以应对上一次可能失败崩溃留下的残局。

* **步骤 0（防摔重置，历史写法）**：曾用 `set_named_target` + `smart_execute` 收回闲置臂。**当前 V6** 改为启动时 `execute_dual_arm_init_home()` + 全程 `_freeze_inactive_arm()`（详见 6.2.2）。
* **步骤 1（工作臂待命）**：同样利用 MoveIt 的 3D 寻路，让准备抓取的工作臂（从任意姿态）柔顺地收回到胸前护胸姿态，为接下来的避障下探腾出完整的规划空间。

#### 5.2.2 步骤 2：OMPL 宏观 3D 避开书本降落至预瞄点

这里是 MoveIt! 大展神威的舞台。

* **目标计算**：基于视觉提供的绝对坐标 `[locked_x, locked_y]`，结合勾股定理倒推 12cm。
* **执行器调用**：
```python
arm.set_pose_target(pre_pose)
# 这里的 smart_execute 内部会调用 arm_group.plan()
smart_execute(arm, arm_pub, "降落预瞄点", total_safe_time=3.5)

```

* **物理表现**：机械臂像打太极一样，划出一道极度流畅、毫无抖动的大圆弧，精准地贴着障碍物外侧 1 厘米划过去，稳稳悬停在预瞄点。

#### 5.2.3 【历史】步骤 3~4：force_straight 流式插值盲插 12cm 并闭合夹爪

这是“接管模式”的核心。

* **步骤 3（盲插切入）**：我们彻底抛弃了 `arm_group.plan()`，也抛弃了那全是 Bug 的 `compute_cartesian_path` 函数。直接启动我们手搓的“直线贯穿引擎”（`force_straight=True`）。
```python
# 直接传入预算好的 q_grasp 终点关节角，强行启动纯数学底层插值
smart_execute(arm, arm_pub, "夹爪切入", total_safe_time=2.5, force_straight=True, fallback_q=q_grasp)

```

此时控制台打印黄字警告：`⚠️ 启用直线强制贯穿模式 (无视点云报警，无视 C++ Bug)！`。手臂如同无视物理法则一般，穿透瓶身点云，笔直推入。
* **步骤 4（闭合夹爪）**：调用原厂夹爪服务 `call_leju_claw([90.0, 10.0], ...)`，以满力矩死死卡住水瓶。

#### 5.2.4 【历史】步骤 5~6：force_straight 抬升与 OMPL 收手

* **步骤 5（垂直拉升）**：依然保持接管模式（`force_straight=True`），垂直拔高 20cm 或 22cm（`LIFT_HEIGHT`），强行将水瓶带离桌面复杂的点云乱局。
* **触发净化机制（历史写法）**：曾在抬升后 `clear_octomap`。**当前 V7** 改为在**水平插入前**调用 `clear_octomap_cache()`（插入前洗地，见 7.3.1）。
* **步骤 6（绕行撤退）**：再次把控制权交还给 MoveIt!（重启 OMPL）。命令其返回 `ready_pose_name`。机械臂带着水瓶，在空中自动规划出一条绕开原本桌面障碍物的折返曲线，完美收工！

### 5.3 【最终完美版】V6/V7 当前稳定抓取主流程

> **🟢 实机按此执行**。经典版 `moveit_auto_grasp.py` 与点云版 `moveit_octomap_grasp.py` 的**阶段 A（抓取段）完全相同**；差异仅在收手（阶段 B）。

#### 5.3.1 经典版 V6 完整状态机

## 第6章 双机协同与物理边界微调

> **🟡 进化史 + 🟢 最终参数**：6.1 TCP 标定为**当前仍在用的最终参数**；6.2～6.3 含踩坑过程，**6.2.2 已更新为现行防摔机制**。

“所有的仿真都是骗人的。” 当我们把完美的算法投射到现实中时，迎面撞上的是机械齿轮的公差、电机自重导致的形变，以及整机重心的灾难性偏移。本章记录了我们如何用代码作为“游标卡尺”，一点点修复物理世界的瑕疵。

### 6.1 机械公差的无底洞：TCP（工具中心点）分参标定

Kuavo 4 Pro 搭载的灵巧夹爪，在其默认安装状态下，原本是水平捏合的。但为了完成竖直抓取水瓶的任务，我们将夹爪在手腕处强行通过软件定义旋转了 90 度（Pitch/Roll 偏转）。这一物理操作，打开了潘多拉的魔盒。

#### 6.1.1 夹爪旋转 90 度的几何异变：上下齿轮距转化为左右偏心

* **几何灾难**：在原厂的 URDF 运动学模型中，工具中心点（TCP，`end_effector`）是定义在两根夹爪的正中心。由于夹爪的机械传动结构并非绝对对称（上下齿条有一定的间距），当我们把它侧翻 90 度后，原本垂直方向的“齿轮错位差”，瞬间变成了水平方向（Y 轴）的“左右偏心误差”！
* **实测现象**：当我们以为坐标绝对精准时，代码一执行：
* 用右手去抓，夹爪总是死死地撞在水瓶的**偏右侧**。
* 用左手去抓，夹爪却又诡异地撞在水瓶的**偏左侧**。
这说明单一的补偿值已经彻底失效了！

#### 6.1.2 左右手非对称误差修复（`TCP_OFFSET_Y_LEFT` vs `RIGHT`）

我们被迫化身为“人肉卡尺”，开始了长达数小时、多轮次的极限标定实验。

* **标定编年史**：
* 第一轮：尝试全局 `Y = 0.03`，结果依然偏前偏右。
* 第二轮：尝试将 X 轴拉回 `X = -0.010`，右臂 `Y = 0.052`，结果仍有少许偏移。
* 第三轮极限微调：经历了无数次微调甚至文件被误改后，我们彻底确立了“左右手独立分参”的绝对真理架构。

* **终极物理补丁参数（当前仓库 `moveit_auto_grasp.py`）**：
```python
# X轴（深度）
TCP_OFFSET_X_LEFT = -0.013
TCP_OFFSET_X_RIGHT = -0.022

# Y轴（横向偏移）：左右手镜像非对称补偿
TCP_OFFSET_Y_LEFT = 0.004   # 左手偏左，补偿值减小
TCP_OFFSET_Y_RIGHT = 0.065  # 右手偏右，补偿值拉大

LIFT_HEIGHT_FALLBACKS_M = (0.22, 0.18, 0.14, 0.10)  # 抬升 IK 递减重试
```

> 上文「第三轮」标定历程中的 `-0.018` / `0.020` 等为历史中间值，保留作演进记录。

### 6.2 质心越界与防摔亡羊补牢

这是整个项目开发周期中，最令人心脏骤停、冷汗直冒的时刻。我们差一点就亲手毁掉了这台价值不菲的设备。

#### 6.2.1 惊险复现：双臂同时前伸导致重心（CoM）前倾摔倒危机

* **灾难重现**：在一次频繁的代码重启测试中。上一次运行因为 Bug 崩溃了，**导致左臂正死死地平举伸在机器人正前方**。此时我修改了代码重新发车，系统判定水瓶在右侧，于是**指挥右臂也开始全速向前方探出**。
* **物理常识的惩罚**：
“双臂同时前伸”，这在人形机器人控制中是极其致命的工程禁忌！两根重达十几公斤的机械臂同时指向正前方，产生了极其恐怖的力矩杠杆。机器人的全局质心（CoM, Center of Mass）瞬间被生生拖拽，超出了其脚掌构成的“支撑多边形（Support Polygon）”边界！
* **惊魂一刻**：在一旁盯着电脑的我（在终端报错：`第一，这次动的是右手，左手还是举着没回来，而且两个手都往前伸直接让机器人整体往前倾`），眼睁睁看着这台一米多高的钢铁巨兽身体猛然向前倾倒！幸亏反应极快，一巴掌拍死了急停按钮，并在最后一刻用身体扛住了倾倒的机身，避免了粉身碎骨的惨剧。

#### 6.2.2 防御机制演进：从 OMPL 收闲置臂 → `_freeze_inactive_arm()`（当前稳定版）

痛定思痛，这是因为早期代码**完全遗漏了对非活动臂的状态保护**！防御策略经历了两代演进：

| 阶段 | 做法 | 状态 |
| --- | --- | --- |
| v3～v5 历史 | 步骤 0：`set_named_target` + `smart_execute` 让闲置臂 OMPL 回 `ready` | 已废弃 |
| **V6 当前** | 启动 `execute_dual_arm_init_home()`；IK/下发时 `_freeze_inactive_arm()` 锁非活动臂为 init | **主路径** |

* **现行核心逻辑**：每次 IK 求解与 `execute_single_pose` / `publish_arm_trajectory_batch` 下发前，非活动臂 7 个关节**强制覆写**为 `DUAL_ARM_INIT_DEG` 对应弧度，杜绝「一手前伸、另一手再探」的质心前倾。
* **救命代码片段（现行）**：
```python
def _freeze_inactive_arm(joints_14_rad, is_left_arm):
 frozen = np.copy(joints_14_rad)
 init = _init_joints_rad()
 frozen[_inactive_arm_slice(is_left_arm)] = init[_inactive_arm_slice(is_left_arm)]
 return frozen
```
* **二次启动**：程序开头即 `execute_dual_arm_init_home()`；若上次异常退出，**须重新跑终端 8** 或手动发 init，勿假设手臂已归位。

### 6.3 异常中断的状态残骸清除

在实际操作中，因为各种网络或人工急停导致的程序异常中断是家常便饭。如果不能优雅地处理上一次失败留下的残局，机器人就永远无法具备真正的自动化能力。

#### 6.3.1 失败后前伸姿态的二次启动适应（从当前错误姿态自动规避退回）

* **用户疑虑**：如果在第三步盲插时中断了程序，机械臂现在正像僵尸一样平举伸在前方（甚至差点顶着水瓶）。下一次启动程序，它会不会直接从这个错误位置开始瞎抓？
* **MoveIt! 起点感知机制（Start State 动态获取）**：
这就是我们为什么要坚持下沉 MoveIt 架构的原因！只要下位机的 `/joint_states` 数据引渡神经元（`kuavo_state_publisher.py`）仍在后台高频运行，MoveIt! 就能实时掌握这残破的局面。
* **智能自适应回防（当前稳定版）**：
二次启动时，**第一步仍是** `execute_dual_arm_init_home()`（硬编码双臂 init，不依赖 OMPL）。若上次失败留臂前伸，init 归位会把双臂拉回安全下垂；随后曲肘护胸 → 预瞄 → 插入。`_safe_return_both_arms()` 异常路径同样回 init，而非 OMPL `ready`。
* **历史方案（v3～v5）**：曾依赖 OMPL 从「僵尸前伸姿态」规划退回 `ready`——该路径已被更简单可靠的 init 归位取代。

## 第7章 跨越静态障碍：引入 OctoMap 真实环境感知

> **🟢 最终完美版章节**：OctoMap 点云版（V7）从本章起与当前代码一致。

在此前的阶段（经典版 `moveit_auto_grasp.py`），机器人的避障仅仅依赖于我们在代码中写死的几何轨迹。然而，物理世界是动态且充满混沌的。如果桌面上随机出现了一个水杯、一摞书或者实验人员的手臂，硬编码的轨迹就会化身为一台无情的“物理破坏机”。为了让 Kuavo 4 Pro 拥有真正的“具身智能”，我们正式引入了深度相机的点云（Point Cloud）流，并在 MoveIt! 内存中实时构建出基于八叉树（OctoMap）的三维碰撞检测树。

### 7.1 从“硬编码L型走廊”到“动态 OctoMap 避障”的进化动机

在没有 OctoMap 之前，我们让机械臂收手不碰桌子的做法是极其原始暴力的：将肩膀强行平行外摆 75°，然后再曲肘收回（即代码中的 `execute_vla_style_return()` 函数）。这种“VLA外摆”策略虽然在空旷桌面上很有效，但它占用的横向空间极大，稍微狭窄的环境就会导致严重的侧向碰撞。我们需要 MoveIt! 能够“看见”障碍物，并在三维空间中动态“钻”过去。

#### 7.1.1 工业标准 MoveIt Pick 管线与本项目简化策略的抉择

在阐述需求并进行技术研讨时，我们梳理了工业界最标准、最正统的 MoveIt 抓取管线（MoveIt Pick Pipeline）：

1. **感知（Perception）**：深度相机向 ROS 吐出稠密点云。
2. **目标剔除（YOLO Mask Filtering）**：利用 YOLO 的分割掩码（Segmentation Mask），将属于“目标水瓶”的那些点从点云中剔除。
3. **建图（OctoMap Mapping）**：将滤除目标后的纯净环境点云喂给 MoveIt，构建仅包含障碍物的 OctoMap。
4. **进场（Approach & Grasp）**：OMPL 规划器在 OctoMap 中规划路径，夹住目标。
5. **附着（Attach Object）**：调用 `attachObject` 接口，告诉 MoveIt“这个水瓶现在是我手臂的一部分了，请把它加入碰撞体积计算”。
6. **撤离（Retract）**：带着被附着的水瓶，在 OctoMap 中规划安全撤离路线。

**本项目的极简降维策略（时机切片控制法）**：
我们评估后发现，要在边缘计算板（Orin NX）上实时做高频的点云 Mask 滤除，算力开销太大且延迟极高。为此，我们确立了一套“化繁为简”的偷鸡策略：

* **抓取阶段**：不依赖 OMPL，继续使用我们的老旧硬编码（单段 IK 插值）盲插水瓶。
* **收手阶段**：在抬升阶段，清空旧点云，然后满血开启 OctoMap 建图，让 OMPL 规划收手路径。
这套策略用“时间切片”完美避开了复杂的点云实时过滤运算。

#### 7.1.2 YOLO Mask 点云滤除的缺失与水瓶被判定为伪障碍的矛盾

采用上述极简策略，不可避免地会遇到一个核心矛盾：**目标本身就是最大的障碍物**。
深度相机会忠实地把你要抓的橘色美年达水瓶，扫描成一根由高密度点云构成的“3D 实体混凝土柱子”。如果不做 Mask 滤除，这个柱子就会死死地矗立在 OctoMap 的正中央。
**踩坑实录**：如果我们尝试在这个状态下，让 OMPL 规划最后 12cm 的切入动作，MoveIt 的碰撞检测引擎（FCL）会立刻报警。终端会抛出 `Fail: ABORTED: TIMED_OUT`，因为系统认为你试图把钢铁夹爪强行塞进一根混凝土柱子内部，这是严重的物理穿透与自残行为。这也正是为什么我们必须在抓取前执行“清空地图”或“强制盲插”的底层原因。

### 7.3 `clear_octomap` 的时机切片控制

OctoMap 就像机器人的“短期记忆”。如果记忆中存在一堆虚假的残影，机器人就会寸步难行。控制这块记忆的删除与写入时机，是我们绕开工业级复杂掩码过滤的终极杀招。

#### 7.3.1 进场阶段：清理残留伪障碍，防止 12cm 切入被拦停

在我们的机械臂向目标进发时，由于前期的移动或者周围人员的走动，OctoMap 里可能会积累大量的“幽灵点云（Ghost Points）”。更重要的是，如前文所述，水瓶自身也会在地图中形成伪障碍。

* **解决方案（`clear_octomap_cache` 函数的重构）**：
在代码中，我们必须在准备执行 12cm 盲插切入之前，向系统下发清空指令。
* **深层踩坑与 API 变异修复**：
最初，我们只是简单地调用 `rospy.ServiceProxy('/clear_octomap', Empty)()`。但这在某些版本的 MoveIt 中完全无效！经过排查发现，MoveIt 根据其命名空间配置的不同，清空服务可能挂载在不同的节点下。
于是我们在代码中加入了多重服务嗅探遍历机制：
```python
def clear_octomap_cache():
    """清除 OctoMap 伪障碍（瓶身点云等），兼容 MoveIt 不同命名空间。"""
    cleared = False
    for svc_name in OCTOMAP_CLEAR_SERVICES:
        try:
            rospy.wait_for_service(svc_name, timeout=0.5)
            rospy.ServiceProxy(svc_name, Empty)()
            rospy.loginfo(f"🧹 OctoMap 已清除: {svc_name}")
            cleared = True
            break
        except rospy.ROSException:
            continue
    if not cleared:
        rospy.logwarn("⚠️ 未找到 clear_octomap 服务，若规划报障碍可检查 move_group 是否加载传感器")
    rospy.sleep(0.15)

```

这一波多重嗅探，彻底保障了我们在盲插进场时，面对的是一片完全真空、无任何点云阻挡的干净空间。

#### 7.3.2 收手阶段：OctoMap 满血重启，实现针对动态杂物的绕行撤离

当机械爪死死卡住水瓶，并执行带载垂直抬升 22cm（`LIFT_HEIGHT`）脱离桌面后，真正的挑战才刚刚开始。此时我们需要带着水瓶撤退，而回退的路径上可能会有随机放置的课本或实验仪器。

这是全新 `moveit_octomap_grasp.py` 内部 `octomap_retract_after_lift()` 函数的巅峰表演时刻：

1. **第一步：刷新环境记忆**
```python
rospy.loginfo(f"\n🛡️ [{side}手] OctoMap 收手：等待点云刷新 {OCTOMAP_SETTLE_AFTER_LIFT}s ...")
rospy.sleep(OCTOMAP_SETTLE_AFTER_LIFT)

```

给相机留出 2 秒钟的时间，让它以 2Hz 的 `max_update_rate` 重新扫描当前桌面。此时，由于水瓶已经被我们拿到了高空，桌面上原本属于水瓶的伪障碍点云已经消失，而那些真实的书本障碍物点云会被稳稳地刻录进 OctoMap 碰撞树中。
2. **第二步：注入物理保护（桌面碰撞盒）**
因为我们之前饱受 URDF 缺失碰撞几何体（Collision Geometry）的毒打（见上一章），为了防止 OMPL 在规划后退路线时为了“抄近道”而让夹爪去“挖地道”砸毁实木桌面。我们在场景中强行硬核注入了一张由空气墙构成的物理实验桌（`DESK_BOX_ID`）。
```python
DESK_BOX_CENTER = (0.42, 0.0, 0.33) # 定义安全桌面的绝对物理中心
DESK_BOX_SIZE = (1.20, 0.90, 0.06) # 定义一张 1.2米 x 0.9米，厚6厘米的无敌空气墙
setup_desk_collision_box(scene_iface)  # moveit_octomap_grasp.py main()

```

3. **第三步：解锁 OMPL 与外扩避障**
在有了重新刷新的 OctoMap 和不可逾越的空气桌面后，我们彻底放开 OMPL 的锁链：
```python
octomap_retract_after_lift(
    arm, arm_pub, ik_client, is_left_arm,
    locked_x, locked_y, locked_z, quat,
    shoulder_x, shoulder_y, dist, side
)
# 内部：plan_and_execute_collision_aware + 失败 → mag.execute_vla_style_return() 

```

4. **第四步：容错降级降落伞**
如果在这一步 OMPL 因为某种奇怪的点云噪点或者 IK 奇异点报错说“规划失败”，我们的系统绝对不会死锁。代码中内置了绝对容错机制，一旦失败，立刻调用 `execute_vla_style_return()`，强行退回到经典版的“肩膀硬编码平行外摆 75°”撤退逻辑，保住重型设备的安全底线。

**实机成功日志特征展现**：
当以上所有机制齿轮咬合、完美运转时，终端中会极度舒适地打印出如下日志：

```text

🚀 阶段A [硬编码抓取] ...
🗺️ 阶段B：OctoMap + 桌面盒 碰撞感知收手...
```

## 第8章 视觉与点云流的底层排障实录（双机协作篇）

在成功构建了纯数学计算的盲插与避障框架后，我们正式向“真 3D 环境感知”发起冲锋。然而，从理论上的 OctoMap 到实机上那如泉水般涌动的百万级点云流，中间隔着无数个底层驱动的深坑。本章将详细复盘我们是如何利用双机协同工作流，一点点撕开底层的黑盒，最终驯服奥比中光（Orbbec）深度相机的。

### 8.1 悬案：话题存在但 `rostopic hz` 报 `no new messages`

当我们满怀信心地在下位机配置好 `sensors_3d_octomap.yaml`，将点云接收话题指向 `/camera/depth/color/points`，并启动 `moveit_octomap_grasp.py` 时，程序却直接卡死在了环境探测阶段。

#### 8.1.1 上下位机双机工作流的确立（突破工作空间隔离）

* **悬案初现**：为了查明为什么 MoveIt 收不到点云，我们在下位机（NUC）打开终端进行了基础诊断：
```bash
# 检查话题是否存在（结果：存在）
rostopic list | grep points 

# 检查点云发布频率（结果：无数据吐出）
rostopic hz /camera/depth/color/points 
# 输出: no new messages

# 对比检查原始深度流与 YOLO 视觉流（结果：均正常）
rostopic hz /camera/depth/image_raw # 显示 ~22 Hz OK
rostopic hz /vla/yolo_target # 显示 ~8.7 Hz OK

```

**结论极其诡异**：相机明明在正常工作，YOLO 也能跑，但点云话题就像一个“空壳”，被注册了却没有任何节点向其发送数据。
* **工作空间隔离的痛点**：要解决这个问题，必须去修改相机启动的 `launch` 文件。但是！我们当前的代码编辑器是挂载在**下位机**（`lab@192.168.26.1`，工作空间 `~/kuavo-ros-opensource`）上的，它根本看不到相机的底层驱动代码！
* **双机协同战术**：为了打破这层物理隔离，我们确立了高级的“双机工作流”：
1. **下位机**：保持连接 NUC，专门负责修改抓取脚本（`moveit_octomap_grasp.py`）和 MoveIt 的配置文件。
2. **上位机**：利用 Remote SSH 插件，新开一个窗口直接穿透登录到上位机视觉工控板（`leju_kuavo@192.168.26.12`，工作空间 `~/kuavo_ros_application`）。只有在这里，我们才能对相机的驱动参数进行开膛破肚。

#### 8.1.2 顺藤摸瓜：追踪 `load_robot_head.launch` 到 Orbbec 驱动深层

在上位机工作区中，我们按 **附录 C** 的排查要点，梳理了庞大的 launch 启动链路。

* **链路追踪**：
沿着终端 5 启动的入口文件一路向下深挖：
1. 入口：`src/dynamic_biped/launch/load_robot_head.launch`（带参 `use_orbbec:=true`）
2. 中继：`src/dynamic_biped/launch/orbbec_sensor_robot_enable.launch`
3. 核心：`src/OrbbecSDK_ROS1/launch/gemini_330_series.launch`

* **破案（根因暴露）**：
在最底层的 `gemini_330_series.launch` 源码中发现了一个致命的参数设定：**`enable_point_cloud` 默认值为 `false`**！此外，即便开启点云，该 SDK 默认吐出的话题名是 `/camera/depth/points`，而我们 OctoMap 以及重映射（remap）配置里死死咬住的是 `/camera/depth/color/points`。这就是为什么话题变成空壳的根本原因。

### 8.2 奥比中光（Orbbec）相机的 SIGSEGV 内存崩溃阻击战

找到了参数，本以为只需简单地把 `false` 改为 `true` 就能大功告成，但随之而来的，是一场灾难级的内存崩溃。

#### 8.2.1 错误尝试：开启 `enable_point_cloud:=true` + relay 转发导致多订阅压垮驱动

* **第一版妥协方案（Relay 转发）**：
为了不破坏 SDK 的底层逻辑，上位机侧给出了第一版修改方案。在 `orbbec_sensor_robot_enable.launch` 中强制开启点云，并增加一个 `topic_tools/relay` 节点进行话题名称转发：
```xml
<arg name="enable_point_cloud" value="true" />
<node pkg="topic_tools" type="relay" name="depth_points_to_color_points"
 args="/camera/depth/points /camera/depth/color/points" />

```

* **灾难发生（SIGSEGV 卡死）**：
代码修改后，我们兴冲冲地开始了实机测试。跑了 1 到 2 轮抓取后，上位机的终端 5（相机进程）突然全线崩溃。
终端疯狂输出令人绝望的底层内存错误：
```text
Received signal: 11 (SIGSEGV)
usbdevfs reset ...
boost: mutex lock failed in pthread_mutex_lock: Invalid argument
Received signal: 6 (SIGABRT)
[camera/camera-1] process has died [pid ..., exit code 11]

```

此时，所有相关的话题（包括 `image_raw` 和点云）全部陷入死寂（`no new messages`）。
* **崩溃根因（并发压垮驱动）**：
为什么会崩溃？因为我们在上位机同时跑着极其复杂的感知流。当把点云生成任务压在相机驱动内部时，YOLO 视觉节点、Apriltag 检测节点（连续检测）、再加上我们下位机 MoveIt OctoMap 跨越局域网来高频订阅点云。**多重高并发的订阅请求瞬间压垮了 Orbbec 驱动底层的线程池。** 随之引发的 USB 复位机制与崩溃线程产生了竞态条件（Race Condition），导致 Mutex（互斥锁）遭遇二次释放，最终以 `SIGSEGV`（段错误）惨烈收场。

#### 8.2.2 正确解法：关闭内置点云，利用 `depth_image_proc/point_cloud_xyz` Nodelet 外部生成

痛定思痛，我们决定**给相机驱动减负**，把点云生成的重算力任务剥离出来。

* **第二版稳定方案（Nodelet 外部解算）**：
我们在 `orbbec_sensor_robot_enable.launch` 中，又把 `enable_point_cloud` 关了回去！取而代之的是，我们引入了 ROS 官方极度稳定且高效的 `depth_image_proc` 图像处理管线，自己搭一个外部点云生成器。
* **终极稳定版 Launch 源码实录**：
```xml
<include file="$(find orbbec_camera)/launch/gemini_330_series.launch">
 <arg name="color_fps" value="30" />
 <arg name="enable_point_cloud" value="false" />
</include>

<node pkg="nodelet" type="nodelet" name="vla_pc_manager" args="manager" />

<node pkg="nodelet" type="nodelet" name="vla_depth_to_pointcloud"
 args="load depth_image_proc/point_cloud_xyz vla_pc_manager --no-bond">
 <remap from="image_rect" to="/camera/depth/image_raw" />
 <remap from="camera_info" to="/camera/depth/camera_info" />
 <remap from="points" to="/camera/depth/color/points" />
</node>

```
**大获全胜**：当这套架构推到上位机后，点云流像瀑布一样源源不断地跨越局域网涌向下位机的 OctoMap，不管跑多少轮抓取测试，相机进程再也没有报过哪怕一次 `SIGSEGV`。感知链路被彻底打通！

### 8.3 OctoMap 实机联调的琐碎 Bug 扫雷

感知链路虽然通了，但 MoveIt! 处理 OctoMap 的机制依然充满了“怪脾气”。在 `moveit_octomap_grasp.py` 的实测中，我们又连续扫除了两个隐藏极深的业务流地雷。

#### 8.3.1 `wait_for_octomap` 引发的 20 秒假死（改为 `probe` 非阻塞探测）

* **现象复现**：在启动 `moveit_octomap_grasp.py` 时，程序正常输出了 `[INFO] Ready to take commands for planning group...`，但随后脚本就陷入了长达近 20 秒的彻底假死状态。
* **挖除阻塞代码**：
最初，为了确保建图完成，我们在代码里写了类似 `wait_for_octomap(20)` 的死板逻辑，强行挂起并死等系统发布 `/octomap_full` 话题。
然而，MoveIt 的机制非常偷懒：**它通常只在内部内存里维护 OctoMap 碰撞树，如果不主动触发对外广播服务，它根本就不会往 ROS 总线上发 `/octomap_full` 话题**！这导致我们的 Python 脚本被活活饿死在等待队列里。
* **优雅解法（探针机制与降级）**：
我们废除了死等话题的逻辑，引入了 `probe_octomap_ready()` 非阻塞探测函数。并在开局增加了对底层点云源头的动态嗅探（`POINT_CLOUD_CANDIDATES`）。
只要我们在“阶段 B”尝试调用 OMPL 规划外移路线时能够迅速（如 2.5 秒内）返回成功，就足以反向证明 OctoMap 已经在内存中生效。如果探测点云话题长达 15 秒无响应，系统则立刻触发**降级模式**，切回经典版的“VLA肩膀硬编码外摆”逻辑，彻底杜绝了因点云卡死导致整个控制流宕机的风险。

#### 8.3.2 桌面安全盒（CollisionObject）的注入，防止点云缺失导致手砸桌面

* **隐患溯源**：在利用 OMPL 规划后退收手路径时（阶段 B），由于之前的 URDF 偷工减料缺失了物理模型的 collision mesh（见前文警告），MoveIt 其实并不知道自己的胳膊有多粗。更可怕的是，**桌子的平面往往因为相机的视角死角或反光，无法在 OctoMap 中形成完整的 3D 屏障**。如果任由 OMPL 发挥，它为了“抄近道”，极大概率会让夹爪“遁地”穿过桌面，造成严重的设备损毁。
* **物理防线（代码实录）**：
我们利用 MoveIt 的 `PlanningSceneInterface`，手写了一个 `setup_desk_collision_box()` 函数，在 OctoMap 启动前，强行在机器人的正下方灌入一个无敌的“空气实验桌”。
```python
def setup_desk_collision_box(scene):
 # 参数提取自附录 G.3
 DESK_BOX_CENTER = (0.42, 0.0, 0.33)
 DESK_BOX_SIZE = (1.20, 0.90, 0.06)

 desk_pose = PoseStamped()
 desk_pose.header.frame_id = "base_link"
 desk_pose.pose.position.x = DESK_BOX_CENTER[0]
 desk_pose.pose.position.y = DESK_BOX_CENTER[1]
 desk_pose.pose.position.z = DESK_BOX_CENTER[2]
 desk_pose.pose.orientation.w = 1.0

 # 强行塞入环境感知树

```
配合外扩容差参数 `AVOID_OUTWARD_Y = 0.10`，这张虚拟桌面盒成为了 Kuavo 双臂的最终底线。即使 OctoMap 的点云再稀疏，OMPL 规划器也绝对不敢跨越雷池半步，完美保障了昂贵硬件的物理安全。

## 第9章 终极双轨运行手册：经典版与点云版

在所有的底层通信、逆解算法（IK）与 3D 点云建图（OctoMap）全部被我们打通并驯服之后，项目进入了最高级别的工程固化阶段。为了兼顾“极致的速度稳定性”与“动态环境的高级避障能力”，我们并没有用新代码粗暴地覆盖老代码，而是确立了“双轨并行”的工业级产品发布策略。

本章将为您全景呈现这套系统的进化脉络，并为您深度解密 V6 经典版（`moveit_auto_grasp.py`）与 V7 点云版（`moveit_octomap_grasp.py`）的底层运行逻辑与实操避坑指南。

### 9.1 版本演进速查树（v0 ~ v7 架构对比）

从我们开始尝试用 MoveIt 接管 Kuavo 机器人到现在，代码经历了 8 次甚至可以用“惨烈”来形容的迭代重构。每一次大版本的跃迁，背后都伴随着终端里刺眼的红字报错、以及物理实体险些砸毁桌面的惊心动魄。

以下是我们从“一地鸡毛”到“登峰造极”的完整版演进史：

* **v0 原教旨主义期（纯 OMPL 全程接管）**：
* **架构**：全程依赖 OMPL 规划。
* **死状**：由于 OMPL 会走关节空间的最短路径，导致机械臂没有像人类一样水平推入，而是极其诡异地从下方斜插、甚至先抬高再前插，屡屡撞飞水瓶。时间轴更是出现 0.00s 狂暴下发，导致机械臂重砸桌面。

* **v1 闭门造车期（手搓 25 点 IK 插值）**：
* **架构**：抛弃 OMPL，试图在 Python 里自己切分 25 个密集航点，循环调用 `/compute_ik`。
* **死状**：由于发给服务端的 `RobotState` 只包含了手臂 14 轴，缺少全身参数，TRAC-IK 直接全线拒绝服务，报出 `26 航点全无解` 的惨剧。

* **v2 几何对齐期（加入 `end_effector`）**：
* **架构**：引入了双臂的 OMPL，并在 IK 请求中修正了末端连杆。
* **死状**：抓取时发现了高达 17cm 的 Z 轴误差，因为之前错误地使用了 `zarm_r7_link` 而非真正的 `zarm_r7_end_effector`。

* **v3~v4 官方 API 幻灭期（调用 `GetCartesianPath` 服务与单段插值）**：
* **架构**：试图利用 C++ 的笛卡尔直线服务，或者单段关节插值。
* **死状**：遭遇了 ROS1 Python3 环境下无解的 `Boost.Python` 签名绑定 Bug 引发的 `SIGSEGV` 闪退。彻底对官方高级封装死心。

* **v5 护胸重塑期（恢复曲肘护胸）**：
* **架构**：找回了“曲肘护胸”的中间防撞构型。曾一度为了减小冗余删除了曲肘步骤，结果实测时直接横扫碰翻了水瓶，于是紧急恢复。

* **v6 终极经典版（当前稳定主分支之一：`moveit_auto_grasp.py`）**：
* **架构**：融合了 v5 的流程，加入了极致的左右手 TCP 分离标定补丁。收手采用纯骨骼运动学的“vla 肩膀外摆”。
* **成果**：速度极快，百发百中，但在复杂桌面上是个“瞎子”。

* **v7 点云避障满血版（当前巅峰之作：`moveit_octomap_grasp.py`）**：
* **架构**：阶段 A 继承 v6 的盲插精准度；阶段 B 抬升后满血复活 OMPL + OctoMap 避障建图。
* **成果**：赋予了机器人真正的“三维视界”，能动态绕开收手路径上的所有临时障碍物。

> **进化史小结**：v0～v5 的每一次「死状」均保留于 2～6 踩坑史；**v6/v7 实机主路径详见 5.3**。

### 9.2 V6 经典版（`moveit_auto_grasp.py`）适用场景与原理解密

在研发末期，用户曾发出过直击灵魂的质疑：“跑 `moveit_auto_grasp.py` 这个程序的轨迹，跟咱们以前根本不用 MoveIt 的 TF2 老版本差不多啊，那 MoveIt 的价值在哪？它真的在用吗？”

这个疑问非常经典，下面我们将从代码逻辑底层彻底打破这个迷思。

#### 9.2.1 无点云环境下的纯骨骼运动学 VLA 外摆收手策略

V6 经典版是一套“不携带点云感知”的雷霆进攻代码。

* **场景定位**：适用于桌面仅有目标水瓶、没有复杂动态障碍物干扰的高通量抓取场景。它不需要消耗大量的 CPU 资源去维护碰撞八叉树。
* **配置特征**：在其对应的启动文件中（`move_group.launch`），所挂载的 `sensors_3d.yaml` 文件内容是空荡荡的 `sensors: []`。
* **撤退逻辑揭秘（`execute_vla_style_return`）**：
既然 MoveIt 在这个版本里看不见桌子，那它抓完水瓶后怎么收手才不会把桌子砸烂？答案是我们抄回了老版本 `vla_auto_grasp_daemon.py` 里的神级硬编码：**肩膀 75° 外摆策略**。
当夹爪闭合并抬升 22cm 后，程序不会去求助 OMPL，而是直接向底层的 `/kuavo_arm_target_poses` 强制下发一组关节指令。这组指令要求手臂的肩膀平行向外侧大幅度外摆（完全脱离桌面上方的垂直领空），然后再进行曲肘护胸收回。这种纯骨骼运动学的物理规避，极其粗暴但100%有效。

#### 9.2.2 “MoveIt 真在用吗？”——打破迷思，明确 OMPL 的生效区间

既然抓取是直插，收手是硬编码外摆，那 MoveIt 究竟在干什么？
在 V6 经典版中，MoveIt 并非不用，而是**被我们极其克制地当作了“高精度逆解算力引擎”**。

1. **URDF 的解析与守护**：MoveIt 在后台默默地加载了庞大的 14-DOF 机器人模型，并维护着 TF 树中 `base_link` 到 `zarm_r7_end_effector` 那无数个连杆之间的实时相对变换。
2. **提供 `/compute_ik` 最强算力**：这是核心！我们不再用简单的几何逼近，而是循环调用 MoveIt 提供的 TRAC-IK 求解器。在抓握、预瞄、以及抬升这三个最关键的点上，MoveIt 结合了我们注入的 `OFFICIAL_IK_SEED_7 = [0,0,0,-π/2,0,0,0]`（手肘强行向下的官方种子姿态），为我们算出了分毫不差、绝不甩尾的完美关节角。
3. **异常回防的后备大脑**：只有在发生异常状态（例如开局时手臂停留在诡异位置），系统调用 `arm.set_named_target("ready")` 时，MoveIt 的 OMPL 才会短暂苏醒，利用纯内部模型的自碰撞检测，将手臂柔顺地抽回胸前。

**结论**：在 V6 版中，MoveIt 藏起了它那不可靠的“碰撞避障锋芒”，甘心化身为一台隐形的、极度精准的底层数学计算器，铸就了这套系统的百发百中。

### 9.3 V7 点云版（`moveit_octomap_grasp.py`）部署指南

如果您希望赋予 Kuavo 真正的“具身 3D 感知”，让它像长了眼睛一样在杂乱的书堆中绕行，那就必须切换到 V7 点云版。但**这绝不是简单地换一个 Python 脚本就能跑通的**，它的上下游依赖有着极其严苛的绑定关系。

#### 9.3.1 必须配套启动的 `move_group_octomap.launch` 注意事项

* **致命的交叉配置错误**：
在实操中最大的坑，就是用户经常在下位机（终端 3）开着普通的 `move_group.launch`，然后去跑 V7 的 Python 脚本。这会导致程序虽然运行，但根本无法建立点云碰撞树。
* **唯一正确的启动阀门**：
当你要运行 `moveit_octomap_grasp.py` 时，**必须先关掉之前的 MoveGroup**，并启动我们专为 OctoMap 打造的顶配启动文件：
```bash
roslaunch kuavo_arm_moveit_config move_group_octomap.launch

```

* **底层连锁反应解密**：
为什么必须是它？因为它在内部调用了 `sensor_manager_octomap.launch.xml`，进而精确地挂载了 `sensors_3d_octomap.yaml`。
只有此时，MoveIt 内部的 `PointCloudOctomapUpdater` 插件才会苏醒。它会死死咬住上位机传来的 `/camera/depth/color/points` 话题，并以 `max_range: 2.5` 米的探测半径和 `max_update_rate: 2.0` Hz 的建图频率，开始在内存中拼装 3D 体素积木。

#### 9.3.2 备用降级逻辑：点云获取失败时的自动无缝切回 VLA 收手

在这个终极版本中，我们展现了极高的软件工程素养：**绝不让系统因为单一传感器的掉线而彻底瘫痪（Single Point of Failure）**。

* **隐患来源**：
点云流极其消耗带宽和相机算力。正如我们在前几章记录的那样，奥比中光相机的底层驱动随时可能发生 `SIGSEGV` 段错误闪退；又或者，现场网络波动导致点云延迟。如果此时程序还在“死等”建图，机械臂就会举着水瓶僵在半空中。
* **自动嗅探与降级机制（Fail-Safe）**：
我们在 `moveit_octomap_grasp.py` 中，编写了一套名为 `resolve_pointcloud_topic()` 的极其强壮的探针逻辑。
1. **多话题捕获**：系统会遍历 `POINT_CLOUD_CANDIDATES` 数组，动态搜寻局域网内当前存活的点云话题。
2. **首帧超时探测**：如果 15 秒内 `rostopic hz` 没有任何响应（`no new messages`），或者 `probe_octomap_ready()` 在 2.5 秒内无法完成碰撞树探测；
3. **优雅降级（Graceful Degradation）**：脚本**绝不抛出异常退出**！它会在终端打印出醒目的警告：`未检测到有效点云，系统降级为盲区操作模式`。随后，在阶段 B 收手时，它会自动挂起 OMPL，无缝切回 V6 经典版里的 `execute_vla_style_return()` 函数（即退回“肩膀外摆 75°”硬编码策略）。

通过这套堪称艺术的“降级防线”，V7 版本不仅拥有了傲视群雄的 3D 避障上限，更保住了无论环境多么恶劣都能把水瓶安全拿回来的托底能力！这就是双轨运行手册的最终底气！

```python
class Quaternion:
    def __init__(self): self.w=0; self.x=0; self.y=0; self.z=0

def euler_to_rotation_matrix(yaw, pitch, roll):
    cy, sy = np.cos(yaw), np.sin(yaw)
    cp, sp = np.cos(pitch), np.sin(pitch)
    cr, sr = np.cos(roll), np.sin(roll)
    return np.array([[cy, -sy, 0], [sy, cy, 0], [0, 0, 1]]) @ np.array([[cp, 0, sp], [0, 1, 0], [-sp, 0, cp]]) @ np.array([[1, 0, 0], [0, cr, -sr], [0, sr, cr]])

def rotation_matrix_to_quaternion(R):
    trace = np.trace(R)
    q = Quaternion()
    if trace > 0:
        q.w = math.sqrt(trace + 1.0) / 2
        q.x = (R[2, 1] - R[1, 2]) / (4 * q.w); q.y = (R[0, 2] - R[2, 0]) / (4 * q.w); q.z = (R[1, 0] - R[0, 1]) / (4 * q.w)
    else:
        i = np.argmax([R[0, 0], R[1, 1], R[2, 2]])
        j, k = (i + 1) % 3, (i + 2) % 3
        t = np.zeros(4)
        t[i] = math.sqrt(R[i, i] - R[j, j] - R[k, k] + 1) / 2
        t[j] = (R[i, j] + R[j, i]) / (4 * t[i])
        t[k] = (R[k, i] + R[i, k]) / (4 * t[i])
        t[3] = (R[k, j] - R[j, k]) / (4 * t[i])
        q.x, q.y, q.z, q.w = t
    norm = math.sqrt(q.w**2 + q.x**2 + q.y**2 + q.z**2)
    if norm > 0: q.w /= norm; q.x /= norm; q.y /= norm; q.z /= norm
    return q

def get_horizontal_claw_quat(target_x, target_y, is_left_arm):
    yaw = math.atan2((target_y - (0.292 if is_left_arm else -0.292)), (target_x - (-0.017)))
    R_base = euler_to_rotation_matrix(yaw, -1.57079633, 0)
    cr, sr = math.cos(CLAW_ROLL_LEFT if is_left_arm else CLAW_ROLL_RIGHT), math.sin(CLAW_ROLL_LEFT if is_left_arm else CLAW_ROLL_RIGHT)
    return rotation_matrix_to_quaternion(R_base @ np.array([[cr, -sr, 0], [sr, cr, 0], [0, 0, 1]]))

def _clamp_elbow_deg(target_deg):
    """肘关节死锁防护（左右臂第 4 轴）。"""
    if target_deg[3] > 0.0:
        target_deg[3] = 0.0
    if target_deg[10] > 0.0:
        target_deg[10] = 0.0
    return target_deg

def _downsample_joint_chain(joint_chain, max_points=MAX_STREAM_POINTS):
    """均匀降采样，防止 OMPL 输出上百点导致每段 dt 过小、手臂过快。"""
    if len(joint_chain) <= max_points:
        return list(joint_chain)
    indices = np.linspace(0, len(joint_chain) - 1, max_points, dtype=int)
    return [joint_chain[i] for i in indices]

def publish_arm_trajectory_batch(arm_pub, joint_chain_rad, total_time_sec, is_left_arm=None):
    """
    一次性发布完整 14 轴轨迹（官方推荐用法），避免逐条消息打断 MPC 导致过快/抖动。
    times 为相对当前时刻的累计到达时间；values 为 deg 扁平数组。
    """
    global last_commanded_joints_rad
    if not joint_chain_rad:
        return False

    if arm_pub.get_num_connections() == 0:
        rospy.logwarn("⚠️ /kuavo_arm_target_poses 暂无订阅者，仍尝试下发（与 auto_grasp 行为一致）")

    chain = _downsample_joint_chain(joint_chain_rad)
    if is_left_arm is not None:
        chain = [_freeze_inactive_arm(q, is_left_arm) for q in chain]
    n = len(chain)
    dt = max(total_time_sec / n, MIN_SEGMENT_DT)
    actual_total = dt * n

    msg = armTargetPoses()
    msg.times = [(i + 1) * dt for i in range(n)]
    for joints_rad in chain:
        target_deg = _clamp_elbow_deg([math.degrees(r) for r in joints_rad])
        msg.values.extend(target_deg)

    arm_pub.publish(msg)
    last_commanded_joints_rad = np.copy(chain[-1])
    rospy.loginfo(
        f"📡 批量下发 {n} 个航点，总时长 {actual_total:.2f}s，"
        f"订阅者 {arm_pub.get_num_connections()} 个"
    )
    rospy.sleep(actual_total + 0.2)
    return True

def _ik_group_profile(is_left_arm):
    """视觉/厂家 IK 目标在夹爪 TCP，对应 URDF 的 *_end_effector（非 wrist link）。"""
    if is_left_arm:
        return "left_arm", "zarm_l7_end_effector"
    return "right_arm", "zarm_r7_end_effector"

def _ee_link_candidates(primary_ee_link):
    """优先 TCP end_effector，失败再回退 wrist link。"""
    if primary_ee_link.endswith("_end_effector"):
        return [primary_ee_link, primary_ee_link.replace("_end_effector", "_link")]
    return [primary_ee_link]

def _inactive_arm_slice(is_left_arm):
    return slice(7, 14) if is_left_arm else slice(0, 7)

def _active_arm_slice(is_left_arm):
    return slice(0, 7) if is_left_arm else slice(7, 14)

def _init_joints_rad():
    return np.radians(DUAL_ARM_INIT_DEG)

def _freeze_inactive_arm(joints_14_rad, is_left_arm):
    """仅动活动臂：非活动臂始终锁在 init_angles，防止左手跟着前伸。"""
    frozen = np.copy(joints_14_rad)
    init = _init_joints_rad()
    frozen[_inactive_arm_slice(is_left_arm)] = init[_inactive_arm_slice(is_left_arm)]
    return frozen

def _build_high_safe_joints(q_lift, is_left_arm):
    """vla_auto_grasp_daemon 同款：抬升后纯关节空间肩膀外摆，清空桌面领空。"""
    q = np.copy(q_lift)
    if is_left_arm:
        q[1] += math.radians(SHOULDER_SWING_AVOID_DEG)
    else:
        q[8] -= math.radians(SHOULDER_SWING_AVOID_DEG)
    return _freeze_inactive_arm(q, is_left_arm)

def _build_ik_seed_for_pose(is_left_arm, active_seed_14, use_official_active=False):
    """非活动臂锁 init；活动臂用 official 或链式 seed。"""
    seed = _init_joints_rad()
    if active_seed_14 is not None:
        active = np.asarray(active_seed_14, dtype=float)
        arm_sl = _active_arm_slice(is_left_arm)
        if use_official_active:
            seed[arm_sl] = OFFICIAL_IK_SEED_7
        else:
            seed[arm_sl] = active[arm_sl]
    return seed

def _solve_single_ik_mg(arm_group, is_left_arm, pose_stamped, seed_joints_rad, ee_link):
    """MoveGroupCommander 内置 IK（与 /compute_ik 共用 TRAC-IK，但自动匹配规划组）。"""
    seed = _build_ik_seed_for_pose(is_left_arm, seed_joints_rad)
    ps = PoseStamped()
    ps.header.frame_id = pose_stamped.header.frame_id
    ps.header.stamp = rospy.Time(0)
    ps.pose = pose_stamped.pose
    try:
        arm_group.set_start_state(_build_robot_state_seed(seed))
        arm_group.set_end_effector_link(ee_link)
        arm_group.clear_pose_targets()
        if not arm_group.set_pose_target(ps):
            return None
        merged = np.copy(seed)
        for j, name in enumerate(arm_group.get_active_joints()):
            merged[joint_names_14.index(name)] = arm_group.get_joint_value_target()[j]
        return _freeze_inactive_arm(merged, is_left_arm)
    except Exception as exc:
        rospy.logwarn(f"⚠️ MoveGroup IK ({ee_link}) 异常: {exc}")
        return None

def _solve_pose_ik(ik_client, arm_group, is_left_arm, pose_stamped, seed_14, step_name):
    """单点 MoveIt IK：official seed + TCP ee_link + MoveGroup/服务双通道。"""
    group_name, ee_link = _ik_group_profile(is_left_arm)
    rospy.loginfo(f"⏳ IK 求解: {step_name} ...")
    seed_variants = [
        _build_ik_seed_for_pose(is_left_arm, seed_14, use_official_active=True),
        _build_ik_seed_for_pose(is_left_arm, seed_14, use_official_active=False),
    ]
    for seed in seed_variants:
        for link in _ee_link_candidates(ee_link):
            if arm_group is not None:
                result = _solve_single_ik_mg(arm_group, is_left_arm, pose_stamped, seed, link)
                if result is not None:
                    rospy.loginfo(f"✅ {step_name} IK 成功 (MoveGroup / {link})")
                    return result
            result = _solve_single_ik(ik_client, group_name, link, pose_stamped, seed, is_left_arm)
            if result is not None:
                rospy.loginfo(f"✅ {step_name} IK 成功 (service / {link})")
                return result
    rospy.logerr(f"❌ {step_name} IK 无解")
    return None

def _solve_lift_ik_with_fallback(ik_client, arm_group, is_left_arm, locked_x, locked_y, locked_z,
                                  quat, q_grasp, side_label):
    """
    垂直抬升 IK：优先 22cm，无解则 18 → 14 → 10cm；全失败才回退抓握点。
    返回 (q_lift, lift_height_m)；lift_height_m=0 表示未抬升（用 q_grasp）。
    """
    for h in LIFT_HEIGHT_FALLBACKS_M:
        lift_pose = _build_pose_stamped(locked_x, locked_y, locked_z + h, quat)
        step_name = f"[{side_label}手] 垂直抬升 {int(round(h * 100))}cm"
        q_lift = _solve_pose_ik(ik_client, arm_group, is_left_arm, lift_pose, q_grasp, step_name)
        if q_lift is not None:
            if h < LIFT_HEIGHT - 1e-6:
                rospy.logwarn(
                    "⚠️ 抬升 %dcm IK 无解，降级为 %dcm"
                    % (int(round(LIFT_HEIGHT * 100)), int(round(h * 100)))
                )
            return q_lift, h
    rospy.logwarn("⚠️ 全部抬升高度 IK 失败，回退使用抓握点（无垂直拔高）")
    return q_grasp, 0.0

def _build_pose_stamped(x, y, z, quat):
    ps = PoseStamped()
    ps.header.frame_id = "base_link"
    ps.pose.position.x = x
    ps.pose.position.y = y
    ps.pose.position.z = z
    ps.pose.orientation.x = quat.x
    ps.pose.orientation.y = quat.y
    ps.pose.orientation.z = quat.z
    ps.pose.orientation.w = quat.w
    return ps

def _build_robot_state_seed(seed_joints_rad):
    """
    构建完整 RobotState 种子：先取 MoveIt 当前全身状态，再覆盖 14 轴手臂关节。
    只传 14 轴会导致 TRAC-IK 全部无解。
    """
    state = _get_robot_commander().get_current_state()
    js = state.joint_state
    positions = list(js.position)
    name_to_idx = {n: i for i, n in enumerate(js.name)}
    for slot, name in enumerate(joint_names_14):
        if name in name_to_idx:
            positions[name_to_idx[name]] = float(seed_joints_rad[slot])
    js.position = positions
    state.joint_state = js
    return state

def _apply_ik_solution_to_14d(seed_joints_rad, solution_joint_state, is_left_arm):
    merged = np.copy(seed_joints_rad)
    for j, name in enumerate(solution_joint_state.name):
        if name in joint_names_14:
            merged[joint_names_14.index(name)] = solution_joint_state.position[j]
    return _freeze_inactive_arm(merged, is_left_arm)

def _solve_single_ik(ik_client, group_name, ee_link, pose_stamped, seed_joints_rad, is_left_arm):
    """调用 /compute_ik 求解单点；非活动臂锁 init。"""
    ps = PoseStamped()
    ps.header.frame_id = pose_stamped.header.frame_id
    ps.header.stamp = rospy.Time(0)
    ps.pose = pose_stamped.pose

    req = GetPositionIKRequest()
    req.ik_request.group_name = group_name
    req.ik_request.ik_link_name = ee_link
    req.ik_request.pose_stamped = ps
    req.ik_request.robot_state = _build_robot_state_seed(seed_joints_rad)
    req.ik_request.avoid_collisions = False
    req.ik_request.timeout = rospy.Duration(0.5)
    try:
        resp = ik_client(req)
    except rospy.ServiceException as exc:
        rospy.logwarn(f"⚠️ IK 服务异常: {exc}")
        return None
    if resp.error_code.val != MoveItErrorCodes.SUCCESS:
        return None
    return _apply_ik_solution_to_14d(seed_joints_rad, resp.solution.joint_state, is_left_arm)

def interpolate_cartesian_line(start_ps, end_ps, num_waypoints=CARTESIAN_NUM_WAYPOINTS):
    """
    在起点与终点之间做 xyz 均匀直线插值，四元数姿态全程锁定不变（纯平移）。
    返回 num_waypoints+1 个 PoseStamped（含起终点）。
    """
    p0 = np.array([start_ps.pose.position.x, start_ps.pose.position.y, start_ps.pose.position.z])
    p1 = np.array([end_ps.pose.position.x, end_ps.pose.position.y, end_ps.pose.position.z])
    locked_ori = start_ps.pose.orientation
    waypoints = []
    for i in range(num_waypoints + 1):
        t = float(i) / float(num_waypoints)
        wp = PoseStamped()
        wp.header.frame_id = start_ps.header.frame_id
        wp.header.stamp = rospy.Time.now()
        wp.pose.position.x = p0[0] + t * (p1[0] - p0[0])
        wp.pose.position.y = p0[1] + t * (p1[1] - p0[1])
        wp.pose.position.z = p0[2] + t * (p1[2] - p0[2])
        wp.pose.orientation = locked_ori
        waypoints.append(wp)
    return waypoints

def _max_arm_joint_jump_deg(prev_joints, new_joints, is_left_arm):
    """计算本臂 7 轴中相邻航点的最大关节跳变（度）。"""
    arm_slice = slice(0, 7) if is_left_arm else slice(7, 14)
    delta = np.abs(new_joints[arm_slice] - prev_joints[arm_slice])
    return float(np.max(np.degrees(delta)))

def solve_cartesian_ik_chain(group_name, ee_link, waypoints, seed_joints_rad, is_left_arm):
    """
    逐航点调用 moveit_msgs/GetPositionIK，上一解作为下一解的种子。
    返回: (success_joints_list, success_fraction)
    """
    ik_client = _resolve_ik_service()
    if ik_client is None:
        return [], 0.0

    chain = []
    current_seed = np.copy(seed_joints_rad)

    for idx, wp in enumerate(waypoints):
        # 起点航点：机械臂已在预瞄位，直接采信当前关节角，不必再 IK
        if idx == 0:
            chain.append(np.copy(current_seed))
            continue

        candidate = _solve_single_ik(ik_client, group_name, ee_link, wp, current_seed, is_left_arm)
        if candidate is None:
            rospy.logwarn(f"⚠️ 航点 {idx}/{len(waypoints)-1} IK 无解")
            continue

        if chain:
            jump = _max_arm_joint_jump_deg(chain[-1], candidate, is_left_arm)
            if jump > MAX_ARM_JOINT_JUMP_DEG:
                rospy.logwarn(
                    f"⚠️ 航点 {idx} 肘关节甩尾跳变 {jump:.1f}° > {MAX_ARM_JOINT_JUMP_DEG}°，丢弃该点"
                )
                continue

        chain.append(candidate)
        current_seed = candidate

    fraction = len(chain) / max(len(waypoints), 1)
    return chain, fraction

def pack_joint_trajectory(joint_chain, total_time_sec):
    """
    将连续 14 轴关节序列封装为 trajectory_msgs/JointTrajectory（带均匀时间戳）。
    可直接供后续控制器或流式泵送逻辑消费。
    """
    traj = JointTrajectory()
    traj.joint_names = list(joint_names_14)
    if not joint_chain:
        return traj
    dt = total_time_sec / len(joint_chain)
    for i, joints in enumerate(joint_chain):
        pt = JointTrajectoryPoint()
        pt.positions = [float(v) for v in joints]
        pt.time_from_start = rospy.Duration.from_sec((i + 1) * dt)
        traj.points.append(pt)
    return traj

def stream_joint_chain(arm_pub, joint_chain, total_time_sec):
    """将 14 轴关节链批量泵送至 /kuavo_arm_target_poses。"""
    return publish_arm_trajectory_batch(arm_pub, joint_chain, total_time_sec)

def execute_cartesian_linear_ik(arm_group, arm_pub, step_name, start_ps, end_ps, is_left_arm,
                                total_time_sec=2.5, num_waypoints=CARTESIAN_NUM_WAYPOINTS):
    """
    完整笛卡尔直线切入管线：插值 → 链式 IK → 封装轨迹 → 流式执行。
    """
    rospy.loginfo(f"\n🧠 自定义笛卡尔 IK 引擎启动: [{step_name}]...")
    clear_octomap_cache()
    wait_joints_settle(0.4)
    global last_commanded_joints_rad
    last_commanded_joints_rad = np.copy(current_joints_rad)

    group_name = "left_arm" if is_left_arm else "right_arm"
    _, ee_link = _ik_group_profile(is_left_arm)

    waypoints = interpolate_cartesian_line(start_ps, end_ps, num_waypoints)
    dist_m = math.sqrt(
        (end_ps.pose.position.x - start_ps.pose.position.x) ** 2 +
        (end_ps.pose.position.y - start_ps.pose.position.y) ** 2 +
        (end_ps.pose.position.z - start_ps.pose.position.z) ** 2
    )
    rospy.loginfo(
        f"📐 直线航程 {dist_m*100:.1f}cm，切分 {len(waypoints)} 个微观航点，姿态全程锁定"
    )

    joint_chain, fraction = solve_cartesian_ik_chain(
        group_name, ee_link, waypoints, last_commanded_joints_rad, is_left_arm
    )
    rospy.loginfo(f"🔗 链式 IK 完成：{len(joint_chain)}/{len(waypoints)} 航点求解成功 ({fraction*100:.1f}%)")

    if fraction < MIN_IK_SUCCESS_FRACTION:
        rospy.logerr(f"❌ [{step_name}] IK 覆盖率 {fraction*100:.1f}% 不足 {MIN_IK_SUCCESS_FRACTION*100:.0f}%！")
        return False

    # 强制末点贴近目标笛卡尔位姿，避免链式丢点导致抓不准
    ik_client = _resolve_ik_service()
    if ik_client is not None:
        end_joints = _solve_single_ik(
            ik_client, group_name, ee_link, waypoints[-1],
            joint_chain[-1] if joint_chain else last_commanded_joints_rad,
            is_left_arm,
        )
        if end_joints is not None:
            if not joint_chain or _max_arm_joint_jump_deg(joint_chain[-1], end_joints, is_left_arm) <= MAX_ARM_JOINT_JUMP_DEG:
                if joint_chain:
                    joint_chain[-1] = end_joints
                else:
                    joint_chain.append(end_joints)

    traj = pack_joint_trajectory(joint_chain, total_time_sec)
    rospy.loginfo(f"✅ 轨迹封装完毕：{len(traj.points)} 点 / {total_time_sec}s")

    ok = stream_joint_chain(arm_pub, joint_chain, total_time_sec)
    if ok:
        wait_joints_settle()
    return ok

def execute_cartesian_path_service(arm_pub, start_joints_rad, end_pose_stamped, is_left_arm,
                                   total_time_sec, step_name, min_fraction=0.95):
    """
    MoveIt 原生 GetCartesianPath 服务（非 Python compute_cartesian_path 绑定）。
    从 start_joints_rad 对应位姿沿直线插值至 end_pose，姿态锁定。
    """
    cart_client = _resolve_cartesian_service()
    if cart_client is None:
        rospy.logwarn(f"⚠️ [{step_name}] 笛卡尔服务不可用")
        return False

    group_name, ee_link = _ik_group_profile(is_left_arm)
    rospy.loginfo(f"\n📏 MoveIt 笛卡尔直线: [{step_name}] ({ee_link})...")

    req = GetCartesianPathRequest()
    req.header.frame_id = "base_link"
    req.start_state = _build_robot_state_seed(start_joints_rad)
    req.group_name = group_name
    req.link_name = ee_link
    req.waypoints = [end_pose_stamped.pose]
    req.max_step = 0.008
    req.jump_threshold = 0.0
    req.prismatic_jump_threshold = 0.0
    req.revolute_jump_threshold = 0.0
    req.avoid_collisions = False

    try:
        resp = cart_client(req)
    except rospy.ServiceException as exc:
        rospy.logwarn(f"⚠️ [{step_name}] 笛卡尔服务异常: {exc}")
        return False

    fraction = float(resp.fraction)
    rospy.loginfo(f"🔗 笛卡尔路径覆盖率: {fraction*100:.1f}%")
    if fraction < min_fraction:
        rospy.logerr(f"❌ [{step_name}] 笛卡尔覆盖率 {fraction*100:.1f}% < {min_fraction*100:.0f}%")
        return False

    traj = resp.solution.joint_trajectory
    if not traj.points:
        rospy.logerr(f"❌ [{step_name}] 笛卡尔轨迹为空")
        return False

    joint_chain = _trajectory_msg_to_14d_chain(traj, start_joints_rad)
    ok = publish_arm_trajectory_batch(arm_pub, joint_chain, total_time_sec, is_left_arm=is_left_arm)
    if ok:
        wait_joints_settle()
        rospy.loginfo(f"✅ [{step_name}] 笛卡尔直线执行完成 ({len(joint_chain)} 点)")
    return ok

def _trajectory_msg_to_14d_chain(traj, base_rad):
    """将 JointTrajectory（可能仅含单臂 7 轴）合并进 14 轴链。"""
    chain = []
    target_rad = np.copy(base_rad)
    for point in traj.points:
        for j, name in enumerate(traj.joint_names):
            if name in joint_names_14:
                target_rad[joint_names_14.index(name)] = point.positions[j]
        chain.append(np.copy(target_rad))
    return chain

def _plan_to_joint_chain(plan, base_rad):
    """将单侧 MoveIt 规划结果合并进 14 轴关节链。"""
    chain = []
    target_rad = np.copy(base_rad)
    for point in plan.joint_trajectory.points:
        for j, name in enumerate(plan.joint_trajectory.joint_names):
            if name in joint_names_14:
                target_rad[joint_names_14.index(name)] = point.positions[j]
        chain.append(np.copy(target_rad))
    return chain

def _merge_dual_arm_chains(chain_l, chain_r, base_rad):
    """将左右臂关节链按时间对齐合并为 14 轴同步轨迹。"""
    n = max(len(chain_l), len(chain_r), 1)
    merged = []
    for i in range(n):
        combined = np.copy(base_rad)
        if chain_l:
            combined[0:7] = chain_l[min(i, len(chain_l) - 1)][0:7]
        if chain_r:
            combined[7:14] = chain_r[min(i, len(chain_r) - 1)][7:14]
        merged.append(combined)
    return merged

def return_both_arms_to_ready(left_arm, right_arm, arm_pub, total_time=4.0, step_name="双臂同步归位"):
    """
    左右手臂同时规划并下发至 left_ready / right_ready。
    程序启动、异常退出、抓取结束均须调用，防止单臂前伸导致重心前倾。
    """
    global last_commanded_joints_rad
    rospy.loginfo(f"\n🛡️ {step_name}：左右手同时退回曲肘护胸...")
    clear_octomap_cache()
    for grp in (left_arm, right_arm):
        grp.set_start_state_to_current_state()
        grp.set_max_velocity_scaling_factor(VELOCITY_SCALING)
        grp.set_max_acceleration_scaling_factor(ACCELERATION_SCALING)
        grp.set_planning_time(8.0)

    left_arm.set_named_target("left_ready")
    right_arm.set_named_target("right_ready")
    ok_l, plan_l, _, _ = left_arm.plan()
    ok_r, plan_r, _, _ = right_arm.plan()

    if (not ok_l or not ok_r or
            len(plan_l.joint_trajectory.points) == 0 or
            len(plan_r.joint_trajectory.points) == 0):
        rospy.logerr("❌ 双臂归位规划失败！请检查 move_group 或人工扶稳机器人！")
        return False

    base = np.copy(current_joints_rad)
    chain_l = _plan_to_joint_chain(plan_l, base)
    chain_r = _plan_to_joint_chain(plan_r, base)
    joint_chain = _merge_dual_arm_chains(chain_l, chain_r, base)
    rospy.loginfo(f"✅ 双臂归位：左 {len(chain_l)} 点 + 右 {len(chain_r)} 点 → 合并 {len(joint_chain)} 点")

    ok = publish_arm_trajectory_batch(arm_pub, joint_chain, total_time)
    if ok:
        wait_joints_settle()
    return ok

    points = plan.joint_trajectory.points

    if ok:

def _run_grasp_sequence(left_arm, right_arm, arm_pub, x_hist, y_hist):
    is_left_arm = np.median(y_hist) > 0.0
    arm = left_arm if is_left_arm else right_arm

    off_x, off_y = tcp_offsets_for_arm(is_left_arm)
    locked_x = np.median(x_hist) + off_x
    locked_y = np.median(y_hist) + off_y
    locked_z = SAFE_LOCKED_Z  

    side = "左" if is_left_arm else "右"
    print(f"\n🎯 TF2 融合打击点: X={locked_x:.3f}, Y={locked_y:.3f}, Z={locked_z:.3f} ({side}手)")

    quat = get_horizontal_claw_quat(locked_x, locked_y, is_left_arm)
    shoulder_x = -0.017
    shoulder_y = 0.292 if is_left_arm else -0.292
    dist = math.hypot(locked_x - shoulder_x, locked_y - shoulder_y)
    if dist <= PRE_GRASP_DIST + 0.01:
        rospy.logerr(f"❌ 目标距肩点仅 {dist*100:.1f}cm，小于预瞄距离 {PRE_GRASP_DIST*100:.0f}cm！")
        _safe_return_both_arms(left_arm, right_arm, arm_pub, step_name="目标过近收手")
        return
    ratio = (dist - PRE_GRASP_DIST) / dist 
    
    pre_x = shoulder_x + (locked_x - shoulder_x) * ratio
    pre_y = shoulder_y + (locked_y - shoulder_y) * ratio
    grasp_pose = _build_pose_stamped(locked_x, locked_y, locked_z, quat)
    pre_pose = _build_pose_stamped(pre_x, pre_y, locked_z, quat)

    _, ee_link = _ik_group_profile(is_left_arm)
    try:
        arm.set_end_effector_link(ee_link)
    except Exception:
        rospy.logwarn(f"⚠️ 无法设置 ee_link={ee_link}，继续尝试 IK")

    ik_client = _resolve_ik_service()
    if ik_client is None:
        _safe_return_both_arms(left_arm, right_arm, arm_pub, step_name="IK服务不可用")
        return

    # === 逆推 IK 验证航点（auto_grasp 同款顺序 + MoveIt TCP/seed）===
    q_grasp = _solve_pose_ik(ik_client, arm, is_left_arm, grasp_pose, last_commanded_joints_rad, f"[{side}手] 终点切入")
    if q_grasp is None:
        _safe_return_both_arms(left_arm, right_arm, arm_pub, step_name="抓握点IK失败")
        return

    q_pre = _solve_pose_ik(ik_client, arm, is_left_arm, pre_pose, q_grasp, f"[{side}手] 退后预瞄 12cm")
    if q_pre is None:
        rospy.logwarn("⚠️ 预瞄点 IK 失败，回退使用抓握点")
        q_pre = q_grasp

    q_lift, lift_height_m = _solve_lift_ik_with_fallback(
        ik_client, arm, is_left_arm, locked_x, locked_y, locked_z, quat, q_grasp, side
    )

    print("\n🚀 执行: 曲肘护胸 → 预瞄 → 单段水平插入 → 夹爪 → 抬升 → vla收手")
    print("   （与 auto_grasp_TF2 一致：禁止 init 直跳预瞄，否则关节空间先上抬扫瓶）")

    # auto_grasp_TF2 验证过的构型：从曲肘位再伸向预瞄，避免 init→预瞄 先抬高手肘碰倒瓶
    ready_rad = np.radians(_auto_grasp_ready_deg(is_left_arm))
    execute_single_pose(arm_pub, ready_rad, 2.5, "曲肘护胸（预瞄前安全构型）", is_left_arm)

    q_pre_exec = _solve_pose_ik(
        ik_client, arm, is_left_arm, pre_pose, last_commanded_joints_rad, f"[{side}手] 曲肘后退至预瞄"
    )
    if q_pre_exec is None:
        rospy.logwarn("⚠️ 曲肘后预瞄 IK 失败，回退使用规划阶段解")
        q_pre_exec = q_pre
    execute_single_pose(arm_pub, q_pre_exec, 2.5, "退至预瞄点（肩→瓶直线 12cm 外）", is_left_arm)

    # 预瞄与抓握共线同高：单段 14 轴插值 = auto_grasp 水平笔直插入（不用笛卡尔多点，防连杆上翘）
    execute_single_pose(arm_pub, q_grasp, 1.5, "水平笔直插入抓握点", is_left_arm)

    print("✊ 执行二指夹爪安全闭合...")
    pos, vel, effort = build_close_cmd(is_left_arm)
    call_leju_claw(pos, vel, effort, tag="close")

    else:
        lift_label = "垂直拔高（IK 全失败，保持抓握高度）"
```

#### 10.1.2 源码深度剖析

这套代码虽然名为 `moveit_auto_grasp.py`，但在最核心的物理接触段，它完全是一套被“改造过”的硬编码逆解流程。它之所以能做到百发百中，全赖以下几个核心模块的支撑：

1. **TCP 非对称分参机制 (`TCP_OFFSET_X_LEFT`, `TCP_OFFSET_Y_RIGHT` 等)**
通过 `tcp_offsets_for_arm()` 派发。当前仓库值为 `TCP_OFFSET_X_LEFT = -0.013`、`TCP_OFFSET_X_RIGHT = -0.022`、`TCP_OFFSET_Y_LEFT = 0.004`、`TCP_OFFSET_Y_RIGHT = 0.065`。抬升阶段使用 `LIFT_HEIGHT_FALLBACKS_M = (0.22, 0.18, 0.14, 0.10)` 在 IK 无解时递减重试。
2. **肘关节死锁强力物理防护 (`_clamp_elbow_deg`)**
在 7 自由度机械臂中，由于冗余角的存在，逆解极易给出“手肘上抬（关节角为正）”的奇怪姿态。我们在每一个下发给底层的目标关节角中，强行切入了一刀 `if target_deg[3] > 0.0: target_deg[3] = 0.0`。这在数学上暴力切断了关节 3 和 10 向反方向折叠的可能，保证手臂的构型永远顺应重力下垂。
3. **逆推法与强制种子继承 (`_solve_pose_ik` 内核)**
为了防止从“预瞄点”到“抓取点”这 12cm 的平移发生甩尾，我们采用了逆推法：先解算出终点姿态 `q_grasp`，然后将 `q_grasp` 作为种子（Seed）直接喂给 `q_pre` 的解算。这种“基因继承”保证了在平移的 12cm 过程中，7 个电机的配置矩阵高度相似，底层插值极其平滑。
4. **VLA 外摆收手防碰撞 (`execute_vla_style_return`)**
由于这套经典代码没有点云感知（为了极致的速度与低算力开销），在抓到水瓶后，如何保证手臂不砸烂桌子？我们通过 `_build_high_safe_joints` 强行向 `q_lift` 中的肩膀关节（1 轴或 8 轴）注入了 `SHOULDER_SWING_AVOID_DEG = 75.0` 度的偏差。这让沉重的金属大臂像一扇门一样，平行于地面猛然向外荡开，彻底脱离了实木桌面的正上方投影领空，随后再安心地收回胸口。
5. **备用引擎（主路径不调用）**
文件末尾仍保留 `execute_cartesian_linear_ik()`、`execute_cartesian_path_service()`、`smart_execute()`、`return_both_arms_to_ready()` 等 v1～v5 演进产物，供 OctoMap 收手或排障对比；**V6 主路径仅走 `_run_grasp_sequence` 内的 `execute_single_pose` 链**。

```python
def resolve_pointcloud_topic():
    """
    自动探测点云话题。
    注意：/vla/yolo_target 是视觉检测坐标，与点云无关；YOLO 能刷屏不代表有点云。
    常见陷阱：rostopic list 能看到 /camera/depth/color/points，但无发布者（hz 为 no new messages）。
    """
    global _resolved_point_cloud_topic
    if _resolved_point_cloud_topic:
        return _resolved_point_cloud_topic

    rospy.loginfo("⏳ 探测点云话题（YOLO 坐标 ≠ 点云，两者独立）...")
    per_topic = max(POINTCLOUD_PROBE_TIMEOUT, 1.0)

    for topic in POINT_CLOUD_CANDIDATES:
        try:
            rospy.wait_for_message(topic, PointCloud2, timeout=per_topic)
            _resolved_point_cloud_topic = topic
            rospy.loginfo(f"✅ 点云已锁定: {topic}")
            return topic
        except rospy.ROSException:
            rospy.logwarn(f"⚠️ {per_topic:.0f}s 内无数据: {topic}")

    # 话题存在但无数据：深度图有、点云无 → 需 depth_to_pointcloud.launch
    depth_ok = False
    try:
        import sensor_msgs.msg
        rospy.wait_for_message("/camera/depth/image_raw", sensor_msgs.msg.Image, timeout=2.0)
        depth_ok = True
    except rospy.ROSException:
        pass

    if depth_ok:
        rospy.logerr(
            "❌ /camera/depth/image_raw 有数据，但点云话题无发布！\n"
            "   原因：相机 launch 未 enable_pointcloud，话题名存在但没人发。\n"
            "   在上位机另开终端执行：\n"
            "     roslaunch ~/kuavo-ros-opensource/src/demo/vla_grasp/launch/depth_to_pointcloud.launch\n"
            "   然后确认：rostopic hz /camera/depth/color/points"
        )

    # 扫描任意含 camera/depth 的 PointCloud2
    all_pc = _list_camera_pointcloud_topics()
    for topic in all_pc:
        if topic in POINT_CLOUD_CANDIDATES:
            continue
        if "camera" not in topic and "depth" not in topic:
            continue
        try:
            rospy.wait_for_message(topic, PointCloud2, timeout=per_topic)
            _resolved_point_cloud_topic = topic
            rospy.loginfo(f"✅ 点云已锁定（自动发现）: {topic}")
            rospy.logwarn(f"   请把 sensors_3d_octomap.yaml 的 point_cloud_topic 改为: {topic}")
            return topic
        except rospy.ROSException:
            continue

    rospy.logerr(
        "❌ 未收到任何相机点云！YOLO 坐标正常只说明 RGB+检测在跑，不代表深度/点云正常。\n"
        "   上位机相机日志若出现 Decode frame failed / MJPG conversion failed，\n"
        "   则深度流损坏，点云不会发布。\n"
        "   请在上位机执行排查：\n"
        "     rostopic list | grep -E 'points|PointCloud'\n"
        "     rostopic hz /camera/depth/color/points\n"
        "     rostopic hz /camera/depth/image_raw\n"
        "   RealSense 需在 launch 里 enable_pointcloud:=true align_depth:=true\n"
        "   奥比中光请确认 use_orbbec:=true 且驱动有点云输出。\n"
        f"   当前图中 PointCloud2 话题: {all_pc or '(无)'}"
    )
    return None

def probe_octomap_ready(timeout=OCTOMAP_PROBE_TIMEOUT):
    """
    启动时短时探测 OctoMap 管线（非阻塞）。
    MoveIt 常在 move_group 进程内建图，不一定对外发布 /octomap_full；
    收手 OMPL 成功即表示 OctoMap 已生效，此处勿长时间等待。
    """
    if not _HAS_OCTOMAP_MSG:
        rospy.loginfo("ℹ️ 未安装 octomap_msgs，跳过 OctoMap 启动探测")
        return False

    cloud_topics = [
        "/move_group/filtered_cloud",
        "/filtered_cloud",
    ]
    octomap_topics = ["/octomap_binary", "/octomap_full", "/move_group/octomap_full"]
    deadline = time.time() + timeout
    while time.time() < deadline and not rospy.is_shutdown():
        for topic in cloud_topics:
            try:
                rospy.wait_for_message(topic, PointCloud2, timeout=0.6)
                rospy.loginfo(f"✅ OctoMap 点云管线就绪: {topic}")
                return True
            except rospy.ROSException:
                continue
        for topic in octomap_topics:
            try:
                rospy.wait_for_message(topic, Octomap, timeout=0.4)
                rospy.loginfo(f"✅ OctoMap 已就绪: {topic}")
                return True
            except rospy.ROSException:
                continue
        rospy.sleep(0.15)

    rospy.loginfo(
        "ℹ️ 启动阶段未见 /octomap_full（常见且正常：MoveIt 在收手规划时内部建图）。"
        "  点云已有数据即可；收手将使用 OctoMap + 桌面碰撞盒。"
    )
    return False


def wait_for_octomap(timeout=OCTOMAP_PROBE_TIMEOUT):
    """兼容旧名；收手前可再探测一次。"""
    return probe_octomap_ready(timeout)

def _solve_ik_avoid_collisions(ik_client, arm_group, is_left_arm, pose_stamped, seed_14, step_name):
    """带碰撞检测的 IK（收手航点用）。"""
    group_name, ee_link = mag._ik_group_profile(is_left_arm)
    seed = mag._build_ik_seed_for_pose(is_left_arm, seed_14)
    ps = PoseStamped()
    ps.header.frame_id = pose_stamped.header.frame_id
    ps.header.stamp = rospy.Time(0)
    ps.pose = pose_stamped.pose

    req = GetPositionIKRequest()
    req.ik_request.group_name = group_name
    req.ik_request.ik_link_name = ee_link
    req.ik_request.pose_stamped = ps
    req.ik_request.robot_state = mag._build_robot_state_seed(seed)
    req.ik_request.avoid_collisions = True
    req.ik_request.timeout = rospy.Duration(0.8)
    try:
        resp = ik_client(req)
    except Exception as exc:
        rospy.logwarn(f"⚠️ {step_name} 碰撞IK异常: {exc}")
        return None
    if resp.error_code.val != MoveItErrorCodes.SUCCESS:
        return None
    return mag._apply_ik_solution_to_14d(seed, resp.solution.joint_state, is_left_arm)


def _build_retract_pose_stamped(locked_x, locked_y, locked_z, quat, is_left_arm, shoulder_x, shoulder_y, dist):
    """auto_grasp_TF2 步骤 E：抬升高度 + 向外 Y + 略后退。"""
    for fallback_dist in (0.15, 0.12, 0.09, 0.06, 0.0):
        ratio = (dist - fallback_dist) / dist if dist > 1e-6 else 1.0
        safe_x = shoulder_x + (locked_x - shoulder_x) * ratio
        safe_y = locked_y + AVOID_OUTWARD_Y if is_left_arm else locked_y - AVOID_OUTWARD_Y
        ps = mag._build_pose_stamped(safe_x, safe_y, locked_z + mag.LIFT_HEIGHT, quat)
        yield fallback_dist, ps


def plan_and_execute_collision_aware(arm_group, arm_pub, step_name, total_time,
                                     is_left_arm=None, use_joint_target=False):
    """
    OMPL 规划并下发；默认 **不清除** OctoMap，真正利用八叉树避障。
    """
    rospy.loginfo(f"\n🗺️ MoveIt 碰撞感知规划: [{step_name}]（保留 OctoMap）...")
    arm_group.set_start_state_to_current_state()
    arm_group.set_max_velocity_scaling_factor(mag.VELOCITY_SCALING)
    arm_group.set_max_acceleration_scaling_factor(mag.ACCELERATION_SCALING)
    arm_group.set_planning_time(10.0)

    if not use_joint_target:
        # pose target 应在调用前已 set_pose_target
        pass

    success, plan, _, error_code = arm_group.plan()
    if not success or len(plan.joint_trajectory.points) == 0:
        rospy.logwarn(f"⚠️ [{step_name}] OMPL 失败 (code={error_code.val})")
        return False

    rospy.loginfo(f"✅ [{step_name}] OMPL 成功，{len(plan.joint_trajectory.points)} 点")
    chain = []
    target_rad = np.copy(mag.last_commanded_joints_rad)
    for point in plan.joint_trajectory.points:
        for j, name in enumerate(plan.joint_trajectory.joint_names):
            if name in mag.joint_names_14:
                target_rad[mag.joint_names_14.index(name)] = point.positions[j]
        chain.append(np.copy(target_rad))

    ok = mag.publish_arm_trajectory_batch(arm_pub, chain, total_time, is_left_arm=is_left_arm)
    if ok:
        mag.wait_joints_settle()
    return ok


def octomap_retract_after_lift(arm_group, arm_pub, ik_client, is_left_arm,
                               locked_x, locked_y, locked_z, quat,
                               shoulder_x, shoulder_y, dist, side):
    """
    抬升后收手：优先 OMPL+OctoMap 到高空外移点，再 OMPL 到 ready；失败走 vla 硬编码。
    """
    rospy.loginfo(f"\n🛡️ [{side}手] OctoMap 收手：等待点云刷新 {OCTOMAP_SETTLE_AFTER_LIFT}s ...")
    rospy.sleep(OCTOMAP_SETTLE_AFTER_LIFT)

    seed = np.copy(mag.last_commanded_joints_rad)
    planned = False

    for fallback_dist, retract_ps in _build_retract_pose_stamped(
        locked_x, locked_y, locked_z, quat, is_left_arm, shoulder_x, shoulder_y, dist
    ):
        q_retract = _solve_ik_avoid_collisions(
            ik_client, arm_group, is_left_arm, retract_ps, seed,
            f"高空外移+后退{int(fallback_dist*100)}cm"
        )
        if q_retract is None:
            continue

        arm_group.set_start_state_to_current_state()
        arm_group.set_joint_value_target(
            [q_retract[mag.joint_names_14.index(n)] for n in arm_group.get_active_joints()]
        )
        if plan_and_execute_collision_aware(
            arm_group, arm_pub,
            f"OctoMap外移(退{int(fallback_dist*100)}cm)",
            3.0, is_left_arm=is_left_arm, use_joint_target=True
        ):
            planned = True
            rospy.loginfo(
                f"💡 OctoMap 收手：外移 {int(AVOID_OUTWARD_Y*100)}cm + "
                f"后退 {int(fallback_dist*100)}cm 规划成功"
            )
            break

    if planned:
        ready_name = "left_ready" if is_left_arm else "right_ready"
        arm_group.set_named_target(ready_name)
        if plan_and_execute_collision_aware(
            arm_group, arm_pub, "OctoMap曲肘ready", 3.5, is_left_arm=is_left_arm
        ):
            mag.execute_dual_arm_init_home(arm_pub)
            return True
        rospy.logwarn("⚠️ OctoMap ready 规划失败，继续 vla 回退")

    # vla 硬编码回退（与 vla_auto_grasp_daemon 一致）
    rospy.logwarn("⚠️ OctoMap 收手失败，回退 vla 肩膀外摆...")
    mag.execute_vla_style_return(arm_pub, mag.last_commanded_joints_rad, is_left_arm)
    return False


def _run_octomap_grasp_sequence(left_arm, right_arm, arm_pub, scene_iface, x_hist, y_hist):
    is_left_arm = np.median(y_hist) > 0.0
    arm = left_arm if is_left_arm else right_arm
    side = "左" if is_left_arm else "右"

    off_x, off_y = mag.tcp_offsets_for_arm(is_left_arm)
    locked_x = np.median(x_hist) + off_x
    locked_y = np.median(y_hist) + off_y
    locked_z = mag.SAFE_LOCKED_Z

    print(f"\n🎯 TF2 融合打击点: X={locked_x:.3f}, Y={locked_y:.3f}, Z={locked_z:.3f} ({side}手)")

    quat = mag.get_horizontal_claw_quat(locked_x, locked_y, is_left_arm)
    shoulder_x = -0.017
    shoulder_y = 0.292 if is_left_arm else -0.292
    dist = math.hypot(locked_x - shoulder_x, locked_y - shoulder_y)
    if dist <= mag.PRE_GRASP_DIST + 0.01:
        rospy.logerr(f"❌ 目标距肩点仅 {dist*100:.1f}cm，过近")
        mag.execute_dual_arm_init_home(arm_pub)
        return

    ratio = (dist - mag.PRE_GRASP_DIST) / dist
    pre_x = shoulder_x + (locked_x - shoulder_x) * ratio
    pre_y = shoulder_y + (locked_y - shoulder_y) * ratio
    grasp_pose = mag._build_pose_stamped(locked_x, locked_y, locked_z, quat)
    pre_pose = mag._build_pose_stamped(pre_x, pre_y, locked_z, quat)

    _, ee_link = mag._ik_group_profile(is_left_arm)
    try:
        arm.set_end_effector_link(ee_link)
    except Exception:
        pass

    ik_client = mag._resolve_ik_service()
    if ik_client is None:
        mag.execute_dual_arm_init_home(arm_pub)
        return

    q_grasp = mag._solve_pose_ik(
        ik_client, arm, is_left_arm, grasp_pose, mag.last_commanded_joints_rad, f"[{side}手] 终点切入"
    )
    if q_grasp is None:
        mag.execute_dual_arm_init_home(arm_pub)
        return

    q_pre = mag._solve_pose_ik(ik_client, arm, is_left_arm, pre_pose, q_grasp, f"[{side}手] 预瞄12cm")
    if q_pre is None:
        q_pre = q_grasp

    q_lift, lift_height_m = mag._solve_lift_ik_with_fallback(
        ik_client, arm, is_left_arm, locked_x, locked_y, locked_z, quat, q_grasp, side
    )

    print("\n🚀 阶段A [硬编码抓取] 曲肘→预瞄→水平插入（auto_grasp_TF2 同款，防碰瓶）")
    print("🚀 阶段B [OctoMap收手]  抬升后 OMPL 碰撞规划（失败→vla外摆）")

    ready_rad = np.radians(mag._auto_grasp_ready_deg(is_left_arm))
    mag.execute_single_pose(arm_pub, ready_rad, 2.5, "曲肘护胸", is_left_arm)

    q_pre_exec = mag._solve_pose_ik(
        ik_client, arm, is_left_arm, pre_pose, mag.last_commanded_joints_rad, f"[{side}手] 曲肘后预瞄"
    )
    if q_pre_exec is None:
        q_pre_exec = q_pre
    mag.execute_single_pose(arm_pub, q_pre_exec, 2.5, "退至预瞄点 12cm", is_left_arm)

    # 水瓶在点云里：插入前短暂清 OctoMap，仅影响最后 12cm 段
    mag.clear_octomap_cache()
    mag.execute_single_pose(arm_pub, q_grasp, 1.5, "水平笔直插入", is_left_arm)

    print("✊ 夹爪闭合...")
    pos, vel, effort = build_close_cmd(is_left_arm)
    mag.call_leju_claw(pos, vel, effort, tag="close")
    time.sleep(2.0)

    if lift_height_m > 0:
        lift_label = f"垂直抬升 {int(round(lift_height_m * 100))}cm"
    else:
        lift_label = "垂直抬升（IK 全失败，保持抓握高度）"
    mag.execute_single_pose(arm_pub, q_lift, 2.0, lift_label, is_left_arm)

    print("\n🗺️ 阶段B：OctoMap + 桌面盒 碰撞感知收手...")
    octomap_retract_after_lift(
        arm, arm_pub, ik_client, is_left_arm,
        locked_x, locked_y, locked_z, quat,
        shoulder_x, shoulder_y, dist, side
    )

    print("👐 松开夹爪")
    pos, vel, effort = build_open_cmd()
    mag.call_leju_claw(pos, vel, effort, tag="release")
    try:
        rospy.ServiceProxy('/arm_traj_change_mode', mag.changeArmCtrlMode)(
            mag.changeArmCtrlModeRequest(control_mode=0)
        )
    except Exception:
        pass


def main():
    moveit_commander.roscpp_initialize(sys.argv)
    rospy.init_node("moveit_octomap_grasp")
    _log_banner()

    rospy.Subscriber("/joint_states", mag.JointState, mag.joint_states_callback)
    rospy.loginfo("⏳ 等待 /joint_states ...")
    while not mag.has_joint_states and not rospy.is_shutdown():
        rospy.sleep(0.1)
    mag.last_commanded_joints_rad = np.copy(mag.current_joints_rad)

    mag._log_ros_network()

    pc_topic = resolve_pointcloud_topic()
    if pc_topic is None:
        rospy.logwarn(
            "⚠️ 无点云：进入降级模式（仅桌面碰撞盒 + vla 收手），"
            "OctoMap 避障不可用。请修复上位机深度流后重试。"
        )
    else:
        rospy.set_param("/move_group/octomap_frame", "base_link")
        # 若 move_group 已启动，尝试同步传感器话题（需与 launch 里 yaml 一致）
        rospy.loginfo(
            f"💡 请确认 move_group_octomap.launch 中 sensors_3d_octomap.yaml "
            f"的 point_cloud_topic = {pc_topic}"
        )

    try:
        rospy.ServiceProxy("/arm_traj_change_mode", mag.changeArmCtrlMode)(
            mag.changeArmCtrlModeRequest(control_mode=2)
        )
    except Exception:
        rospy.logwarn("⚠️ arm_traj_change_mode 失败，继续...")

    arm_pub = rospy.Publisher("/kuavo_arm_target_poses", mag.armTargetPoses, queue_size=10)
    rospy.sleep(0.3)

    print("🖐️ 双爪张开...")
    pos, vel, effort = build_open_cmd()
    mag.call_leju_claw(pos, vel, effort, tag="open")
    time.sleep(1.0)
    mag.execute_dual_arm_init_home(arm_pub)

    x_hist, y_hist = mag._collect_vision_targets_tf2_style()
    if len(x_hist) < 10:
        rospy.logerr("❌ 视觉失败")
        mag.execute_dual_arm_init_home(arm_pub)
        return

    rospy.loginfo("✅ 视觉就绪，加载 MoveIt（约 2～3s，属正常）...")
    left_arm = moveit_commander.MoveGroupCommander("left_arm")
    right_arm = moveit_commander.MoveGroupCommander("right_arm")
    for grp in (left_arm, right_arm):
        grp.set_pose_reference_frame("base_link")

    if pc_topic is not None:
        probe_octomap_ready()
    else:
        rospy.logwarn("⚠️ 跳过 OctoMap 启动探测（无点云）")

    scene_iface = moveit_commander.PlanningSceneInterface()
    setup_desk_collision_box(scene_iface)

    try:
        _run_octomap_grasp_sequence(left_arm, right_arm, arm_pub, scene_iface, x_hist, y_hist)
    except Exception as exc:
        rospy.logerr(f"❌ 异常: {exc}")
        mag.execute_dual_arm_init_home(arm_pub)
        raise
    finally:
        remove_desk_collision_box(scene_iface)


if __name__ == "__main__":
    main()

```
#### 10.2.2 源码深度剖析

这个拓展版脚本不仅仅是一个简单的执行流，它集成了大量的“防御性编程”思想（Defensive Programming），能够在严酷的物理环境中保证机器人绝不犯低级错误。

1. **动态点云嗅探与降级 (`resolve_pointcloud_topic`)**
MoveIt! 是极其僵化的。如果你在 `sensors_3d.yaml` 中配置了一个错误的话题，MoveIt 并不会报错，只会原地等死。通过 `POINT_CLOUD_CANDIDATES` 列表，脚本会在开机前主动遍历包括 `/camera/depth/color/points` 在内的所有潜在话题。无点云时打印警告并进入降级模式，收手退回 VLA 外摆。
2. **OctoMap 探针与反死等 (`probe_octomap_ready`)**
废弃死等 20 秒 `/octomap_full` 的逻辑，改为 2.5 秒非阻塞探测；MoveIt 常在内部建图不对外广播，收手 OMPL 成功即反向证明 OctoMap 生效。
3. **桌面防穿透空气墙 (`setup_desk_collision_box`)**
在 `Z=0.33m` 处注入 `DESK_BOX_ID` 碰撞盒，弥补 URDF 无桌面 collision 的缺陷。
4. **阶段 B 避障收手与容错嵌套 (`octomap_retract_after_lift`)**
渐进式外扩 15→12→9→6→0 cm 尝试 OMPL；全线失败则 `mag.execute_vla_style_return()` 降级。

#### 10.11 MoveIt 配置与 Launch 文件说明

* **10.3 `move_group_octomap.launch`**：在标准 `move_group.launch` 基础上，将 `sensor_manager.launch.xml` 替换为 **10.4** 的 OctoMap 版，是唯一正确的点云版终端 3 入口。
* **10.4 + 10.5**：`rosparam load sensors_3d_octomap.yaml` 后，MoveIt 的 `PointCloudOctomapUpdater` 订阅 `/camera/depth/color/points`（`max_update_rate: 2.0` Hz，防 CPU 过载）。
* **10.6**：Setup Assistant 生成的占位文件，仅声明 `moveit_sensor_manager=biped_s49`；实际传感器列表由 10.4 注入。
* **10.7**：经典版 **故意保持空** `sensors: []`，确保 `move_group.launch` 不订阅点云、不算力吃紧。
* **10.8**：当上位机相机 launch 未带点云时的**下位机侧备用**；正常情况用上位机 10.9 内置的 `vla_depth_to_pointcloud` 即可。
* **10.9 / 10.10**：**SIGSEGV 防崩溃关键**——`enable_point_cloud:=false`，由 `depth_image_proc/point_cloud_xyz` 外部生成点云；`color_fps:=30` 降负载。10.9 含 apriltag 与头部 TF 静态发布；10.10 为精简版（仅相机+点云）。
