# 🤖 Kuavo 机器人 LejuClaw 真机调试与防锁死（Wedge Lock）实战笔记

**项目背景**：在 Kuavo 人形机器人（v42/v49 版本，基于 `biped_s49` 模型库）上进行 LejuClaw 物理夹爪的真机打通。目标是脱离仿真环境，实现夹爪的实机控制，并为后续的具身智能（Embodied AI）视觉抓取任务构建安全、鲁棒的底层控制链。

**v49 左爪专项（2025-06 实机结论）**：左爪偶发自锁，经多轮排查确认**不是纯硬件报废**，而是 **启动标定关限位大电流 homing + 抓取大力闭合** 在蜗杆结构上触发机械自锁；同一次 launch 内 ROS 控制链路可正常工作。下文第六、七阶段记录完整原因、代码改造与测试过程；**第九阶段**补充 2026-06 双爪联调、MCP 前置条件与「启动像赌运气」的根因。

---

## 🛠️ 第一阶段：软件层打通与环境排雷

在实机上运行哪怕是最简单的测试脚本，也必须跨越 Linux 权限和 ROS 环境变量的连环坑。

### 1. 配置文件切换

在实机下位机主控启动前，必须让系统大脑知道我们挂载的是夹爪而非灵巧手。

* **文件路径**：工作空间对应的 `kuavo.json`。
* **核心修改**：将末端执行器字段修改为 `"EndEffectorType": ["lejuclaw", "lejuclaw"]`。
* **注意**：下位机主程序只有在**开机启动的那一瞬间**才会读取 `kuavo.json` 并执行开机标定（一开一合）。如果先开机再改配置，夹爪将被忽略。

### 2. LejuClaw 驱动配置（v49 易踩坑）

* **运行时实际读取路径**：`/home/lab/.config/lejuconfig/config.yaml`（launch 日志可见 `[LEJU claw] config_path:...`）。
* **模板路径**：`src/kuavo_assets/config/config.yaml`；复制到工作区根目录 `config.yaml` **不会生效**。
* **左爪标定相关**：日志中 `Negtive joint ID:` 为空时，左爪开限位寻址可能失败（见第六阶段）；右爪（关节 16）关限位寻址失败时开≈关，表现类似（见第六、九阶段）。

### 3. 编译与权限踩坑 (Permission Denied)

* **报错现象**：执行 `catkin build kuavo_sdk` 时报 `[Errno 13] Permission denied: '/home/lab/kuavo-ros-opensource/build/.built_by'`。
* **原因剖析**：由于涉及到硬件底层驱动，历史编译通常由 `root` 用户执行。普通 `lab` 用户无权覆盖 `root` 创建的 `build` 文件夹。
* **终极解法**：在真机操作中，强烈建议全程提升权限。使用 `sudo su` 切换至 root 后再进行编译和执行。

### 4. 环境与依赖踩坑 (ModuleNotFoundError)

* **报错现象**：在 root 下运行 Python 脚本时，报找不到 `kuavo_msgs` 模块。
* **原因剖析**：`sudo su` 会打开一个干净的 Shell 环境，丢失了之前 source 过的 ROS 工作空间路径。
* **终极解法**：切换 root 后，必须第一时间执行 `source devel/setup.bash`，给当前终端“开眼”，让它找到自定义的 ROS 话题和服务消息类型。

## 📡 第二阶段：ROS 通信架构与夹爪 API 深度解析

夹爪的控制逻辑并非盲目的单向开闭，而是基于“一听（Topic）一发（Service）”的状态机闭环。

### 1. 状态反馈链路 (ROS Topic)

* **话题名称**：`/leju_claw_state`
* **消息类型**：`kuavo_msgs/lejuClawState`
* **核心反馈数据**：
* `state`: 当前执行状态（`-1` Error，`0` Unknown，`1` Moving，`2` Reached，`3` Grabbed）。
* `position`: 夹爪当前百分比位置（**v49 实机已验证**）。
* `velocity`: 当前真实速度。
* `effort`: 当前实际电流/力矩（**极其重要**，用于堵转与自锁判断）。

### 2. 指令下发链路 (ROS Service)

* **服务名称**：`/control_robot_leju_claw`
* **服务类型**：`kuavo_msgs/controlLejuClaw`
* **传入参数**：
* `pos`: 目标位置列表 `[左爪, 右爪]`。
* `vel`: 速度限制（通常设为 `[50, 50]` 防撞击）。
* `effort`: 力矩/电流限制（抓取易碎品需调低，如 `0.2`~`0.6`；盲目用 `1.5` 易触发自锁）。

### 3. position 方向（v49 实机已验证，以 ROS 为准）

**本仓库 v49 实机 + `debug_left_claw.py` 扫参结论**（与部分旧文档/WebSocket 描述相反，**以实机为准**）：

* **`0` = 完全张开**
* **`100` = 完全闭合**

旧版示例脚本若把「数值越大越闭合」理解反了，或直接用 `[90, 90]` 双爪大力闭合，会在空载时把蜗杆顶死。VLA 抓取代码已统一走 `claw_safe.py`（见第七阶段）。

**文档一致性说明**（2026-06 联调核对）：

* **以实机 + `docs/4开发接口/接口使用文档.md` 为准**：`0` = 张开，`100` = 闭合。
* `kuavo_msgs/srv/controlLejuClaw.srv`、`lejuClawCommand.msg` 及 `installed/include/hardware_plant/lejuclaw_controller.h` 里的英文注释写成了 `0: closed, 100: open`，**与驱动实际行为不符**，可忽略。
* 底层 `liblejuclaw.a` 中 `move_paw` 映射公式（反汇编确认）：

  `关节角 = joint_end + (joint_start - joint_end) × position / 100`

  即 `position=0` → 开限位（`joint_end`），`position=100` → 关限位（`joint_start`）。**上层 `claw_safe.py`、`debug_left_claw.py` 与此一致，测试脚本本身无方向 bug。**

### 4. 启动标定（load_kuavo_real.launch）

`load_kuavo_real.launch` 起来后、跑任何抓取脚本**之前**，驱动自动执行「一开一合」限位寻找：

1. 找开限位  
2. **找关限位：约 1.5~1.8A 电流顶到堵转**

这一步**与用户是否事先用手掰开无关**，都会再用力关一次。左爪开限位标定失败时，关限位 homing 极易触发蜗杆自锁（见第六阶段）。

## 💥 第三阶段：史诗级 Bug 现场 —— 机械锁死 (Wedge Lock)

### 1. 故障表现

机器人开机时，夹爪毫无动静（失去开机标定动作）。运行任何控制脚本，软件层毫无报错，通信全通，但实机双爪完全闭合、死死咬在一起，彻底断电后用纯人力完全无法掰开。

### 2. 抓包溯源（早期 v42 确诊铁证）

保持机器人静止，新开终端运行 `rostopic echo /leju_claw_state`，抓取到以下冰冷的数据：

* `state: [2, 2]`
* **`position: [7.407, 62.962]`**

对比 `kuavo.json` 中官方设定的安全防线：

* `"max_joint_position_limits": [..., 30, 25]`（右爪最大闭合极限量为 25）。

### 3. 硬件崩坏原理还原

* **指令过载**：脚本发送了过大闭合量 + `effort: 1.0~1.5` 的高电流，电机在空载时疯狂向内挤压。
* **突破防线**：在突破了安全限位后，电机无视阻力继续强行发力，硬生生把机械连杆挤压到 **`62.962`** 等异常高位。
* **物理自锁**：强电过度挤压导致齿轮副之间产生了巨大的机械应力，发生了严重的**过度拧紧自锁（Wedge Lock）**。
* **保护触发**：电机被彻底憋停，瞬间触发 `kuavo.json` 里的 `"locked_rotor_protection": 2`（堵转保护）。底层芯片为了防烧，直接切断动力，夹爪从此陷入“上电即堵转断电”的死循环。

## 🔧 第四阶段：物理破局与满血复活

面对纯物理死锁，重启软件已经无效，必须实施物理干预。

### 1. 拆壳退轴（卸载应力）

* **操作**：拆卸夹爪外壳，暴露出中央的蜗杆与联轴金属圆盘。
* **解套**：使用内六角或其他工具撬动金属圆盘，**向着“张开”的反方向手动大力旋退**。
* **目标**：直到把死死咬合的爪尖旋开，让 position 回到安全区（早期案例从 `62.96` 退回 `25` 以内）。
<img src="images/249166cf511c7c3ff5712dcca2460205.jpg" width="700" />
### 2. 健康状态判定（非常重要）

* **视觉标准**：双爪分开，肉眼可见间隙。
* **手感标准**：不通电时，爪尖能够自由晃动 **0.5cm ~ 1cm**。
* **原理解析**：这个晃动量叫做**机械回差（Backlash）**，是齿轮与多级连杆正常啮合的健康标志。如果 0 间隙，通电后极易因摩擦力过大发热卡死。通电后，伺服电机会形成位置闭环，瞬间消除这个松动量，因此完全不影响抓取精度。

### 3. 修复代码，杜绝复发

物理复原后，**不要**再使用 `[90, 90]` + `effort 1.5` 这类参数。VLA 抓取统一使用 `src/demo/vla_grasp/claw_safe.py` 限幅（默认左爪 close **80** / effort **0.5**），详见第七阶段。

## 🛡️ 第五阶段：具身智能抓取的防御性编程 (Defensive Control)

### 0. 机械结构原理解析：双蜗轮单蜗杆的“双刃剑”
<img src="images/df4d3341f0bd8ef0a6a0a219562c4024.jpg" width="400" />

在拆开夹爪外壳进行物理退轴时，我们清晰地发现该夹爪采用了“单蜗杆-双蜗轮”的传动结构。作为算法开发者，理解这一硬件特性是编写防御性代码的物理前提：

* **优点（为何如此设计）**：
* **超大减速比**：在极小的手腕空间内，能让微型电机爆发出巨大的捏合力。
* **零功耗保持（物理自锁）**：由于蜗轮蜗杆只能单向传动，抓紧物体后即使彻底断电，也能靠机械静摩擦力死死咬住重物不掉落。这不仅极大节省了电池功耗，更有效防止了电机因长时间堵转而烧毁。


* **缺点（致命隐患）**：
* **极易发生破坏性自锁**：一旦上层软件发送了越界的大行程指令，在没有抓取真实物体作为“缓冲垫”的情况下，两片爪子会直接互相死磕。巨大的驱动力矩被全部转化为内部传动副的挤压应力，瞬间就会造成如同死结一般的机械自锁（也就是本次事故的根本原因）。



鉴于这种结构的“非反向驱动”特性，在未来的高级具身智能任务中，绝不能让上层视觉或 AI 算法直接“盲目”指挥底层硬件。必须在控制脚本中构筑以下四道防线，实现力位混合抓取：

### 防线 1：输入源头数据裁剪 (Command Saturation)

绝不信任上层传入的 Raw Data。在调用服务前，强行引入硬边界（已实现于 `claw_safe.py`）：

```python
# claw_safe.py 默认（可通过 ROS param ~claw_* 覆盖）
CLAW_OPEN = 10.0
CLAW_CLOSE_LEFT = 80.0      # 左爪更保守
CLAW_CLOSE_RIGHT = 85.0
MAX_CLOSE_POSITION = 85.0   # 硬顶，禁止 close 100 用于抓取
MAX_CLOSE_EFFORT = 0.6
```

### 防线 2：动态力矩限制 (Torque Profiling)

不要一上来就把 `effort` 拉满。

* **空载逼近**：使用极低力矩（如 `0.2`~`0.5`），即使撞到硬限位，电机也会因为力气小而安全停转，不会卡死。
* **包络抓取**：确认接触物体后，再短暂提升力矩（如 `0.5`~`0.6`）稳固抓持；**不要盲目用到 1.5**。

### 防线 3：基于反馈的主动截断 (Effort Monitoring - 触觉防线)

在闭合过程中监听 `/leju_claw_state` 话题。

* 若 `effort` 绝对值突然飙升（如 ≥ `1.0A`），意味着堵转或顶死。
* **立刻中止闭合**，并尝试 reopen（`claw_safe.py` 的 `SafeClawController.call()` 已实现）。

### 终极抓取状态机伪代码示例

```python
def adaptive_grasp_cup():
    target_pos = 70.0  # 保守闭合，勿用 100
    while not rospy.is_shutdown():
        cmd_pos = min(85.0, target_pos)  # claw_safe 硬顶
        call_leju_claw([cmd_pos, 10.0], [50, 50], [0.5, 0.5])
        time.sleep(0.05)
        if abs(claw_state.effort[0]) > 0.6:
            print("【触觉反馈】阻力过大，停止追加闭合")
            break
        target_pos += 2.0
```

**总结**：一次完美的实机硬件卡死排查，胜过在仿真里跑通一万次。理解物理边界（回差、单蜗杆双蜗轮自锁、堵转），敬畏硬件极限，用软件架构兜底，这就是 Sim-to-Real（仿真到现实）的核心要义。

## 🔍 第六阶段：v49 左爪问题根因、现象与 launch 标定

### 1. 问题不是「纯硬件坏了」

| 现象 | 说明 |
|------|------|
| launch 日志 `关节15 开≈+3.7°、关≈+3.7°` | **开限位标定失败**（健康左爪开限位约 **-43°**，关约 **+3.7°**） |
| `debug_left_claw.py` pos 随 cmd 变化，effort≈0.1A | **同一次 launch 内**电机/CAN/ROS 正常，非电机报废 |
| 手动掰开仍被关死 | **启动标定关限位 homing** 在抓取前已用大电流关一次，与手动预张开无关 |
| 抓取 `close 90 + effort 1.5` | 设计上的强力闭合，在标定异常或机械余量小时触发二次自锁 |

结论：**软件触发（标定 + 参数）+ 硬件结构（蜗杆自锁）** 共同作用；非单一原因。

### 2. launch 标定日志判读

在 WBC 终端搜索：`[LEJU claw]:关节` / `开爪限位位置` / `关爪限位位置`。左右爪**独立标定**，可一侧正常、一侧失败。

**左爪（关节 15，CAN ID 0x0F）**

| 关节15 开 | 关节15 关 | 含义 |
|-----------|-----------|------|
| ~**-43° ~ -47°** | ~**0° ~ +4°** | 标定正常，可抓取 |
| ~**+3.7°** | ~**+3.7°** | **开限位标定失败**；可试同 session 内 debug，**勿重启 launch** |
| debug 该侧 pos 不随 cmd 变化 | — | 行程映射塌缩或物理自锁 |

**右爪（关节 16，CAN ID 0x10）**

| 关节16 开 | 关节16 关 | 含义 |
|-----------|-----------|------|
| ~**-43° ~ -48°** | ~**0° ~ +4°** | 标定正常 |
| 开 ≈ 关（如 **-48° / -48°**） | 同上 | **关限位寻址失败**（超时后驱动把当前位置当作关限位）；该侧 `pos` 常卡在 **100.0**，扫参不动 |

**v49 实机失败样例（2026-06-25 launch 日志）**：

```
关节 15 开: -46.9505   关: -0.0473022    ← 左爪正常（行程 ~47°）
关节 16 开: -48.0369   关: -48.0064      ← 右爪异常（行程 ~0.03°）
```

同一次 launch 下 `debug_left_claw.py --side left` 左爪 pos 随 cmd 变化；`--side right` 右爪 pos 始终 **100.0** → **不是测试程序问题，是右爪标定塌缩。**

### 3. 「以前同样启动没事，现在有事」

* 首次大电流自锁会损伤蜗杆间隙，左爪开限位比右爪更极端（约 -43° vs -9°），更易在关限位 homing 时顶死。
* 机制健康时同样流程能扛；余量变小后**同一条启动链**反复越过临界点。

### 4. 抓取前物理与流程 Checklist

1. **退轴（左右都要）**：爪尖不通电能晃 **0.5~1cm**；**不要只修一侧**（只进轴/退轴一边会导致下次 homing 不对称）。
2. **只 launch 一次**，核对 **关节 15、16 的开/关角** 均有明显行程（见上表）。
3. 跑完诊断测试后**同 session 抓取**，不要无故重启 launch。
4. 重启前**两侧**再次退轴。

### 5. 为何「同一 launch、啥配置都没改」却像赌运气？

并非 ROS 或 Python 随机出错，而是 **每次 launch 前机械初始状态不同** + **homing 必做一次大电流关限位**：

| 你做的（物理） | 常见结果 |
|----------------|----------|
| 直接 launch，右爪关限位寻址失败 | 右爪 dead，左爪可能正常 |
| 仅右爪**进轴**（收一点）再 launch | 右爪可能恢复，**左爪关 homing 可能顶死** |
| **两侧退轴**后再 launch | 两侧开/关限位都能寻到 → 双爪正常 |

结论：**变的是蜗杆回差与初始角，不是配置文件。** 第三次「都正常」是因为退轴把余量救回来了，不是运气。

## 🧪 第七阶段：诊断脚本、测试方法与 v49 左爪实测结果

### 1.1 零脚本探针（可选，launch 后第一条指令）

```bash
rosservice call /control_robot_leju_claw "data:
  name: ['left_claw', 'right_claw']
  position: [10.0, 10.0]
  velocity: [50.0, 50.0]
```

* **通过**：双爪微张/保持张开，`rostopic echo /leju_claw_state` 中 effort **< 0.6A**。
* **失败**：某侧往关的方向死死顶紧、effort **> 1A**，或驱动刷 `检测到卡死(关闭方向)`、`+3A冲击模式` → **停测，退轴，不要继续 sweep/抓取**。

### 2. 测试一：基础扫参

```bash
python3 src/demo/vla_grasp/debug_left_claw.py --side left --pause 3
```

### 2. 驱动层机制（排查时必读）

* **服务链路**：`rosservice` → `LejuClawController` → `liblejuclaw.a` → CAN **0x0F / 0x10**，中间无 Python 反转。
* **关限位 homing**：约 **1.5~1.8A** 顶堵转；与手动是否掰开无关，**每次 launch 都会执行**。
* **卡死保护**：`control_thread` 检测到关闭方向堵转后，可能进入 **`+3A` 冲击模式**（`IMPACT_CURRENT = 3.0A`），此时 `success: True` 只表示服务收到，**不代表夹爪按意图运动**。
* **配置路径**：运行时读 `/home/lab/.config/lejuconfig/config.yaml`；launch 日志 `[LejuClawController] new config path` 指向 `src/kuavo_assets/config/config.yaml` 仅为拷贝来源。
* **`Negtive joint ID:` 为空**：日志中常见；左爪开限位 historically 更敏感。若反复单侧关限位失败，可与售后确认是否在 `negtive_address` 中加入 **0x0F / 0x10**（**改前备份**，勿盲目双侧同改）。
