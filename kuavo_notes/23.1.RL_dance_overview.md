## 📌 前情提要与最高工程红线

在本项目的前置阶段，基础的行走强化学习（Walking RL）已经成功在悬挂龙门架安全保护绳的真实 Kuavo 机器人上顺利跑通。当操作手第一次在真机实测中按下遥控器的 `B` 键切换至 RL 控制模式时，机器人展现出了像棉花一样柔顺的踏步姿态。然而，当我们准备跨越这道技术门槛，让这台钢铁之躯执行复杂的“全身舞蹈（跳舞）”任务时，内心的恐惧感与初次成功的成就感形成了极其鲜明的对比。

必须明确的是，这台极其昂贵的 Kuavo 四代机器人并非个人资产，一旦在测试中损坏任何一个核心物理关节，其返厂维修、拆解、重新标定装配的周期将长达数月之久。因此，在整个项目推进的过程中，我们确立了不可撼动、至高无上的安全纪律防线，其最高优先级被死死锁定为：**龙门架绝对不解绳、动作姿态维持极低动态、操作现场必须时刻具备物理级急停能力**。

* **适用硬件终端**：Kuavo 四代机实体（底层系统物理标定为 `ROBOT_VERSION=49`）。

* **系统环境栈**：Ubuntu 20.04/22.04 操作系统 + Isaac Lab 1.4.1 仿真框架 + Isaac Sim 4.2.0 物理引擎 + ROS Noetic 底层通信架构 + MuJoCo 验证引擎。

* **文档体系说明**：本系列属于 `23.x` 舞蹈专项深度延伸（该系列是 `15.x` 的基础行走标轨 RL 系列的进阶补充）。

## 第一章：项目背景、核心动机与安全战略定调

### 1.1 从基础行走 RL 跨越至全身协调舞蹈

在此之前，`15.x` 系列已经把 Kuavo 四代机器人的速度跟踪行走任务彻底跑通了。实践证明，基于 PPO 算法 + `rsl_rl` 算法库 + ONNX 模型导出 + MuJoCo/真机底层部署的这条闭环技术链路在四代机硬件上是完全可行的，这也构成了我们向高难度“全身舞项目”进军的最强信心来源。

然而，行走与舞蹈在控制目标和技术实现上存在着本质的差异：

* **基础行走任务**：网络只需要跟踪 `(vx, vy, ω)` 这三个方向的速度指令变化，速度指令在训练中是随机采样的，机器人的观测空间约为 87 维（基于 Flat 平地环境），奖励函数的核心目标是跟踪线速度（`lin_vel`）。

* **原地舞蹈任务**：速度指令被死死锁定且恒定为 0（即执行绝对的原地律动控制），机器人的观测空间从 89 维暴涨延伸至 115 维，奖励函数的核心目标转变为严格跟踪 CSV 文件提供的关节运动寿命轨迹。

在深度强化学习中，我们在同一张 8GB 笔记本显卡（如 RTX 3070）上实测：官方适配的 **S42/S46 历史链**可稳定拉起 **`num_envs=8192`**，DIY 的 **S49 满血链**可稳定拉起 **`num_envs=4096`**，二者显存占用均约在 **`6.8GB`** 附近。若仍触发 CUDA OOM，再逐级下调并发数。

#### 1.1.1 为什么 S49 链 `num_envs` 只能减半，但总显存仍约 `6.8GB`、且 `steps/s` 明显低于 S42/S46？

这不是「49 版本更费显存」的简单结论，而是 **单环境（per-env）算力与显存成本暴涨**，只能靠把并行数从 **8192 砍到 4096** 才能在同一张 8GB 卡上站稳——砍完以后，总占用仍与 S42/S46 的 8192 档接近（均约 **`6.8GB`**），所以看起来像「环境数减半、显存却没省」。

* **单 env 观测与网络缓冲区更大**：S49 满血链 Policy 输入 **115 维**（较 89 维基线多出 26 维 `reference_joint_pos`），Rollout 采样张量、PPO 的 Actor/Critic 前向与优势估计，对每个 env 的**每步缓存**都同比变胖。

* **单 env 每步计算更重**：除 115 维 `ObsTerm` 外，还有 `track_punch_*` + **`upright` 门控**、`flat_orientation`/`base_height` 等额外奖励算子；1565 帧 CSV 的 GPU 常驻轨迹表（较 642 帧约 **2.4×** 更长）也占去一块固定显存——这些在 S42/S46 早期链上要么没有、要么更轻。

* **物理资产管线更重（`steps/s` 主因）**：S42/S46 历史链加载的是 **`biped_s46.usd` 预烘焙资产**，PhysX 直接吃离线 USD，控制台 `total_fps` 可达 **~43000 steps/s**。S49 链走 **`biped_s49_26dof_lite.urdf` 运行时 URDF→USD 转换**（即使用了 lite 剥离碰撞，仍比预烘焙 USD 多一段编译与求解开销），实测 **`~16000 steps/s`**——约为 S46 的 **37%**。

* **工程直觉公式**：$\text{总显存} \approx \text{num\_envs} \times \text{单 env 成本}$。S49 的 **单 env 成本 ≈ 2× S46**，所以 **4096 × 2 ≈ 8192 × 1**，总显存打平在 `6.8GB` 附近；同时 **单 env 物理+奖励更慢**，故即使 `num_envs` 减半，**全局采样吞吐（steps/s）仍显著低于 S42/S46**。


在算力仍然紧张的困境下，如果让纯粹的强化学习从零开始盲目探索编舞，是根本不现实的。因此，引入**模仿学习+强化学习（IL+RL）**，通过导入 CSV 动作文件给予机器人极其强大的“编舞先验知识”，成为了在有限算力下唯一务实且明智的突围路径。

### 1.2 模仿学习与强化学习（IL+RL）控制管线四步走

为了将训练好的模型安全地部署到物理世界，我们确立了严密的四步走串联控制管线：

1. **第一步：生成参考轨迹 CSV 文件**
利用外部的动捕数据、DeepMimic 动作库，或者是乐聚官方提供的轨迹生成器，提取出目标关节角度 $q_{\text{ref}}$ 的时间序列矩阵。核心铁律：CSV 表格中的列排序（关节顺序）必须与底层常量 `KUAVO_RL_JOINT_NAMES` 达成 100% 绝对一致，哪怕错位一个索引，训练再久出来的依然是令人崩溃的僵尸站姿。

2. **第二步：设计严苛的模仿奖励函数**
在 `rewards.py` 文件中手搓 `track_punch_*` 相关的指数核正向奖励函数，逼迫机器人的动作向参考轨迹靠拢，动作越像参考动作得分越高。然而，单靠模仿是远远不够的，为了防止机器人为了追踪动作而放弃平衡生存，必须同步施加极高权重的 `flat_orientation`（躯干抗倾倒惩罚）、`base_height`（底盘物理中心高度保持惩罚）以及原地禁移动惩罚。

3. **第三步：启动 PPO 并行仿真训练**
在 Git 仓库中切出完全独立的 `feat/punch` 分支，彻底隔离原有的行走 baseline。根据版本与显卡压力：S42/S46 链设定 `num_envs=8192`、S49 链设定 `num_envs=4096`（8GB 卡实测约 `6.8GB` 显存）；若爆显存再降。在训练过程中，必须时刻死盯 TensorBoard 曲线面板，监控 `mimic` 模仿奖励与姿态惩罚之间是否达成平衡——如果模仿得分极高但机器人在仿真里却趴在地上，那绝对是陷入了致命的“局部最优解”。中途可用 **`train.py --video`** 录训练视频肉眼抽检（**23.6** §1.4.1）。

4. **第四步：ONNX 导出 → Sim2Sim 验证 → Sim2Real 真机部署**
导出前对候选 checkpoint 跑 **`play.py --video`** 肉眼验收（**23.6** §1.4.2），再 headless 执行 `play.py` 进行 ONNX 导出，随后执行 `catkin build` 重新编译底层 C++ 节点。在 MuJoCo 仿真器中验证观测链路无误后，最终进入真机，挂上龙门架保护绳进行低动态的首次点火验收。在部署侧，哪怕 obs 数据切片错了一位，神经网络收到的就是一串杂乱无章的垃圾向量，对齐难度极高。

### 1.3 为什么必须摒弃经典的“角度开开环重放”？

经典控制领域的开环重放做法极其直白：直接给双臂电机下发角度轨迹，同时将双腿死死锁在默认的站立角度上。这种方法在仿真器里看数据的“确定性确实更高”，但它完全忽略了上半身质量所带来的巨大动量对整机重心的灾难性耦合影响。

* **开环重放的物理缺陷**：机器人一次大幅度地向前挥拳，在物理层面上等同于给一个倒立摆施加了一记巨大的脉冲外力扰动。如果下半身没有进行主动的闭环力矩补偿，这股巨大的惯性在真机上会立刻转化为剧烈的前后晃动。当晃动幅度突破阈值，真机内部的 IMU 状态估计器就会彻底发散，随即触发底层的灾难性硬件保护。

* **IL+RL 闭环控制的高阶价值**：而在我们设计的强化学习闭环策略中，网络在每一个仿真帧都会综合 `projected_gravity`（投影重力）、`joint_pos`（当前角度）以及 `reference_joint_pos`（参考目标角度）等上百维状态，智能地演算并下发 26 维度的实时力矩输出。它既能柔顺地跟踪 CSV 的动作轨迹，又能主动维持零矩点（ZMP）死死处于双脚构感知的支撑多边形内部。此外，借助于域随机化（Domain Randomization）机制，训练出的策略对真机的装配公差表现出极强的脱敏与免疫能力，这是手写固定 PD 轨迹永远无法启及的。

### 1.4 选择 `humanoid3d_punch` 作为首验动作的战略考量

在极其紧张的项目周期和严酷的安全红线约束下，来自 DeepMimic 库中的 `humanoid3d_punch.txt` 原地冲拳动作数据成为了我们挂绳首验的完美理想型。原因在于它具备**原地站立、低动态、以绝对上肢运动为主**的黄金特性。在进行数据映射重定向时，我们可以极其安全地直接丢弃人类模型的下肢捕捉数据，并将数据表中的第 0 至第 11 列硬编码填满四代机默认的安全静态站立角度。用打拳动作去验证整个 IL+RL 闭环控制的骨架通路，比一上来就高估算去啃极其庞大复杂的 AMASS 全动作库要务实和安全无数倍。

随着项目的推进，动作库也经历了从单兵打拳到全身共舞的平滑演进：我们首先提取了 Roban S14 以 100fps 采集的新年舞数据，通过 `convert_punch` 脚本提炼出 642 帧的 `PERFECT_LIMIT` 版本 CSV；随后又通过 `adapt_dance_csv fullbody_inplace` 脚本对其进行了绝对的原地化剪裁。整个迭代过程的动态水平严格遵循“从低到高”的迭代规律，真机的物理安全边界始终保持极限收紧。

与 15.x 系列的分工与隔离纪律如下：

* **15.1**：环境安装部署；**15.2**：通用强化学习训练；**15.3**：通用 Sim2Sim 验证；**15.4**：真机操作黑皮书规范。

* **23.x 专项延伸**：属于全尺寸舞蹈动作的深度探索，为了保持极高的信息密度，绝不重复赘述 `15.x` 中的通用环境搭建内容。

* **分支隔离纪律**：行走模型和配置属于 `master` 分支，舞蹈模型在 `feat/punch` 分支。**绝不允许在 `master` 分支上修改打拳（punch）的任何奖励函数配置；绝不允许使用行走任务跑出来的 checkpoint 作为舞蹈策略的初始化权重**。如果在深度魔改中把底层奖励逻辑彻底改崩，只需要在终端输入一句轻巧的 `git checkout master` 即可瞬间完成时光回溯。

### 2.1 30秒通读指南：建立高效率的阅读检索顺序

为了让后来者不再重复我们经历过的痛苦摸索，我们被 MuJoCo 仿真引擎中的“离奇后倒”、“诡异劈叉”以及“失控侧移”各路摧残了一遍之后，特此将整个舞蹈项目的技术细分模块梳理出了**最少踩坑、最高效的黄金阅读与排雷路径**：

1. **分卷 23.2** —— 聚焦于动作数据的绝对源头，阐述如何利用数据转换管线脚本将零散、不兼容的外部动捕原始动作 CSV 矩阵，进行跨代次、高难度的重定向剪裁与插值，生成符合 4 代机 26 DoF 电机拓扑顺序的黄金文件。

2. **分卷 23.3** —— 专注剖析隐藏在 `feat/punch` 专属平行分支底下的核心训练架构，深度解剖重组后的 115 维大长数组观测空间，以及为了让老旧系统在运行新一代模型时不发生保护性发散崩溃，而必须强行在前置阶段干掉的 S49 硬件四大核心基础代码天坑抢修大手术方案。

3. **分卷 23.4** —— 当你在 MuJoCo 仿真中按下 `R` 键进入闭环控制，机器人却发生各种诡异畸变动作时，指导你如何将底层 C++ 拼装的 `singleInputData` 结构体与 Python 训练侧生成的 `env.yaml` 配置文件进行逐字逐行的逆向地毯式对齐审计。

4. **分卷 23.5** —— 记录 duck_sit（`49duck_sit.onnx`）在 MuJoCo 上按 R 即倒、Hybrid WBC+RL 扫参与 bag 验收的全部部署攻坚实录；**调试期 launch 调参只写在本卷，不写进 23.6**。

5. **分卷 23.6** —— 系列最后一卷：无脑、原子级终端命令工卡（Train → Export → 编译 → **默认 launch** → 真机）；**不收录**某次调试的 `sim_wbc_*` roslaunch 后缀。

6. **规范 15.4** —— 搬出写满物理鲜血教训的真机实体验收点火 SOP，时刻约束操作人员紧绷神经，铭记舞蹈模式的动态容差和出拳速度必须严格低于行走模型。

### 2.2 当前 S49 统一推荐链路速查卡（2026-06 现场版）

当新人第一次接手四代机舞蹈链路、需要在五分钟内建立「训练—导出—部署—Sim2Sim」的全局坐标系时，请把下面这张速查卡当作导航罗盘——它浓缩了我们在 iter 7461 惨案之后最终拍板定稿的唯一推荐标准，而不是历史过渡态的 S42/S46 混合链路：

* **观测维度**：115 维（89 维基线 + 26 维 `reference_joint_pos` + 相位仍固定在最末 2 维）。

* **策略导出物**：`policy_s49.onnx`（115 维入参，与 `numSingleObs=115` 死锁）。

* **Sim2Sim 最低门槛**：MuJoCo 在 **joystickSimulator 窗口**完成 **R→F→P** 后原地跳舞 **≥30 秒**、无侧移、无劈叉、无挺尸——细则见 **23.4** §7.1 与 **23.6** 第四章末尾。

## 第三章：版本错位危机剖析与统一 S49 技术决策

### 3.1 跨版本缝合的“三体”历史技术债危机

在项目前期走向真机部署的过程中，由于缺乏统一的版本规划，我们长期面临着一个极其割裂的“三体”混合部署危机：

* **训练侧（神经网络大脑）**：训练配置文件的名字虽然叫 `S42`，但其实底层在实例化资产时加载的却是 `Kuavos46_CFG`，并且调用的是包含 26 个自由度预烘焙的 `biped_s46.usd` 模型。

* **部署侧（控制骨架皮囊）**：底层的 C++ 仿真验证节点为了能够跑通接口，强行在启动变量中指定加载 `ROBOT_VERSION=46`。

* **真实物理侧（硬件真身躯壳）**：实验室中等待指令下发的真实 Kuavo 物理硬件，其物理出厂硬件代次是标准的 `ROBOT_VERSION=49`。

#### 3.1.1 这种“移花接木”为何前期能够跑通？

在基础行走 RL 的调试阶段（`15.4` 章节），我们为了绕过控制系统对文件夹命名的“启动白名单拦截”，曾经铤而走险地采用了配置 `export ROBOT_VERSION=45` 的系统伪装欺骗手段，以此强行把 v49 硬件测得的物理真值参数塞进了 v46 的强化学习接口骨架中。然而，这种移花接木无法掩盖机身连杆几何长度、整机质量分布、转动惯量以及脚踝连杆力学解算器在跨代硬件上所表现出的巨大断层。随着舞蹈动作动态的加大，代码里堆叠的补偿和补丁开始疯狂冲突发散。

#### 3.1.2 长期不统一版本的毁灭性风险清单

如果我们继续固执地沿用旧骨架，以下雷区将随时引爆系统：

* **通道严重错位**：观测空间和关节控制序列在跨版本解析时极易发生毁灭性的索引移位。

* **逆向换算失效**：用于导出的 `lab2gym` 映射矩阵是专门针对 S46 的老旧 USD 资产进行硬编码标定的，在 S49 硬件上完全失效。

* **力矩计算 NaN**：由于 S49 原始 URDF 的几何形体和碰撞网格与老旧骨架不同，物理引擎在求解接触力矩时会瞬间算错，产生大量的 NaN（非数字）空值，从而被底层安全防线当场下发 `dof_pos_illegal` 死亡终止斩杀信号，触发系统崩溃。

* **结论**：真机物理硬件是 S49，如果训练却依然套用 S46 的骨架结构，那么每一次的下发部署都等同于是在拿昂贵的机器人赌博。

### 3.2 终结历史债务：全量统一 S49 平行战略

认清这一严酷的物理现实后，我们做出了彻底终结混乱代差的最高技术决策：**训练配置、MuJoCo 验证、真机点火下发，三端的所有环境和变量全部强制、无条件统一为 `ROBOT_VERSION=49`！**

## 第五章：Sim2Sim 部署的观测链路死锁与“五幕剧”灾难复盘

当我们在 Isaac 的无界面 headless 仿真中训练完成、看完了完美的 play.py 动作回放、并满怀信心地将 89/115 维的 policy ONNX 大脑挂载进 MuJoCo 仿真中（在 **joystickSimulator 窗口**完成 **R→F→P** 解锁连招）时，机器人却频繁发生直接后倒砸碎、或者原地瞬间劈叉解体抽搐的惨烈事故。这一连串灾难，在我们的开发日志里留下了刻骨铭心的“五幕大剧”。

### 5.1 记录在控制台里的 Sim2Sim 五幕剧


> **术语**：下文「按 R 键 / RL 接管」在 **Sim2Sim** 中指 **joystickSimulator 窗口 R→F→P** 整套连招（**15.3** §5.2）；**真机**指 **F 拨杆 WBC 起立 → B 键**（**15.4** §5.2）。单按一键不足以完成解锁。

#### 🎭 第一幕：按 R 键的瞬间直接“挺尸后倒”

* **【发病表象】**：MuJoCo 启动顺利，Run 状态指示灯成功变绿，WBC 托举状态极其稳定。但当调试手敲下键盘上的 `R` 键将控制权交给强化学习的一瞬间——机器人犹如被高压电瞬间抽掉了整条脊髓一样，毫无抵抗地笔直向后砸倒在地面上，而最可怕的是终端控制台里一片寂静，没有任何一行标红的报错红字。

* **【深度病理剖析】**：由于同一权重在 Isaac 原生的 `play.py` 里跳舞极度规范，直接排除了权重有问题的可能。我们被迫在 C++ 源码中植入 `std::cout`，逐个维度打印拼装完毕的 89 维 `singleInputData` 长向量，终于将病因大白于天下：本应随着时间轴自增滚动的相位数据 `commandPhase` 被堆在了整个张量的最前面，并且真值永远被死锁在 `[0,1]` 之间毫无波动；而本该是 `[-1,0,0]` 的投影重力三维坐标位置，居然被塞入了一套表示姿态的 `baseEuler` 欧拉角角度！同时关节速度被乘了 0.05。

* **【工程解药】**：这是一场由于 `skw_rl_param` 配置里的字典键名顺序完全写反、`play.py` 内部解包时把打拳相位错插进了关节角度矩阵的中央、以及 `cycleTime` 时间轴断裂所引发的三层底层 bug 的史诗级叠加。我们必须通过重构 `play.py` 的 `forward` 算子，在 ONNX 内部写死一套空间重组矩阵，将 C++ 传过来的 8 键数据重新像扭魔方一样，按照字母表的顺序重组后，再喂给网络大脑。

#### 🎭 第二幕：下盘能舞动，但如同梦游般顽固地往前跌蹭

* **【发病表象】**：观测链路的错位抢修完毕后，机器人再次在 MuJoCo 里拉起，按 R 键后终于不再倒地，而是稳稳站住并开始了极具科技感的全身舞蹈——但仅仅跳了 3 秒钟，机体就伴随着上肢的挥舞，开始不可控制地、一颠一跛地朝着前方连续滑步跌蹭，并最终因为摩擦力失衡而往前栽倒。

* **【深度病理剖析】**：我们仔细审阅了前期为了在 S42 过渡阶段图省事而提炼出的 `PERFECT_LIMIT` 动作 CSV 文件，发现原作者在动捕录制时，脚步发生了严重的迈步与重心转移，导致数据表中的右腿部分（第 7–11 列关节数据）在时间轴上出现了高达 `3.24 rad` 的大范围弧度起伏。即便我们的 `velocity_commands` 发令枪已经死死限制在 0，也无法阻止这种骨骼和轨迹内置的迈步位移冲动。

* **【工程解药】**：我们必须在系统外部启动专门研发的裁剪洗沙脚本 `adapt_dance_csv fullbody_inplace` 对动作进行绝对的“原地化”物理阉割；并且在训练侧的 `RewardsCfg` 里重新追加挂载了一项高达 `-1.3` 权重惩罚的 `base_lin_vel_xy_stationary` 函数，双管齐下，才在下一炉的策略中彻底压制住了这种致命的物理前栽漂移。

#### 🎭 第三幕：令人毛骨悚然的按 R “瞬间劈叉”解体案

* **【发病表象】**：为了给机器人注入更强大的高维空间感官，我们将观测维度从 89 维暴跳升级至 115 维，正式上线了全关节追踪器 `referenceJointPos`。然而，当编译完毕、拉起仿真并敲下 `R` 键接管的一瞬间——伴随着一声闷响，机器人像被无形的巨力硬生生撕扯成了一字马劈叉，关节瞬间发生毁灭性的过载畸变。

## 第六章：S49 训练端的连环爆雷陷阱与 Iter 7461 躺平局部最优

在版本号彻底大一统为 S49 后，我们本以为能够迎来大丰收，但在真正的开炉炼丹过程中，一系列由算法原理和物理层撞车引发的“连环爆雷”，再次将团队推向了深渊。

### 6.1 PPO 训练大军的物理机制崩塌

在训练的开局阶段，由于我们直接将官方的完整版大 URDF 模型塞入了环境，导致在并行采样（Rollout）时，PyTorch 的显存开销和物理计算图的规模发生了灾难性暴涨。

在经历了几千代迭代后，TensorBoard 吐出了极其诡异的读数：动作追踪得分 `track_punch_imitation` 飙升到了不可思议的极高点，但策略的生存奖励却大幅度发散，模型在 `play.py` 可视化时直接表现为：**一出生就立刻摔倒，然后像一滩软泥一样舒适地瘫软平躺在零海拔的地板上，在地面上极其认真、精准地挥舞着手脚，跟随着 CSV 的节奏在重演舞蹈动作！**

#### 6.1.1 致命的“躺平刷分”局部最优解因果直觉

这是深度强化学习（PPO 算法）中最具黑色幽默的**局部最优解（Local Minima）陷阱**。

* **网络的小聪明**：双足机器人的站立平衡（维持 ZMP）需要耗费极高的探索代价。智能体在经历了无数次开局摔倒后，突然发现了一条绕过规则的“作弊捷径”——与其费尽心思顶着几十公斤的自重去跟重力做对抗，不如直接在第 1 个 step 就主动“爬倒挺尸”躺在地上。

* **机制漏洞**：因为一旦躺倒在平地上，它就永远不可能再次跌倒、也不可能再次触发 `base_contact` 死亡终止，在这个极度安全的“平躺安全屋”里，它只需要悠闲地挥舞双臂，就能把我们设定的 `weight=15.0` 的巨额打拳模仿奖励给轻松全量刷满！这就导致了 TensorBoard 曲线华丽无比、但策略在物理世界上形同废纸的“iter 7461 躺平惨案”。

### 6.2 引入域随机化（Domain Randomization）给机器人注入抗噪肌肉

为了让策略彻底放弃躺平思维，并在未来的真机部署上对装配公差表现出极强的免疫力，我们在 `EventCfg` 环境事件中激活了极其残暴的**全量域随机化护城河**：

1. **质量随机化（Mass Randomization）**：在 4096 个平行环境中，每个机器人的机身躯干（`base_link`）质量会被随机赋予 `±10%` 的物理漂移。这能强迫神经网络主动感知由于质量突变带来的惯性差异，学会在出拳时自适应地调节调节力矩。

2. **摩擦力随机化（Friction Randomization）**：地表的静摩擦力和动摩擦力系数在 `(0.0, 2.0)` 之间进行高频的随机抽样。有的宇宙滑如冰面（摩擦力趋近零），有的宇宙粘如地毯（摩擦力高达2）。这能逼迫机器人的脚踝和髋部学会在不同的抓地力环境下牢牢锁死下盘，不发生侧移滑步。

3. **电机强度随机化（Actuator Gain Randomization）**：各个关节电机的最大力矩限制（`torqueLimit`）和位置增益系数在 `±5%` 之间发生随机漂移，以此抹平真机因为电机长期使用、发热过载所带来的硬件物理衰减。

## 第八章：全局排查与点火核对清单、常见误区与未完之战

### 8.1 必须彻底澄清的四大开发认知误区

#### ❌ 误区一：只要训练时的 Loss 低、Reward 分数高就能直接部署真机

* **【真相大白】**：这是最经典的算法工程师自嗨陷阱。如果你的网络输出重组逻辑 `play.py` 里由于乱序把 `lab2gym` 漏掉了，那么在 C++ 控制器的眼里，你的大脑输出的就是一串毫无规律的垃圾向量。网络以为自己发出了精妙的平衡力矩，实机上电机却在疯狂反向打架。**比把Loss降下去难上十倍的，是确保两端数据链路的语义和索引严丝合缝。**

#### ❌ 误区二：改完奖励和配置文件后，直接继承（resume）之前的旧权重继续训能省时间

* **【真相大白】**：在统一 S49 的决策落地后，我们对抗性的四块核心代码修复（修改执行器算子、精简 URDF 碰撞体、挂载 `upright` 门控）已经彻底颠覆了原有的马尔可夫决策决策环境。旧的 checkpoint（例如大名鼎鼎的 7461 权重）其内部神经元携带的完全是不兼容的旧环境旧记忆，如果强行挂载 resume，只会让网络在错误的道路上南辕北辙，继续导致起飞失败。**必须破釜沉舟，清空缓存，从 iter 0 重新点火重训**。

#### ❌ 误区三：在 `rewards.py` 里每一步都用 Pandas 读 CSV 会严重拖慢物理引擎的采样频率

* **【真相大白】**：不用自己吓自己。只要你像我们一样，在代码里写了全局静态张量缓存机制 `if _PUNCH_REF_DATA is None:`，那么整张 1565 帧的打拳动作表在开局的第 1 个 step 就已经被一次性读完并深度常驻留在了 `cuda:0` 的 GPU 连续高频显存中。在后续的几千万步更新里，系统执行的纯粹是极高效率的内存切片（Slicing）索引查表访问，根本不会产生任何磁盘 I/O 阻塞瓶颈。

#### ❌ 误区四：在 MuJoCo 验证抗扰性能时，直接用鼠标在窗口里拉扯机器人身体最直观

* **【真相大白】**：这是严重的刻舟求剑。乐聚原厂的 `mujoco_node.cc` 仿真节点在编写时存在闭环漏洞，它的物理求交积分大循环处于一个完全独立的后台线程 `PhysicsLoop` 里，并没有在主线程中去调用渲染器的 `mjv_applyPerturbForce` 函数。你在窗口里用鼠标拉出的红色好看箭头，外力根本无法参与物理结算。正确的抗扰测试方法，是严格通过 ROS 接口向 `/external_wrench` 话题高频灌入真实的力矩消息，强行让 C++ 侧把数据写进 `d->xfrc_applied` 缓冲区中。

### 8.4 从 chat 沉淀的三条「最后一公里」备忘

1. **Checkpoint 不是越大越好**：2500–3000 iter 常为 mimic+平衡双巅峰；6000+ 可能梯度污染（详见 **23.3** §7.3、**23.6** §2.2）。

3. **终端环境**：Zsh **`zoxide` 钩子**可吞多行 `export`；Isaac 点火优先 **Bash + 单行 `&&`**（**23.6** §0.1.4）。
4. **Sim2Sim 操作窗**：R→F→P 必须在 **`joystickSimulator.py` 窗口**（launch 自动弹出；Docker 无弹窗时手动 `python3 .../joystickSimulator.py`，见 **23.4** §9.5.2、**23.6** F13）。
5. **训练侧行走惩罚**：从 master 行走配置 fork 时，`stand_still_without_cmd` / `joint_deviation_arms` 等须置 **0** 或注释，否则双臂被锁死（**23.3** §1.4）。
