# Kuavo (夸父) 机器人仿真与深度调试全攻略 (综合标准版)

本指南涵盖了从基础 Docker 仿真环境搭建、默认模型运行，到高阶外部地图加载及 Debug 排错的完整 SOP。

## 一、 仿真环境部署与管理 (Docker)

在没有真实机器人硬件的情况下，推荐使用 Docker 环境进行代码编译和仿真。

### 1.1 镜像获取与导入
1. 确保已经完成下位机代码仓库的克隆：[下位机代码仓库](https://gitee.com/leju-robot/kuavo-ros-opensource.git)
2. 获取 Docker 镜像（自行构建或下载官方编译版）：
   ```bash  
   # 官方容器下载链接：
   [https://kuavo.lejurobot.com/kuavo_research_editiion/docker_images/kuavo_opensource_mpc_wbc_img_v1.3.0.tar.gz](https://kuavo.lejurobot.com/kuavo_research_editiion/docker_images/kuavo_opensource_mpc_wbc_img_v1.3.0.tar.gz)
````

3.  导入容器镜像：
    ```bash
    docker load -i kuavo_opensource_mpc_wbc_img_v1.3.0.tar.gz
    ```
宿主机环境准备:
# 1. 设置机器人版本（以4.5版本为例）
echo 'export ROBOT_VERSION=45' >> ~/.bashrc
source ~/.bashrc

# 2. 准备机器人总质量配置文件（必须，否则模型校准会报错）
mkdir -p ~/.config/lejuconfig
echo "45" | sudo tee ~/.config/lejuconfig/TotalMassV${ROBOT_VERSION}

# 3. 允许 Docker 访问宿主机显示器（解决 Gazebo 界面显示问题）非常重要
xhost +local:docker
```
### 1.2 启动与进入环境 (核心常规操作)

根据你的环境状态，选择不同的进入方式：

**情况 A：第一次初始化（创建新容器）**
在宿主机的 `/home/lab/kuavo-ros-opensource` 目录下执行官方脚本创建并进入容器（默认映射目录为 `/root/kuavo_ws`）：

```bash
./docker/run.sh
```

**情况 B：日常开发（已有容器，直接运行）**
如果容器已经创建（假设命名为 `kuavo_container`），日常开发只需两步：

```bash
# 1. 唤醒/启动后台容器
# 2. 进入交互式终端 (推荐使用 zsh 获取更好的命令行体验)

docker start kuavo_container
docker exec -it kuavo_container zsh
```

### 1.3 编译代码工程

进入容器后，在工作空间目录下执行以下命令开始编译：

```bash
# 清理旧缓存（推荐）
rm -r devel/ build/
catkin clean -y

# 配置编译参数 (Important! 避免找不到 ccache 编译器)
catkin config -DCMAKE_ASM_COMPILER=/usr/bin/as -DCMAKE_BUILD_TYPE=Release 

# 刷新环境并编译
source installed/setup.zsh
配置编译参数（强制 C++17 标准并指定编译器以支持 ccache）
catkin config -DCMAKE_ASM_COMPILER=/usr/bin/as -DCMAKE_BUILD_TYPE=Release -DCMAKE_CXX_STANDARD=17
catkin build humanoid_controllers # 会自动编译所有依赖项
catkin build humanoid_controllers gazebo_sim
```

*注：如果编译失败，请再次执行 `rm -r devel/ build/` 后重新编译。*

## 二、 基础仿真运行 (Gazebo)

编译完成后，可以先运行默认场景验证环境是否正常。以下命令将使用 Gazebo 物理引擎启动仿真。

### 2.1 运行完整全身仿真

此命令将在 Gazebo 的空地图（empty world）中生成完整的机器人模型，并启动控制器、MPC 和 WBC。

```bash
source devel/setup.zsh
roslaunch humanoid_controllers load_kuavo_gazebo_sim.launch record_bag:=false

```

### 2.2 运行仅上半身(带下肢支撑)仿真

适用于测试上半身动作（如编舞、转头），此时底盘相当于被固定或处于安全站立模式，大幅降低底层平衡算法发散导致 Gazebo 崩溃的风险：

```bash
source devel/setup.zsh
roslaunch humanoid_controllers load_kuavo_gazebo_sim_half_up_body.launch record_bag:=false
```

## 三、 高阶原理：仿真底层认知

在进行更复杂的 Gazebo 仿真及自定义地图开发前，请牢记以下三个底层逻辑：

1.  **地图与模型分离**：`.world` 文件只是图纸，真实的 3D 模型存在 `models` 文件夹里。Gazebo 必须知道去哪里找模型。
2.  **底层强锁原点**：Kuavo 的 C++ 插件 (`GazeboShmInterface`) 会在启动瞬间，把机器人强制拉回其 JSON 文件中定义的数学原点。
3.  **缓存机制**：底层自动微分库 (CppAd) 对物理参数极度敏感，修改任何质量、重心或初始位置后，旧缓存就是毒药，必须清除。

## 四、 高阶实战：加载外部地图标准执行顺序 (SOP)

请**严格按照以下 1 至 4 的顺序**执行，切勿跳步或颠倒。

### Step 1: 静态文件配置与代码修改 (改图纸)

*此步骤为静态修改，只需在第一次引入新地图或想改变机器人坐标时执行一次。*

1.  **改造 Launch 参数透传**：
      - 打开 `load_kuavo_gazebo_sim.launch`，在顶部定义参数，并在 `<include>` 中透传：
        ```xml
        <arg name="world_name" default="$(find gazebo_ros)/worlds/empty.world"/>
        <arg name="spawn_x" default="0.0"/>
        <arg name="spawn_y" default="0.0"/>
        <arg name="use_grab_box_scene" default="false"/>
        ```
      - 打开 `gazebo-sim.launch`，确保 `spawn_robot` 节点使用了传入的坐标：
        ```xml
        <node name="spawn_robot" pkg="gazebo_ros" type="spawn_model" 
              args="-urdf -model biped_s$(arg robot_version) -param robot_description -x $(arg spawn_x) -y $(arg spawn_y) -z $(arg spawn_z)"/>
        ```
2.  **处理机器人出生点冲突（二选一）**：
      - **方案A（推荐，改地图）**：写一个 Python 脚本，将 `.world` 文件里的建筑物整体反向平移，让目标房间对准 Gazebo 的绝对原点 `(0,0)`。
      - **方案B（改大脑）**：修改 `/root/kuavo_ws/src/kuavo_assets/config/kuavo_v42/kuavo.json` 第 68 行的 `"base_position": [15.0, 0.0, 0.74]`。
3.  **保存所有文件**：确保 VS Code 标签页上的小白点消失。

### Step 2: 声明环境变量 (铺路)

*必须在启动仿真的同一个终端窗口内、运行 `roslaunch` 之前执行！这决定了 Gazebo 能否找到建筑物等 3D 模型。*

```bash
export GAZEBO_MODEL_PATH=$GAZEBO_MODEL_PATH:/root/kuavo_ws/src/aws-robomaker-hospital-world/models
```

*(💡 进阶提示：执行 `echo 'export GAZEBO_MODEL_PATH=$GAZEBO_MODEL_PATH:/你的模型绝对路径' >> ~/.zshrc`，可使其永久生效。)*

### Step 3: 清理底层大脑缓存 (洗脑)

*只要你在 Step 1 中修改了 `kuavo.json`（比如改了质量、初始坐标等），就**必须**执行此步。*

```bash
rm -rf /var/ocs2/kuavo_v42
```

### Step 4: 启动仿真引擎 (点火)

*注意：如果刚执行了 Step 3 的清缓存，此步会在终端卡住 3-5 分钟显示 `cppad building start`，请耐心等待编译完成。*

```bash
roslaunch humanoid_controllers load_kuavo_gazebo_sim.launch \
  record_bag:=false \
  use_grab_box_scene:=false \
  world_name:=/root/kuavo_ws/src/aws-robomaker-hospital-world/worlds/hospital_shifted.world \
  spawn_x:=0.0 \
  spawn_y:=0.0
```

## 五、 🕵️ 侦探式 Debug 方法论

当终端报错被刷屏时，按照以下三步法排查：

### 1\. 从下往上找“遗言”

程序死亡前的核心原因，90% 都在倒数 10\~20 行内。眼睛只抓取带 `[FATAL]`, `[ERROR]`, `died` 的字眼：

  * **`required process [nodelet_manager] has died`**：核心控制节点阵亡。
      - *防闪退技巧*：去 `load_kuavo_gazebo_sim.launch` 里把 `nodelet_manager` 节点的 `required="true"` 改为 `false`，可防止 Gazebo 窗口闪退，方便观察摔倒姿态。
  * **`wbc_planned_joint_acc is out of range!`**：算法发散，算出离谱加速度。
      - *原因*：机器人出生卡在墙体内，或从高空坠落导致物理引擎排斥力过大。
  * **`Unable to find uri[model://...]`**：找不到 3D 模型。
      - *原因*：漏做了 [Step 2: 声明环境变量]。

### 2\. 检查 "战前总结" (SUMMARY)

运行 `roslaunch` 时，终端最开始输出的 `PARAMETERS` 列表极其重要。
如果传了参数（如 `world_name:=xxx`）但没生效，向上翻到 `SUMMARY` 区域。如果显示该参数是空字符串 `""` 或默认值，说明 Launch 文件透传逻辑写错了。

### 3\. 坚守“控制变量法”

  * 遇到问题先回退，运行一遍默认空地图测试。
  * 只有当空地图正常运行时，再逐步添加新变量（新地图、新坐标）。一次只改一个变量，不行就马上回退。
