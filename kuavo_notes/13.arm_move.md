# 🤖 人形机器人编舞与运动学控制：底层原理与实战排坑指南

这份笔记将从最顶层的编舞逻辑，到底层的微积分与矩阵算法，再到实际 Ubuntu 终端里的指令与报错排查，进行一次全链路的深度剖析。

---

## 一、 机器人运动控制的两大流派（编舞核心）

在人形机器人的运动控制中，“让手怎么动”是一个极其复杂的系统工程。根据控制维度的不同，业界存在两大流派：**关节空间控制**与**笛卡尔（任务）空间控制**。在实际的机器人编舞中，这两种方法往往是结合使用的。

| 对比维度 | 关节空间 (Joint Space) 编舞 | 笛卡尔空间 (Cartesian Space) 编舞 |
| :--- | :--- | :--- |
| **核心思想** | 直接控制每个电机的旋转角度 | 控制末端执行器（手心）在空间中的 XYZ 坐标 |
| **典型输入** | 关键帧角度数组（如 `[20, 0, -30...]`） | 三维坐标与四元数（如 `pos: [0.3, 0.2, 0], quat: [...]`） |
| **轨迹特征** | **弧线**（流畅、圆滑、不可预料的物理轨迹） | **直线或精确几何图形**（指哪打哪） |
| **依赖算法** | 贝塞尔曲线、三次样条插值（用于角度平滑过渡） | 逆运动学 (IK)、雅可比矩阵、微积分轨迹切分 |
| **适用场景** | 极具张力的舞蹈（波浪舞、扩胸）、基础姿态展示 | 精确抓取、机械舞（身体动但手悬停）、动捕映射 |
| **控制范围** | 易于全身统筹（包含手臂、灵巧手、头部） | 主要针对手臂的宏观空间定位 |
| **计算消耗** | 极低（纯角度下发） | 极高（需高频实时求解非线性矩阵方程） |

---

## 二、 逆运动学 (IK) 的两次底层技术跃迁

如果采用“笛卡尔空间编舞”，如何把空间坐标 $(X, Y, Z)$ 翻译成各关节角度 $\theta$？业界经历了从“解析法”到“微分法”的根本性跃迁。

现代高级机器人（如 Kuavo、特斯拉擎天柱、波士顿动力）已经全面抛弃了传统的“暴力算角度”，转向了基于雅可比矩阵的“微积分算速度”流派。

| 对比维度 | 阶段 1：位置级逆运动学 (Analytical IK) | 阶段 2：速度级逆运动学 (Differential IK) |
| :--- | :--- | :--- |
| **核心思想** | **解析法**：利用三角函数和几何学，强行写出从坐标求角度的绝对代数方程。 | **微分法 (数值迭代)**：放弃直接算位置，利用微积分思想，将位置误差转化为期望速度，步步逼近。 |
| **求解目标** | 绝对关节角度 ($\theta$) | 瞬时关节角速度 ($\omega$) |
| **数学方程** | 非线性方程组（极其复杂，包含大量正余弦交叠） | 线性方程组 $V = J \cdot \omega$ （利用雅可比矩阵降维打击） |
| **自由度适应性** | 仅适用于 **6自由度及以下** 的简单机械臂。 | 完美适配 **7自由度以上** 的冗余机械臂，自由度再多也能统一用矩阵扩展。 |
| **多解与奇异点处理** | **灾难级**：冗余机械臂有无数个解，算法无法挑选最优解，导致动作“抽搐”；遇到奇异点会因“除以零”导致程序崩溃。 | **极其优雅**：利用矩阵伪逆 ($J^{\dagger}$) 和零空间投影技术，可以从容挑选能耗最小的解，并完美绕开奇异点。 |
| **附加约束能力** | **无**：算出来是几度就是几度，根本不管会不会撞到自己。 | **极强**：由于是迭代优化过程，可以轻松加入“关节限位保护”、“自我防碰撞”、“能耗最低”等约束条件。 |
| **真实运算过程** | **一步到位**：代入公式，一次性算出一个终点角度。如果轨迹跨度大，中间路径完全不可控。 | **高频切分**：<br>1. 算误差：目标坐标 - 当前坐标 = 误差向量<br>2. 化速度：将误差当成短时间内的期望速度 $V$<br>3. 雅可比翻译：$\omega = J^{\dagger} \cdot V$<br>4. 微动：电机按 $\omega$ 转动 1 毫秒<br>5. 循环：高频重复上述步骤逼近目标。 |
| **工业界现状** | 已淘汰于复杂人形机器人，仅在传统低端工业机械臂中保留。 | **绝对的工业标准**（你使用的 Kuavo 仿真底层 SNOPT 优化器就是采用此架构）。 |

## 三、 案例一：手臂轨迹规划实战 (样条法与贝塞尔)

此案例对应直接发送关节角度的“关节空间控制”测试。

### 💻 标准启动终端指令 (Ubuntu 原生环境)

**Step 1: 彻底清理并编译 (极度重要)**
```bash
cd ~/kuavo_ws
rm -rf build/ devel/ logs/ .catkin_workspace
source /opt/ros/noetic/setup.zsh
catkin config -DCMAKE_BUILD_TYPE=Release -DCMAKE_CXX_STANDARD=17
catkin build humanoid_controllers gazebo_sim
```

**Step 2: 物理替换地图 (绕过底层强制加载代码)**
```bash
cd ~/kuavo_ws/src/gazebo/gazebo-sim/worlds/
mv grab_box.world grab_box.world.bak
cp maze_test.world grab_box.world
```

**Step 3: 声明模型路径、清缓存并启动仿真**
```bash
cd ~/kuavo_ws
export GAZEBO_MODEL_PATH=$GAZEBO_MODEL_PATH:~/kuavo_ws/src/gazebo/gazebo-sim/models
sudo rm -rf /var/ocs2/kuavo_v42
source devel/setup.zsh
roslaunch humanoid_controllers load_kuavo_gazebo_sim.launch record_bag:=false
```

**Step 4: 启动规划节点与测试脚本**
```bash
# 新开终端 1：启动后台轨迹规划服务
source devel/setup.zsh
roslaunch humanoid_plan_arm_trajectory humanoid_plan_arm_trajectory.launch robot_version:=42

# 新开终端 2：运行 Python 脚本
source devel/setup.zsh
python3 src/demo/examples_code/hand_plan_arm_trajectory/plan_arm_traj_cubicspline_demo.py
```

### 🚨 踩坑与排错指南
*   **坑位 1：原生环境与 Docker 缓存冲突**
    *   **现象**：各种莫名其妙的 `C++` 编译报错或链接失败。
    *   **解决**：必须执行 `rm -rf build/ devel/` 进行物理超度，使用宿主机的 GCC 重新编译。
*   **坑位 2：服务端报错 `Expected 0, got 14` 且一直提示 `Joint not found`**
    *   **现象**：Python 脚本抛出 `ServiceException` 报错。
    *   **原因**：规划节点（服务端）没读到机器人的 URDF 模型，错以为手臂有 0 个关节。
    *   **解决**：启动 launch 文件时必须强制传入版本号参数（如 `robot_version:=42`），且**必须先启动仿真器，后启动规划节点**，保证参数服务器上已有机器人描述文件。
*   **坑位 3：Python 脚本返回空列表 `[]` 导致崩溃**
    *   **原因**：机器人还没在 Gazebo 里站稳，或者设置的初始目标点太离谱，导致底层运动学解算无法获取有效的初始位姿。

## 四、 案例二：手臂正逆运动学实战 (MuJoCo 联合调试)

此案例对应输入 XYZ 坐标的“笛卡尔空间控制”测试。

### 💻 标准启动终端指令

**Step 1: 启动 MuJoCo 仿真器 (作为底层物理执行器)**
```bash
source devel/setup.zsh
roslaunch humanoid_controllers load_kuavo_mujoco_sim.launch
```

**Step 2: 启动 IK 逆运动学解算节点 (关闭 RViz 防止报错)**
```bash
source devel/setup.zsh
roslaunch motion_capture_ik ik_node.launch robot_version:=42 visualize:=false print_ik_info:=true
```

**Step 3: 夺取底层控制权 (极其关键)**
```bash
source devel/setup.zsh
rosservice call /humanoid_change_arm_ctrl_mode "{control_mode: 2}"
```

**Step 4: 运行 IK 测试脚本下发坐标**
```bash
source devel/setup.zsh
rosrun motion_capture_ik example_ik.py
```

### 🚨 踩坑与排错指南

*   **坑位 1：RViz 画面截肢，终端狂刷 `Could not load resource`**
    *   **现象**：RViz 里的机器人隐形，只剩几只红绿颜色的手飘在半空，终端提示找不到 `../../meshes/torso.STL`。
    *   **原因**：原代码 URDF 中使用了相对路径 `../../`。在原生环境下工作目录改变，导致找不到 3D 模型文件。
    *   **解决**：最快的方法是在 Step 2 中加上 `visualize:=false` 直接关掉 RViz。彻底治本的方法是去 URDF 文件里把相对路径全部替换为标准的 `package://kuavo_assets/meshes/`。
*   **坑位 2：IK 计算成功，但机器人死活不动**
    *   **现象**：MuJoCo 终端弹出黄字警告 `current armControlMode 1 != EXTERN_CONTROL`。
    *   **原因**：机器人的底层处于安全锁定或 VR 随动模式，拦截了你的外部代码指令。
    *   **解决**：必须执行 Step 3 的指令，调用 service 强行将手臂控制权切换为 2（外部控制模式）。
*   **坑位 3：IK 节点报错 `Left hand out of workspace!`**
    *   **原因**：你在 Python 测试脚本里写的 XYZ 坐标，超出了机器人手臂的物理极限（比如把它伸到了身体后面或远远超出臂长）。
    *   **解决**：修改脚本中的目标点，设为胸前的合理坐标（例如 `pos_xyz: [0.3, 0.2, 0.0]`）。
*   **盲点避坑：为什么仿真里灵巧手毫无反应？**
    *   **原因**：利用 `rostopic info /control_robot_hand_position` 会发现该话题毫无订阅者 (`Subscribers: None`)。为了防止物理引擎处理复杂的小说部件碰撞导致算力崩溃，Gazebo/MuJoCo 仿真中专门移除了灵巧手的活动关节。灵巧手在仿真里本质是个静态的“摆设”，测试手指动作需直接上实车硬件调试。
