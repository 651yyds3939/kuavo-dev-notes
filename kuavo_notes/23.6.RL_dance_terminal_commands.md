## 第 0 章：前情提要与四段式路线图

在人形机器人强化学习的真实上机和仿真部署调试中，**在复制粘贴任何一行终端命令前，必须彻底理解其背后的力学与软件假设（WHY）**——否则你将陷入严重的恶性死锁怪圈：Kit 引擎无故挂起、显存爆掉 CUDA OOM、或者模型导出后在 MuJoCo 仿真里一接管就摔碎。届时你只能盲目试错。

本篇指南按**时间先后顺序**将命令分为四段：**训练 → 导出 ONNX → Sim 编译链 → MuJoCo 仿真 → 真机**；同一条命令全文只出现一次，跨段引用用章节号跳转。

* **最高版本对齐红线**：训练、MuJoCo、真机三端 **`ROBOT_VERSION` 必须与所选链路一致**——**S42/S46 链路 A 用 `46`**，**S49 链路 B 用 `49`**；混用会导致 `kuavo_assets` / 观测维 / ONNX 文件名全面错位。
* **显存硬件并行基准（8GB 笔记本实测约 `6.8GB`）**：S42/S46 历史链 **`--num_envs=8192`**；S49 DIY 推荐链 **`--num_envs=4096`**；若 CUDA OOM 再逐级下调。

* **系列分卷**：本篇为 **23.6（23.x 最后一卷）**——只收录 **不随一次调试而变** 的稳定终端命令。Hybrid 扫参、按 R 即倒排障、bag 解读见 **23.5**；不要在本文堆砌 `sim_wbc_rl_action_blend:=0.75` 等调试期 launch 后缀（黄金值已写入 `load_kuavo_mujoco_sim_dance_s49.launch` 源码 default）。

* **S49 比 S42/S46 更吃算力、更慢的根因（读命令前必知）**：S42/S46 用 **`biped_s46.usd` 预烘焙** + 较轻观测/奖励，可 **8192 envs、~43000 steps/s、~6.8GB**；S49 用 **lite URDF 实时转 USD** + **115 维观测** + **1565 帧 GPU 轨迹** + **upright 门控 mimic**，单 env 成本约 **2×**，故只能 **4096 envs** 换同档 **~6.8GB**，且 **~16000 steps/s**。详见 **23.1** 1.1.1节。

以下命令只需在新下位机工作站首次部署、或者你的机器人更换了全新的原始 URDF 资产结构后执行一次。**严禁直接跳过本章**，否则系统在开局阶段要么报 import 崩溃，要么因 48 关节未裁剪引发 Action/Obs 维度大爆炸。

### 1.1 Isaac 训练环境注入（每次开训 / 开 play 前重复）

在进行 headless 无界面训练或导出前，必须用断头铡式强制清理后台 Kit 僵尸进程、拔除 X11 显示锁、并全量注入路径环境变量。

#### 为什么必须执行 `pkill kit`？

如果上一次训练因为意外断流或者中途报废退出了，Ubuntu 后台会非常顽固地残留下一堆由 Omniverse 派生出的 `kit` 僵尸守护进程。这些进程会死死霸占着 GPU 显存池不释放，并对底层的物理缓存锁（Cache Lock）文件进行死锁——这会导致你开启新一炉训练时，程序永远卡死在开局的 Splash Screen 黑屏加载线处弹不出 Progress 图表。

#### 为什么必须在 Headless 前执行 `unset DISPLAY`？

在 Headless 盲跑模式下，如果系统的 `DISPLAY` 环境变量依然顽固指向宿主机的 X11 转发端口，Kit 引擎会因为强行去进行 GLFW 图形视窗初始化而发生冲突挂起死锁。请保持绝对冷静：**在 Headless 模式下，控制台日志滚动弹出的 `GLFW initialization failed` 警告，完全是剥离图形管线时的正常力学排异日志，不是报错，直接忽略即可。**

#### 为什么更换 URDF 后必须清空 `kit/cache`？

Kit 内部挂载了极其顽固的 USD 资源记忆缓存。如果你修改或精简了机器人的 URDF 骨骼，但没有去清空临时缓存，PhysX 求解器在开局第 1 帧依然会固执地去内存里加载上一轮旧模型的 48 关节错误资产，直接导致 Action 空间在训练开局两秒内崩溃解体。

```bash
# 1. 物理级强杀残留僵尸进程，砸碎 Kit 缓存锁
pkill -9 -f kit
pkill -9 -f python
rm -rf ~/.local/share/ov/pkg/isaac-sim-4.2.0/kit/cache/*

# 2. 激活 Conda 隔离环境，彻底拔除 DISPLAY 图形显示锁
conda activate isaaclab
cd ~/kuavo_all/leju_robot_rl
unset DISPLAY

# 3. 注入英伟达原厂硬核环境变量总线（WANDB强制离线）
source ~/.local/share/ov/pkg/isaac-sim-4.2.0/setup_python_env.sh
export EXP_PATH=$HOME/IsaacLab/source/apps/isaaclab.python.headless.kit
export CARB_APP_PATH=$HOME/.local/share/ov/pkg/isaac-sim-4.2.0/kit
export ISAAC_PATH=$HOME/.local/share/ov/pkg/isaac-sim-4.2.0
export VK_ICD_FILENAMES=/usr/share/vulkan/icd.d/nvidia_icd.json
export WANDB_MODE=offline
```

* **健康运行期望判据**：后续执行 `train.py` 时，系统没有任何 import 语法报废红字，控制台前 5 行顺利滚动出真实的 Isaac Sim 官方标称版本号。
* **排卡 Troubleshooting 门哨**：如果提示 `which python` 依旧指向系统原生路径，说明你刚才粘贴命令时发生了 Conda 环境变量被 Zsh 插件截断的意外，请重新打点执行一遍 `conda activate isaaclab`。

#### Zsh `zoxide` 钩子与多行粘贴截断（chat 高频踩坑）

> 若终端反复弹出 **`__zoxide_hook:1: command not found: zoxide`**，在 Zsh 里**多行粘贴** `export` / `source` 时，钩子报错会在行间截断后续路径，Python 启动 Isaac 时变成「找不到 Kit 动态库」的睁眼瞎——这与训练代码本身无关。

* **根治 A（推荐）**：训练点火前 **`bash`** 切到纯净 Bash，再执行 1.1 与 2 的单行 `&&` 串联命令（Bash 无 zoxide 钩子）。
* **根治 B**：注释掉 `~/.zshrc` 里的 zoxide 初始化（Ubuntu 20.04 上 `apt install zoxide` 常报「无法定位软件包」，可跳过安装直接注释）：
  ```bash
  sed -i 's/.*zoxide init.*/# &/' ~/.zshrc && source ~/.zshrc
  ```
* **纪律**：Isaac 训练环境注入**优先用单行 `&&` 串联**，避免在 Zsh 里分多行粘贴；ROS / Docker 编译同理。

```bash

  --load_run ${RUN_DATE} \
  --checkpoint ${CHECKPOINT} \
  --num_envs 256 \
  --headless \
  --video \
  --video_length 400

# 录完立刻归档，避免下一次 play --video 覆盖 rl-video-step-0.mp4
mkdir -p ${VIDEO_DIR}/${CHECKPOINT%.pt}
mv ${VIDEO_DIR}/rl-video-step-0.mp4 ${VIDEO_DIR}/${CHECKPOINT%.pt}/rl-video-step-0.mp4
mv ${VIDEO_DIR}/rl-video-step-0.meta.json ${VIDEO_DIR}/${CHECKPOINT%.pt}/ 2>/dev/null || true
```

**批量对比多个 checkpoint（推荐，录完自动归档）**：

#### 拷贝 ONNX 与舞蹈 CSV 到部署仓

S49 与 S42/S46 一样，部署源文件是 **`exported/policy_s42.onnx`**（见本章开头「导出三件套」）。拷进 `model/networks/` 时**默认仍叫 `policy_s42.onnx`**，也可按 **3.2.1节** 改成任意名字（须在 `skw_rl_param_dance.info` 的 **`networkModelFile`** 里写同名）：

* **技术断代警示**：**禁止**把 **旧 89 维走路权重** 导出的 ONNX 或 **`policy.onnx`（无重排外壳）** 当 S49 115 维舞蹈策略用；须用 **当前 S49 run 新导出的 `policy_s42.onnx`**（115 维输入）。维数/观测语义不对会在 MuJoCo 接管瞬间挺尸或 map 越界。

#### 3.2.1 自定义 ONNX 文件名（cp 时爱叫什么叫什么，参数卡里对齐即可）

> **就两步**：① `cp exported/policy_s42.onnx` 到 `model/networks/` 时**随便取新文件名**；② 在 **`skw_rl_param_dance.info`** 顶部把 **`networkModelFile`** 改成同名。**不用**改 launch、不用改 `play.py`、不用改 C++。

`play.py` 在 run 里固定生成 **`exported/policy_s42.onnx`**（C++ 可加载版）；**`policy.onnx` 仅供调试，不要直接部署**。拷进 `model/networks/` 时文件名可以随便起，例如 `49dance_3000.onnx`、`49duck_sit.onnx`，不必仍叫 `policy_s42.onnx`。

* **目录（launch 默认）**：`humanoid_controllers/model/networks/`
* **文件名（参数卡指定）**：`skw_rl_param_dance.info` 顶部的 **`networkModelFile`**

C++ 侧逻辑：`networkModelPath_ = <networks 目录> + <networkModelFile 字符串>`（见 `humanoidController.cpp`）。因此**只要 cp 的源是 `policy_s42.onnx`、目标名与 `networkModelFile` 一致**，就能加载，无需改 launch、无需改 `play.py`。

**示例：导出 iter 3000 权重并命名为 `49dance_3000.onnx`**

```bash
export RUN_DATE=2026-06-20_04-17-57
export CHECKPOINT=model_3000.pt
export ONNX_NAME=49dance_3000.onnx    # 部署仓里的文件名，随意

# 1. headless 导出（run 内生成 policy_s42.onnx）
python3 scripts/rsl_rl/play.py \
  --task Legged-Isaac-Velocity-Flat-Kuavo-S49-Play-v0 \
  --num_envs 32 --headless \
  --load_run ${RUN_DATE} \
  --checkpoint ${CHECKPOINT}

# 2. 务必 cp policy_s42.onnx，不是 policy.onnx
cp logs/rsl_rl/Kuavo/s49/dance/${RUN_DATE}/exported/policy_s42.onnx \
   ~/kuavo_all/kuavo-rl-opensource/kuavo-robot-deploy/src/humanoid-control/humanoid_controllers/model/networks/${ONNX_NAME}
```

**3. 只改参数卡一行（与 cp 目标同名）**

打开 `config/kuavo_v49/rl/skw_rl_param_dance.info`，修改顶部：

```text
networkModelFile          /49dance_3000.onnx;
```

* 行首 **`/`** 是历史写法，表示相对 `model/networks/` 的文件名，**不是** Linux 根目录。
* 多版 checkpoint 可并存：`policy_s42.onnx`、`49dance_3000.onnx` 等，靠 **`networkModelFile`** 切换。
* **验收**：`roslaunch` 后日志应打印 `networkModelPath_.../model/networks/49dance_3000.onnx`；若仍指向旧名，说明只 cp 了文件没改 **`skw_rl_param_dance.info`**，或 launch 读的不是 dance 参数卡（见 **7.1节 F9**）。
* **勿改**：ONNX 输入维（115）、`numSingleObs`、`singleInputData` 键序与 CSV/`cycleTime` 仍须与训练一致；**改名只换文件，不换网络语义**。

### 两链路共用

### 3.3 Checkpoint 选取（共用）

S42/S46 与 S49 链均适用：**勿机械盲选最大 iter**。

S42 历史链实战表明：策略常在 **iter 2500–3000** 达到 mimic + 平衡双巅峰；**model_6000.pt** 虽下肢极稳，但靠近后期 **inf/NaN 梯度爆炸** 前，参数易饱和高频抽搐；**model_3000.pt** 往往是更健康的导出候选。S49 链同理——**勿机械套用 `model_9000.pt`**，应：

1. TensorBoard 找 **`flat_orientation_l2 → 0`** 且 **`base_contact → 0`** 仍维持 **`track_punch_*` 高位** 的 iter 段；
2. 对 **2500 / 3000 / 6000 / 9000** 各跑一轮 **2.8节** `play.py --video` 肉眼比动作；
3. 若 mimic 高但 play 挺尸/抽搐，优先 **更早 checkpoint** 或 **iter 0 重训**（见 **23.3** 7.3节）。

> **过拟合信号**：跑满 10000 iter 后 reward 断崖、entropy 归零、mimic 独高而 play 僵硬——早停即可，磁盘里每隔 500–1000 iter 的 `model_*.pt` 切片就是保险库。

**报错 A — 白名单不认 49**（`update_mass.sh` 阶段即失败）：
