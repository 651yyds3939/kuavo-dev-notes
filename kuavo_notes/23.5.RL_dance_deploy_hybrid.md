## 第一章：duck_sit 首秀 — Isaac 跪跳很美，MuJoCo 为何完全不是一回事

### 1.1 训练 run 与策略文件

| 项目 | 值 |
| --- | --- |
| 训练 run | `2026-06-20_04-17-57` |
| 导出 ONNX | `play.py` 默认 `policy_s42.onnx` → 部署重命名为 **`49duck_sit.onnx`** |
| RL 参数 | `kuavo_v49/rl/skw_rl_param_dance.info` |
| 舞蹈 CSV | `kuavo_action_S49_FROM_S54_INPLACE_RAD.csv`（1565 帧，cycleTime=31.30） |
| Launch | `load_kuavo_mujoco_sim_dance_s49.launch` |
| 训练奖励 v1 全表 | **附录 L.1**（v2 防跪重训见 **L.2**） |

用 `play.py --video` 看 `model_9999` 或相近 checkpoint 时，**所有 env 里的机器人从一开始就没「站稳过」**：腿颤几下 → 突然跪下/鸭子坐 → 坐着挥臂跳舞。歪打正着，观感还挺有趣。

### 1.2 部署到 MuJoCo 后的三种表现（实测归纳）

| 模式 | 现象 | 结论 |
| --- | --- | --- |
| **纯 WBC 站立**（未按 R） | 可一直站稳 | 经典控制没问题 |
| **纯 RL**（`sim_keep_wbc_with_rl:=false` 或 `pure_rl.launch`） | 按 R 后**直挺挺往前倒**，倒地后抽搐 | 关节 PD 扛不住 |
| **Hybrid 默认**（`sim_keep_wbc_with_rl:=true`） | 能跳，但早期幅度小、动作慢；调参后可长时间不摔 | **当前唯一可交付路径** |

用户最初期望：MuJoCo 里也应该「站着颤腿 → 跪下 → 坐着跳」，和 Isaac 视频一样。

**工程真相**：Isaac 里是 **纯关节 PD + 训练物理**；MuJoCo Hybrid 里是 **WBC 强制把机身锁在站立附近**，policy 只贡献部分关节参考。所以部署侧「站着慢慢跳」**不是 bug，是架构差异的必然结果**。

### 1.3 启动即低头 — 头部无力（未切 RL 时）

S49 MuJoCo 一打开，头就低着，像没力。根因与 **23.4** 中 S46 不同：

* S46 `scene_rl` → `biped_s46_collision.xml`，头关节映射简单；
* S49 `biped_s49.xml` 含手指，`qpos` 55 维，头在 RL 26-DoF **之后** 追加；
* 未切 RL 时 WBC 对头关节力矩弱或未锁期望位姿。

后续在 `humanoidController` 增加 **`[sim] Head hold: lock desire to current yaw/pitch`** 日志路径，spawn 后把头期望锁在当前测量值。这与「46 版默认锁住头」的体验对齐，但实现机制不同。

### 1.4 按 R 前 WBC 站稳，按 R 后立刻倒 — 用户反复确认的事实

多轮测试中一致：

1. 按 R **之前**：MPC+WBC 站稳，gravity_body ≈ `[0.07, 0.01, -0.997]`（pitch≈0.05 训练站姿）；
2. 按 R **之后**：若走纯 RL 力矩切换，机身开始**缓慢或瞬间前倾**；
3. 倒下后 policy 仍在推理 → 地上**抽搐**（腿张开、高增益 PD 打地）。

这说明：**不是「没接上 RL」**，而是 **接上的瞬间控制栈从 WBC 平衡切到了站不住的 PD 轨迹**。

## 第二章：按 R 即倒 — 三十余轮调试血泪时间线


### 1.5 Isaac `--video` 与 MuJoCo 的「跪 vs 站」对照

用户 run `2026-06-20_04-17-57` 的 play 视频特征：

1. spawn 后**从不稳定站立**开始；
2. 双腿高频**颤抖**；
3. 约 1～2s 内**跪/坐**；
4. 上身按 CSV 跳舞。

同一 ONNX 在 MuJoCo Hybrid 下：

1. WBC spawn **站立**（z=0.85, pitch=0.05）；
2. 按 R 后 **保持站立附近**；
3. 上身/腿按 blend 参考动。

**这不是部署失败**，是 **WBC 作为外骨骼** 改写了下身工况。若目标是「仿真里也要跪」，需要关掉 WBC 并解决第六章的纯 RL 物理问题。

### 1.6 `49duck_sit.onnx` 命名与 info 配置

用户操作：

1. `play.py` 导出默认名 **`policy_s42.onnx`**（42/46/49 训练皆同，见 23.6 §3.2.1）；
2. 拷贝到 `humanoid_controllers/model/networks/` 并**随意重命名**，如 `49duck_sit.onnx`；
3. 在 `skw_rl_param_dance.info` 修改 `networkModelFile /49duck_sit.onnx;` 即可。

**不要**以为文件名必须含 s49；**只要 info 与磁盘文件名一致**。

### 2.1 早期：以为是神经网络或 obs 又坏了

| 阶段 | 症状 | bag 线索 |
| --- | --- | --- |
| 第 1～5 轮 | 按 R 瞬间倒 / 顿 1s 后倒 | obs 115 维，gravity 正常，action 很大 |

### 2.2 关键 bag 读数（`analyze_r_takeover_bag.py`）

典型 **按 R 后前倒** 的 bag（obs≈200）：

```
gravity_body   : [0.065, 0.007, -0.998]   ← 接管瞬间仍直立
jointPos max   : ~0.07 rad                 ← 相对默认姿偏移很小
action max     : 3.0～8.0（clip 后饱和）   ← policy 在出力
commandPhase   : sin/cos 在涨             ← 相位在走
```

**解读**：obs 链路基本正常；倒下的主因是 **力矩/参考切换**，不是「网络没收到输入」。

### 2.3 中间插曲：浮空、闪退、蹲不下去

| 现象 | 原因 | 处理 |
| --- | --- | --- |
| 机器人一直浮空 | `nodelet.launch` 用环境变量 `ROBOT_VERSION`，未 export 49 时加载 **S42 场景** | **必须** `export ROBOT_VERSION=49`，日志见 `biped_s49/xml/scene_rl.xml`、`totalMass: 58.3953`、`qpos 55` |
| MuJoCo 闪退 | 场景/模型不匹配或 init qpos 长度错 | 对齐 S49 scene + `buildMujocoTrainingStandQpos` |
| 跪着跳不出来 | Hybrid 下 WBC 不让大幅下蹲 | 预期行为；要跪跳需纯 RL + 物理对齐 |

### 2.4 第一次「能跳」：按 R 不动 → 慢跳 → 久跳

| 轮次 | 现象 | 意义 |
| --- | --- | --- |
| 某轮 | 按 R **完全不动** | RL 未真正输出力矩 / 被 blend 压死 |
| 下一轮 | **开始跳**，幅度大、慢动作、后半摔 | Hybrid 生效但参数激进 |
| 再调 | **连续跳很久不摔**（bag obs≈1253） | **Sim2Sim 首次达标** |

用户四个追问（当时原文）：

1. 腿/臂幅度小、慢 — 正常吗？
2. 头没力 — 怎么办？
3. Isaac 跪着跳，MuJoCo 站着跳 — 为什么？
4. launch 刷屏 ERROR — 有问题吗？

**答复要点**：

1. **正常**：blend&lt;1、offset 限制、filter 导致「慢且小」；
2. 头：spawn hold + 后续可接头关节力矩；
3. 架构差异（见 1.2）；
4. `Maximum number of working set recalculations`：WBC QP 吃力，**偶发可接受**，频繁则需降 blend/增 tau。

### 2.6 三十轮失败的「症状分类学」

我们把所有失败 bag 按 **倒下方向** 与 **时序** 分成四类，便于以后对号入座：

#### 2.6.1 A 类：瞬间前倒（最常见）

* **体感**：按 R 后 0.2～0.5s 内身体像被抽掉髋关节，**直挺挺面向前**倒下；
* **bag**：obs[0] 的 gravity_body 仍正常（0.06, 0.01, -0.997 量级），说明**倒下的第一帧 obs 仍像站着**；
* **action**：第一帧即有大动作（±3～8），clip 后手臂饱和；
* **结论**：WBC→RL 力矩切换瞬间，**前向力矩矩不平衡**。

#### 2.6.2 B 类：缓慢前倒（「慢动作倒」）

* **体感**：按 R 后 1～3s 身体像灌铅一样**慢慢前倾**，倾倒过程中已在录 bag；
* **用户原话**：「是不是倒下的时候还没接入 RL？」—— 实际上 bag 显示 **RL 已接入**，是 **接入后参考/力矩让重心前移**；
* **结论**：常见于 blend 偏高或 filter tau 过小、WBC 跟不上。

#### 2.6.3 C 类：侧后倒（纯 RL 路径）

* `sim_keep_wbc_with_rl:=false` 时偶发 **侧后方**倒，像「下腰但斜一点」；
* 与前倒同属 **纯 PD 无法持衡**，只是初始姿态微差导致倒向不同。

#### 2.6.4 D 类：倒地后抽搐

* **共同特征**：人已趴地，腿**张开高频抖**；
* **原因**：policy 仍在 50Hz 推理，大地接触下 PD 增益过高 → 不是「网络发疯」，是 **错误工况下仍在执行跟踪**。

### 2.8 编译与 Docker 映射（反复踩坑）

用户强调：「mujoco 没包，不需要编译」—— **对**，但 **`humanoid_controllers` 必须 catkin build** 才生效 C++ 改动。

Docker 内 `~/kuavo_ws` 映射到宿主机 `kuavo-robot-deploy` 时：

1. 宿主机改 `humanoidController.cpp`；
2. 容器内 `catkin build humanoid_controllers`；
3. `source devel/setup.zsh`；
4. **再** launch。

只改 launch/info **不编 controller** = 「改了等于没改」的高频来源。


## 第三章：根因分层 — 不是单一 bug，而是三层叠加

### 3.1 第一层：Deploy Bug（已修）

| Bug | 现象 | 修复 |
| --- | --- | --- |
| **ROBOT_VERSION 未 export** | `totalMass: 55`、`qpos 35`、S42 场景 | 启动前 `export ROBOT_VERSION=49` |
| **S49 手指 qpos 错位** | RL obs 把手指当手臂 | `mujoco_node.cc` → `initRlJointSensorMap()` |
| **Spawn 高度/俯仰** | 与训练 default 不符 | `buildMujocoTrainingStandQpos`：z=0.85, pitch=0.05 |
| **launch `robot_version` 与 env 不一致** | 参数写 49 仍加载 42 | `nodelet.launch` 读 `optenv ROBOT_VERSION` |

### 3.2 第二层：控制架构（根本限制）

```
纯 WBC：     站稳 ✅  跳舞 ❌
纯 RL PD：   按 R 倒 ❌（action=0 也倒）
Hybrid：     站稳 + 跳舞 ✅（同一 ONNX）
```

**`sim_rl_debug_zero_action:=true`** 实验：policy 强制输出 0，仍前倒 → **证明不是神经网络单独的问题**，是 **MuJoCo 上纯关节 PD 无法维持训练站姿**。

### 3.3 第三层：物理 / 任务 Sim2Sim 未闭环

| 对比项 | 训练（Isaac） | 部署（MuJoCo） |

## 第四章：突破性转折 — Hybrid WBC+RL 路径跑通

### 4.1 参数 `sim_keep_wbc_with_rl` 的含义

在 `humanoidController.cpp` 中：

* **`true`（Hybrid）**：`is_rl_controller_` 后 **WBC 继续算力矩**；`optimizedState_mrt_` 关节参考由 RL **混合**进 WBC；
* **`false`（纯 RL）**：WBC 力矩逐步交出，MuJoCo 走 `updateRLcmd` native PD；
* **官方 `kuavo-ros-opensource` 无此参数** — 为本仓库 S49 duck_sit 调试期新增。

日志关键字：

```
[RL mode] RL activated: WBC tracks RL targets (blend=0.75, filter=10ms) — full-speed policy, no ramp
```

### 4.2 Hybrid 关节参考公式（实现级）

对每个 RL 关节 \(i\)：

```text
rl_joint   = action[i] * actionScale + defaultJointPos[i]
stand_joint = initial_status_[12+i]    # WBC 站立模板
delta      = clamp(rl_joint - stand_joint, ±max_offset)
blended    = stand_joint + active_blend * delta
filtered  += alpha * (blended - filtered)   # alpha = dt / tau
optimizedState_mrt_[12+i] = filtered
```

Launch 参数：

| 参数 | 含义 |
| --- | --- |
| `sim_wbc_rl_action_blend` | 稳态 blend（0～1） |
| `sim_wbc_rl_blend_start` | 接管初始 blend |
| `sim_wbc_rl_blend_ramp_sec` | 从 start → 稳态 的爬升时间 |
| `sim_wbc_max_rl_joint_offset_rad` | 单关节相对站立的最大偏移 |
| `sim_wbc_ref_filter_tau` | 低通滤波时间常数（秒） |

### 4.3 代码与文件改动清单（本轮）

| 文件 | 改动摘要 |
| --- | --- |
| `humanoidController.cpp` | Hybrid blend、spawn settle、`sim_rl_debug_zero_action`、head hold、training stand qpos |
| `humanoidController.h` | `sim_keep_wbc_with_rl_`、blend 相关成员 |
| `mujoco_node.cc` | `initRlJointSensorMap()`、native PD 路径 |
| `load_kuavo_mujoco_sim.launch` | 传播 hybrid 参数 |
| `load_kuavo_mujoco_sim_dance_s49.launch` | S49 舞蹈专用 + **黄金默认** |
| `load_kuavo_mujoco_sim_dance_s49_pure_rl.launch` | 纯 RL 实验（预期会倒） |
| `skw_rl_param_dance.info` | `clipActions=8.0`（WBC 路径） |

### 4.4 Hybrid 与纯 RL 在 `humanoidController` 里的分叉（读代码）

当 `update()` 主循环执行时，关键分支：

**分支 A — `use_wbc_torque == true`（Hybrid）**

```text
条件：!is_rl_controller_
   OR rl_wbc_torque_handoff_
   OR (!is_real_ && sim_keep_wbc_with_rl_ && is_rl_controller_)
```

即：**仿真 + RL + sim_keep_wbc_with_rl** 时 **永远走 WBC 力矩**，不会把平衡完全交给 `updateRLcmd`。

**分支 B — 纯 RL**

`sim_keep_wbc_with_rl=false` 且 handoff 结束后：`actuation = updateRLcmd()`，MuJoCo `mujoco_node` 若 `joint_kp[0]>1` 走 **native PD**：

```text
ctrl = ff + kp*(q_des-q) + kd*(v_des-v)
```

**分支 C — `sim_rl_debug_zero_action`**

`inference()` 内强制 `actions_=0`，用于证伪网络。

### 4.5 `rl_from_start` 与 spawn settle

`load_kuavo_mujoco_sim_dance_s49.launch` 默认 `rl_from_start=false`（需按 R）。

若 `rl_from_start=true`（`pure_rl.launch` 默认）：

* 先 **2.5s WBC hold spawn**（`kRlSpawnSettleSteps=2500`）；
* 再切 RL inference；
* 纯 RL 路径下仍 **站不住**。

舞蹈验收推荐：**manual R**，站稳后接管。

### 4.6 `Maximum number of working set recalculations` 技术释义

来自 WBC 二次规划（QP）求解器：接触力/摩擦锥约束在参考突变时 **working set 反复变更**，达上限后打印 ERROR。

**与摔倒关系**：

* 偶发 ERROR、机身仍稳 → 可忽略；
* ERROR 风暴 + 机身发散 → **blend 过高或 tau 过小**，WBC 追 RL 追太猛。

处理顺序：**降 blend** → **增 tau** → **增 ramp_sec** → 最后才动 Kp。

### 5.7 四个「成功后的小问题」详解

#### 5.7.1 腿/臂幅度小、慢 — 正常吗？

**正常，且符合当前架构。**

* `blend=0.75`：WBC 只跟 75% 的 RL 偏移；
* `max_offset=0.5`：单关节偏离站立模板不超过 0.5 rad；
* `tau=0.01`：10ms 低通 → 动作被抹平；
* `clipActions=8`：Isaac 可用 100，这里砍到 8。

要更大：先 **略增 blend**（0.78 试探），**不要一步 0.85+**。

#### 5.7.2 头部无力

* **未切 RL**：WBC 对头力矩弱；已加 spawn head hold；
* **切 RL 后**：26-DoF policy 不含头；头在 `head_joint_map` 用 WBC/被动；
* S46 看起来「默认锁住」是因为 **模型简单 + 旧版映射**，不是 magic flag。

#### 5.7.3 Isaac 跪跳 vs MuJoCo 站跳

| 环境 | 平衡谁管 | 下身行为 |
| --- | --- | --- |
| Isaac play | 仿真 PD，policy 学跪姿 | 颤腿→跪→跳 |
| MuJoCo Hybrid | **WBC 管** | 站立附近跳 |

要跪跳：必须 **纯 RL + 物理对齐**，并接受与 Hybrid 不同风险。

#### 5.7.4 launch ERROR 刷屏

`Maximum number of working set recalculations performed` = WBC QP 接触力/加速度约束在 blend 高、参考跳变大时 **求解器迭代不够**。

* **偶发（<5 次/分钟）**：可继续验收；
* **持续刷屏 + 摔**：降 blend、增 tau、增 ramp。


### 5.4 为什么要固化、能不能纯 RL

**要固化** — 边界已摸清，这是 duck_sit 在 MuJoCo 的 **正确 deliverable**。

**纯 RL 不是「永远不行」**，而是 **当前 task + 未对齐物理** 下不现实。要纯 RL 需另开 Sim2Sim 对齐项目（见第九章），不与本次交付绑死。

## 附录 G：从 chat / RL_issue 并入的本轮工程决策

1. **统一 S49 链路**（训练/MuJoCo/真机皆 49）不变；duck_sit 是 S49 链上的高难度用例。
2. **不移花接木 v46 骨架跑 v49 舞**（除历史对照）；编译失败补丁见 23.6 §4.4。
3. **舞蹈 Sim 键盘**：R 可一键激活 RL（用户确认舞蹈不需 W/A/S/D）；行走仍 R→F→P。
4. **ONNX 默认名 policy_s42.onnx**：拷贝时随意改名，info 对齐即可。
5. **Hybrid 为 S49 duck_sit 当前唯一验收标准**；纯 RL 列入第九章可选，不阻塞交付。
