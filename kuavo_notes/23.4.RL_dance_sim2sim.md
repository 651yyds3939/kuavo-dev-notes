## 📌 前情提要与核心痛点

在强化学习控制器的开发管线中，经常会遇到一个极其典型的幽灵表象：机器人在 Isaac 训练沙盒的离线回放脚本 `play.py` 里舞得极其平滑、欢快，但在 MuJoCo 验证端一按下键盘的 `R` 键接管控制，机体就会瞬间向后瘫倒。

这种从天堂跌入地狱的巨大落差感，足够让任何算法工程师开始怀疑人生。**请必须记住：当 Isaac 端的 Checkpoint 权重已经被证实能够完美追踪跳舞动作时，实机或 MuJoCo 部署侧的任何崩溃均与训练步数（Iter）无关，故障几乎 100% 锁死潜伏在观测数据链路（Observation Pipeline）的组装和语义对齐上。** 本文完整记录了我们团队跨越“第一幕后倒”到“第五幕 GUI 拖不动”的全部项目排雷实录。



* **RL 切换逻辑（全系列统一）**：
  | 场景 | 操作界面 | 连招 |
  | --- | --- | --- |
  | **MuJoCo Sim2Sim** | **`joystickSimulator.py` 窗口**（键盘） | **R → F → P**（**15.3** §5.2） |
  | **真机 H12** | 物理遥控器 | **F 拨杆** WBC 起立 → **B 键** RL（**15.4** §5.2） |

* **核心联动配置文件**：`skw_rl_param_dance.info`（下位机控制参数卡）、`play.py`（模型导出脚本）、`humanoidController.cpp`（C++ 核心状态机）、`flat_csp.info`（部署空间定义表）。
* **最高统一物理版本**：`ROBOT_VERSION=49` | **观测特征空间**：115 维满血状态 | **当前部署策略目标**：`policy_s49.onnx`。
* **训练侧背景（与本文部署排障无关，但解释 S49 更慢）**：S49 链因 **115 维观测 + URDF 运行时转 USD + 1565 帧 GPU 轨迹**，同一张 8GB 卡上仅宜 **`num_envs=4096`、~16000 steps/s**，低于 S42/S46 的 **8192 / ~43000**；因果见 **23.1** §1.1.1、**23.6** §1.0.3。

### 0.2 三分钟极速离线验收指令规范（正文）

在仿真拉起后，无需通过肉眼观察机器人是否跌倒，直接新开一个终端，砸入 ROS 原生调试命令，捕获下下位机塞给神经网络模型的第一帧原始输入快照：

```bash

```

* **黄金健康期望指标**：① 特征长向量的物理总维度输出必须精准等于 **`115`**；② 末尾的打拳时间相位 `commandPhase` 随着高频时钟不断产生 sin/cos 的周期性有规律起伏；③ 第二块重力切片 `gravity_body` 在机器人笔直站立时必须雷打不动地逼近 **`[0.0, 0.0, -1.0]`**；④ 关节角速度 `jointVel` 量级正常，展现出正常的噪声微颤。

> ### 🛑 恶性 Bug 案例：特征维数明明 100% 对齐，为什么机器人 **RL 接管后**（Sim：**R→F→P**）依然“后倒砸碎”？
> 
> 
> * **发病发头发病现象**：控制台输出的 `singleInputData` 读数明确显示出了完美的 115 维尺寸，打拳相位的 sin/cos 值也在随着时钟高频跳变。但当调试员完成 RL 接管（**Sim：joystickSimulator 窗口 R→F→P**；**真机：F 拨杆起立后 B 键**）的一瞬间，机器人依然倔强、毫无征兆地向后仰倒。
> * **深度病理剖析**：特征维数一致只能说明 C++ 侧提取的数据“块数总量”和总内存空间开辟对了。但由于乐聚底层控制器的字典解析器极其粗糙，**块内部的微观空间切片顺序或者物理尺度（Scale）极易发生大面积错位**！我们通过行级拆解，抓到了三大隐形内鬼：① 26 维的当前关节相对位置 `jointPos` 与上一步策略动作 `action` 的数据块空间发生了前后颠倒对调；② 第二块重力通道表面上配置为了 `gravity_body`，但底层 C++ 实际上依然把表示姿态的 Roll-Pitch-Yaw 欧拉角（`baseEuler`）塞了进去；③ 关节速度通道被错误地缩放了 20 倍。神经网络接到这些乱码重力后，以为自己正在处于垂直倒立的状态，直接引发了大脑的错误力矩反馈。
> * **唯一工程解药**：立刻打开训练 run 目录下的 `params/env.yaml` 配置文件，以各数据块的起始内存偏移量（Offset）为根据，逐行打印并核对 ROS 的张量切片数值量级。
>

## 第 1 章：P0 级挺尸后倒大病灶：观测空间链路全尺度全量拆解

当遭遇第一幕“按 R 即刻后倒挺尸”时，我们逐个维度疯狂打印了下位机发送的二进制缓冲区。我们发现，C++ 控制器拼出来的张量语言与 Isaac 训练时大脑习惯的字母表语义完全是南辕北辙的。我们在此将**训练侧真实 89 维基线**与 **115 维满血舞蹈扩展特征空间**的绝对起止点和映射键名进行全量大解密：

### 1.1 训练侧真实观测特征向量绝对顺序清单（Ground Truth）

#### 1.1.1 `base_ang_vel`（3 维，绝对起始偏移量 Offset: 0）

* *微观物理含义*：机器人根躯干机体系下（body frame）的传感器原始三轴角速度。
* *训练侧缩放尺度*：`scale = 1.0`。
* *部署侧 Info 字典专属绑定键名*：无条件匹配并查找 **`bodyAngVel`** 字典结构体。

#### 1.1.2 `projected_gravity`（3 维，绝对起始偏移量 Offset: 3）

* *微观物理含义*：重力矢量在机身当前坐标系下的高频三轴投影向量。机器人笔直直立站稳时，其物理量级必须无条件逼近于 **`[0.0, 0.0, -1.0]`**。
* *致命部署地雷*：**绝对、千万不能使用 Roll, Pitch, Yaw 欧拉角来冒充此项数据！**
* *部署侧 Info 字典专属绑定键名*：无条件匹配并查找 **`gravity_body`** 字典结构体。

#### 1.1.3 `velocity_commands`（3 维，绝对起始偏移量 Offset: 6）

* *微观物理含义*：用户通过摇杆下发给底盘的目标期望运动线速度与偏航角速度 $(v_x, v_y, \omega)$。
* *特技舞蹈约束*：在全身舞与原地冲拳模式下，此项特征组在仿真和实机中**恒定用全 `[0.0, 0.0, 0.0]` 填满死锁**。
* *部署侧 Info 字典专属绑定键名*：无条件匹配并查找 **`command`** 字典结构体。

#### 1.1.4 `joint_pos_rel`（26 维，绝对起始偏移量 Offset: 9）

* *微观物理含义*：当前全车 26 个自由度关节的实时传感器编码器角度值，减去默认的静态屈膝直立平衡角度后的相对偏差。
* *排列语义*：严格采用 Isaac Lab 资产按英文字母表升序排列出的原生字典序（即左腿->右腿->左臂->右臂）。
* *部署侧 Info 字典专属绑定键名*：无条件匹配并查找 **`jointPos`** 字典结构体。

#### 1.1.5 `joint_vel_rel`（26 维，绝对起始偏移量 Offset: 35）

* *微观物理含义*：机器人 26 个电机的实时关节轴运动角速度。
* *训练侧缩放尺度*：在 `env.yaml` 中标记为 `scale = null`（即原滋原味保留 1.0 物理真实标度值传入网络）。
* *部署侧 Info 字典专属绑定键名*：无条件匹配并查找 **`jointVel`** 字典结构体。

#### 1.1.6 `last_action`（26 维，绝对起始偏移量 Offset: 61）

* *微观物理含义*：上一个仿真控制步策略网络输出的 26 维位置期望控制历史残留缓存值。
* *训练侧缩放尺度*：在训练包底层被死死施加了 `actionScale = 0.30` 的物理限幅弧度放缩。
* *部署侧 Info 字典专属绑定键名*：无条件匹配并查找 **`action`** 字典结构体。

#### 1.1.7 `action_phase`（2 维，绝对起始偏移量 Offset: 87）

* *微观物理含义*：通过公式 $[\sin(2\pi \cdot \text{frame}/\text{total}), \cos(2\pi \cdot \text{frame}/\text{total})]$ 演算得出的二维高频打拳律动连续时间轴相位。
* *排列规范*：**必须且永远只能死死堆叠在 89 维或 115 维长张量的最末端！**
* *部署侧 Info 字典专属绑定键名*：无条件匹配并查找 **`commandPhase`** 字典结构体。

#### 1.1.8 89 维基线总容量校验

总维数数学算式严格闭环：$3 + 3 + 3 + 26 + 26 + 26 + 2 = 89$ 维。部署参数卡里的 `numSingleObs` 必须与 ONNX 模型的第一个输入维度保持完全一致。

#### 1.2.3 `skw_rl_param.info` 到底错在哪？

#### 1.2.4 特征位置空间发生了完全的“逆向反转”

在乐聚原厂下发的默认配置文件中，单时间步特征排版顺序是反的：它固执地把 `commandPhase(2)` 扔在了最前面的第 0 位，而把表示姿态的 `baseEuler(3)` 堆在了大数组的末尾。

* **致命误区**：**请必须铭记，在底盘控制器的 C++ 解析中，单个 `startIdx` 参数指示的只是源数据指针的初始偏置，它根本无法用来调整输出张量里的先后次序！真正的特征排版顺序，完全取决于你在 info 文件里各字典名称书写的先后排版次序！** 请立即打开编辑器，将你的 `skw_rl_param_dance.info` 数据特征块排版，强制精修为以下通过严格递增排版对齐的标准格式：

```text
singleInputData
{
    ; 🟢 严格遵从 env.yaml 字母表顺序，用文本的书写先后次序控制 C++ 的内存叠加
    bodyAngVel      { startIdx 0  numIdx 3   obsScales 1.0 }
    gravity_body    { startIdx 0  numIdx 3   obsScales 1.0 }
    command         { startIdx 0  numIdx 3   obsScales 1.0 }
    jointPos        { startIdx 0  numIdx 26  obsScales 1.0 }
    jointVel        { startIdx 0  numIdx 26  obsScales 1.0 }  ; ⚠️ 核心注意：放缩比例必须改回 1.0
    action          { startIdx 0  numIdx 26  obsScales 1.0 }
    commandPhase    { startIdx 0  numIdx 2   obsScales 1.0 }
}

```

#### 1.2.5 隐藏的“僵尸腿”慢性毒药：`jointVel obsScales=0.05`

原厂配置里错误地将关节速度的比例设为了 `0.05`。这导致神经网络大脑在 MuJoCo 中接到的全车角速度反馈**被无情缩减、掐灭了足足 20 倍**！大脑误以为机器人的双腿和下盘在物理世界上正处于“绝对静止不动”的状态，因此不会下发任何抗震代偿力矩，导致机体在仿真里表现得极度僵硬，沦为毫无平衡能力的“僵尸腿”。**必须将该项放缩系数强制复原修改为 1.0**。

> ### 🛑 经典复盘：baseEuler 欧拉角恶意冒充投影重力向量
> 
> 
> * **发病发头发病现象**：`rostopic` 查数显示第 3 到第 5 维切片（重力通道）的数据读数一直在正负 $\pi$ （折合 3.14）的超大弧度量级范围内疯狂抖动；并且在机器人绝对站直时，数值也不是干净的 `[0, 0, -1]`。
> * **深度病理剖析**：由于我们在 info 里误用了原厂带有 `baseEuler` 属性的底层接口，导致 C++ 控制器把机身俯仰、横滚、偏航的三个欧拉角传给了网络。这在依赖 `projected_gravity` 重力加速度计方向投影进行站立平衡的策略大脑看来，等同于接收到了一串毫无物理规律的垃圾乱码，模型会瞬间因为误判自己正在侧翻而疯狂输出自杀出力。
> * **唯一工程解药**：在 `skw_rl_param_dance.info` 字典里，将该项键名改写为标准的 `gravity_body`，强行指引底层指针去抓取 IMU 结算出的投影重力，并利用 `rostopic echo` 验证直立快照必须收敛在 `[0, 0, -1]` 附近。
>

### 1.3 模型导出脚本 `play.py` 内部隐含的 ONNX 导出维度地雷

有些开发人员在面对网络输入维度从经典的 87 维膨胀至 89 维时，为了骗过 ONNX 的 Tracing 动态追踪工具，擅自在 `_OnnxPolicyExporter` 导出类的内部，将 2 维的打拳相位时间轴 `command_phase` 直接粗暴地插在了 26 维的关节角度 `jointPos` 数组的中间位置（即强行嵌在了第 26、27 维上）。

* **技术反思**：这是一种极其业余且饮鸩止渴的错误打补丁手段。因为这种随意的中间插入，会直接打断后方 `joint_vel` 等所有块在内存上的物理连续性。
* **高阶重构思路**：**保持 C++ 侧发送向量的绝对干净连续。C++ 怎么拼，ONNX 就怎么解包！** 导出时，前置的张量重组函数仅针对 `jointPos`、`jointVel`、`action` 这三大经典块进行原厂 `gym2lab` 的关节序列转换，并在输出层外壳包裹 `[:, self.lab2gym]` 确保电机拓扑映射无误；**严禁、绝对严禁在中间执行任何跨类别的特征切片强行穿插操作！**

```python
    def forward(self, x):
        # 🟢 最优雅的无损追踪重组：只在块内部做字母表还原，保持时间相位的末尾连续性
        x_shuffled = self._reorder_joint_blocks_gym_to_lab(x)
        return self.actor(self.normalizer(x_shuffled))[:, self.lab2gym]

```

### 1.4 备用存根 `flat_csp.info` 内部潜伏的两个隐形暗坑

即便你在主运行卡里修改完了一切，在核对 `flat_csp.info` 这一份备用存根时，也要时刻警惕两个容易引发编译器报废的致命细节：

* **暗坑一：`actionPhase` 键名拼写分歧**：在有些老版本的 C++ 控制节点中，解析器对大小写极度敏感。如果主卡里写的是 `commandPhase`，而你在存根里误写成了驼峰式的 `actionPhase`，会导致下位机在加载该字典时发生键值丢失，让打拳相位的特征缓冲区恒定填零。
* **暗坑二：试图通过修改 `startIdx` 来硬调数据顺序**：有些工程师在发现 MuJoCo 观测错位后，试图通过将 `startIdx` 分别改成 3、6、9、35 等数值来一厢情愿地认为可以“指定数据在输出向量里的落脚点”——**这是对乐聚 C++ 字典解析器底层逻辑的严重误解！** 前文已强调，原厂规定所有的 `startIdx` 必须保持为 `0`，系统内部是纯粹靠文本块的**上下排版次序**来自动累加内存偏移的。瞎改这个数字，只会直接引爆 Eigen 矩阵越界断言，引发 `nodelet_manager` 闪退灾难。

### 1.5 部署「内鬼配置文件」：`flat_csp.info` vs 真正被执行的 `skw_rl_param*.info`

S42/S46 历史 Sim2Sim 最大坑之一：**你在 `flat_csp.info` 里改好了 `networkModelFile`，但 `roslaunch` 默认加载的是 `skw_rl_param.info`（原厂深蹲/走路任务卡）**，终端仍打印 **`Kp:60`** 和 **`0115_p120.onnx`** 等内鬼网络——表现为能走不能舞、或一接管就软倒。

* **S49 舞蹈推荐解**：专用 launch **`load_kuavo_mujoco_sim_dance_s49.launch`** 已通过 `rl_param` 指向 **`skw_rl_param_dance.info`**（含 **`networkModelFile /policy_s49.onnx;`**、`numSingleObs=115`、`cycleTime=31.30`）。
* **手动 override（历史链）**：`roslaunch ... rl_param:=/绝对路径/你的.info` 强行指定参数卡；**改文件必须改「launch 实际加载的那一份」**。
* **`flat_csp.info` 角色**：观测维数 **`numSingleObs`** 与 ONNX 接口的**备用存根/对照表**；**不能替代**主 RL 参数卡里的 `singleInputData` 与 PD 增益。

#### 1.5.1 下肢 `jointKp=60` 软脚挺尸（chat 经典「一按接管就向后拍倒」）

原厂 **`skw_rl_param.info`** 为走路/深蹲设计，髋膝 **`jointKp` 仅 ~60**，RL 接管后 MPC/WBC 托举消失，58kg 机身 + 双臂大惯量会把软腿**直接压垮**。S49 舞蹈卡 **`skw_rl_param_dance.info`** 已克隆训练侧黄金增益（髋 **100**、膝 **150**、踝 **40** 等）；若 Sim2Sim 仍挺尸，第一件事是对照 launch 日志 / `printRLparam` **确认运行时 Kp 不是 60**。

## 第 2 章：时间周期（`cycleTime`）错拍与 `cmdStance` 相位冻结死锁

当经历第二幕“机器人能舞动起来、但手脚动作完全跟不上音乐拍子、动作慢了十倍”的诡异扭动现象时，我们盯着控制台里的打拳相位连续看了一个下午，最终在下位机的 C++ 时间状态机源码里抓到了核心内鬼。

### 2.1 C++ 底层相位的动态自增推进方程

我们翻阅并审查了 `humanoidController.cpp` 中的相位递增推进核心逻辑：

```cpp
if (is_rl_controller_) {
    // 根算式：根据周期与频率推算出当前动作的总期望帧数
    const int num_frames = std::max(1, (int)std::lround(cycleTime_ * inferenceFrequency_));
    phase_ = (double)(episodeLength_ % num_frames) / (double)num_frames;
}

```

#### 2.1.1 斩断历史残留：`cycleTime` 必须与 CSV 实际帧数死死绑定

在乐聚原厂的配置文件中，`cycleTime` 被默认硬编码为了微小的 **`1.2s`**。这是专门为了 1.2 秒基础走路步伐设计的。

* **错拍子根因**：而我们通过模仿学习炼出的经典 642 帧 CSV 数据表，在 50Hz 的控制步长下，其真实的物理动作总长度是 $642 \div 50 = 12.84\text{s}$！原厂 1.2 秒的错误配置，逼迫下位机状态机以快了整整 **10.7 倍的疯狂超频速度**在脑海里疯狂刷新动作拍子，导致神经网络收到的相位与实际 CSV 的行索引完全脱节错位，动作自然陷入生涩的错乱。
* **工程解药**：必须在 `skw_rl_param_dance.info` 顶部，将 `cycleTime` 强行覆写修改为标准的 **`12.84`**（针对 642 帧）或 **`31.30`**（针对 1565 帧）。

#### 2.1.2 隐蔽的内鬼开关：`cmdStance` 参数引起的相位永久冻结死锁

在老版本的走路模型里，有一行极度具有误导性的硬编码代码：当全局遥控状态常数 `cmdStance=1` 时，状态机内部会强行将 `phase_` 截断重写为固定值 `0`！

* **死锁表象**：这导致无论仿真跑了多少个步数，输入给 ONNX 模型最末端的两个连续时间相位特征永远被锁死在临界点 **`[sin0, cos0] = [0.0, 1.0]`** 上，毫无任何波动变化！这等同于在精神层面上给机器人实施了定身术，机器人的动作会永久卡死在 CSV 轨迹的第 1 帧，展现出滑稽的手臂僵直。
* **工程解药**：在进入舞蹈强化学习模式前，必须通过参数卡将 `cmdStance` 初始默认值强制配置为 **`0`**；并且在 C++ 源码中严格规范：唯有当接管标志位 `is_rl_controller_` 完全为真时，内部的时间滴答自增器 `episodeLength_` 才允许执行 `++` 递增。

#### 2.1.3 推理频率（`inference_frequency`）必须无条件锁定 50Hz

部分开发人员为了追求极高的实机响应速度，在 launch 文件里擅自将控制推理频率改成了 `inference_frequency=100Hz`。

* **维度断层大灾难**：我们在 Isaac Lab 里训练打拳和全身舞时，标定的物理交互步长是严格的 `dt=0.02s`（即 50Hz）。如果部署端超频到 100Hz，状态机内部计算出来的总帧数 `num_frames` 会直接翻倍，导致机器人以双倍的惊人速度在仿真里疯狂跳舞，其关节角速度反馈也会彻底撑爆网络承受极限，直接触发由于力矩过载引起的炸机崩溃。**必须在 launch 文件中将推理时钟死死锁定在 50Hz**。

## 第 3 章：前栽踉跄大起底：CSV 轨迹内置位移与奖励函数分配（Trade-off）

当全套观测特征空间和时间轴拍子全部对齐后，机器人终于能够优雅地在 MuJoCo 中站立并跳舞了——但仅仅运行到第 5 秒，机体就伴随着上臂的出拳，开始顽固地、一颠一跛地朝着前方连续发生滑步侧移，并最终向前踉跄栽倒。

### 3.1 完美限制版 `PERFECT_LIMIT` 动作表内深藏的“人类出拳迈步惯性”

#### 3.1.1 数据转换管线的回溯审计

我们开启了对这一版 642 帧 CSV 的最深层行级解码。追溯其历史数据链我们发现：这套数据来自于 Roban 14 真实机甲在新年展会上面表演的全身舞蹈捕捉。原作者在录制动捕动作时，为了把拳头挥得更具视觉冲击力，**其右腿在空间层面上发生了极其明显的向前跨步、转移身体重心的真实移动动态！** 经过 `convert_punch.py` 转换重采样后，这些向前跨步的角度变化轨迹，被原封不动地保留在了数据表的下肢控制总线上。

#### 3.1.2 右腿控制总线（col 7~11）的微观异动

数据特征统计显示：在动作进行到中盘时，右大腿和右膝关节的控制弧度在时间轴上出现了高达 **`3.24 rad`** 的大范围起伏！这在人形机器人控制中是一个绝对的迈步跨越信号。当我们的强化学习策略在底层极度忠实、严丝合缝地去强行跟踪复现这一段 CSV 的关节期望时，双腿就会在现实里产生向前迈步的冲动。

#### 3.1.3 策略目标函数的约束空白（“原地打拳”约束遗失）

在训练侧的奖励函数配置中，虽然我们把目标速度指令输入锁死为了 0。但神经网络是一个极其狡猾的数学机器，在原有的奖励配置中，**如果我们缺少了强力惩罚机身根节点根躯干发生任何真实物理位移的控制大棒（即 `base_lin_vel_xy_stationary` 惩罚项）**，智能体就会敏锐地发现：即使机身在物理世界里往前跌蹭、发生了真实的 XY 轴漂移，它也根本不会受到任何扣分处罚！于是它选择全盘照抄 CSV 里的迈步轨迹，导致机器人在 MuJoCo 里直接往前栽倒挺尸。

* **工程解药**：必须在训练侧的 `RewardsCfg` 配置单里，将原地防滑步物理约束 `base_lin_vel_xy_stationary` 赋予高达 **`-1.3`** 的重型高负面惩罚权重，直接从数学原理上死死压制住智能体任何试图朝前跌蹭的欲望。

### 3.2 熟练运用四种动作剪裁 Profile 实现完美的原地化自平衡

为了从根源上清洗掉 CSV 动作表里潜伏的位移毒素，我们通过开发 `adapt_dance_csv.py` 物理滤波器，为整套数据链定制了四种完全不同的重组 Profile：

1. **`inplace` （下肢小幅弱化律动模式）**：等比例成倍压缩下肢 12 个电机的动作起伏跨度。在仿真中能显著减少前栽几率，但由于右腿跨步的残余应力仍在，机器人依然会有极微弱的前倾趋势。
2. **`arms_only` （下肢死锁钢筋混凝土模式）**：最为极端粗暴的手段，直接将 col 0~11 全量强制抹杀并替换为默认的屈膝站立角，数据流里只有双臂在挥舞。在 MuJoCo 仿真中表现为下半身如雕塑般纹丝不动，只有手臂在完美比划编舞招式。**这是整个项目在真机实体带电挂绳首次点火验证时的最高保命防爆盾牌！** 建议第一步首验先采用此 Profile。
3. **`expressive` （对称大屈膝后倒大陷阱）**：盲目追求视觉张力，人为制造了大面积、极夸张的对称双膝深蹲屈膝波形。这在人形控制中是毁灭性的灾难，大幅度的出拳由于缺少骨盆的主动平衡代偿，会导致整机重心瞬间突破脚后跟物理红线，机器人在 MuJoCo 仿真中会直接报废向后仰倒。**这是一个极其惨烈的失败 Profile，未来禁止再次调用**。
4. **`fullbody_inplace` （同反相分解终极选用模式）**：全项目组的最终数学结晶。脚本利用复杂的波动方程，将左右腿的原始动作波形分解为了同相分量（两腿同步微调，保留两腿跟着音乐拍子在原地上下微微小幅弹跳的舞蹈美感）与反相分量（彻底截断和压缩导致身体发生平移迈步的左/右交替重心转移分量）。**这是经受住了 S42 与 S49 满血闭环验证的最终首选 Profile，能完美达成原地律动与手臂出拳的协调自平衡**。

在下位机 C++ 节点中，状态机每次通过 `episodeLength % num_frames` 提取出当前时间戳下的 CSV 目标期望角度时，由于开发人员的逻辑失误，**错误地调用了正向重排矩阵 `gym2lab`，而没有去使用必须的逆向打包换算解码器 `lab2gym`！**

#### 4.1.2 彻底终结劈叉：C++ 侧 `lab2gym` 逆置换常数矩阵硬编码锁死

为了彻底将这个隐形地雷摘除，必须在 `humanoidController.cpp` 的重组核心模块里，强行硬编码锁死置换矩阵常数，确保其取值完全等于 `play.py` 离线导出的完美逆变换序列。

```cpp
// 🟢 舞蹈/打拳专属：RL 接管（joyCallback BUTTON_RL / 键盘 R→F→P 之 R）瞬间，执行绝对的记忆清洗
if (transition_to_rl_mode_triggered) {

}

```

## 第 5 章：侧移滑步暗鬼：WBC 状态机与仿真执行层物理假设断层

在解决了劈叉 Bug 后，机器人能坚持跳舞相当长的一段时间，但随着动作幅度的加大，它又暴露出了一个诡异的毛病：**身体开始持续、缓慢、顽固地朝着侧面发生斜向滑步侧移，并最终在挪出安全区域后失去平衡砸倒。**
诡异的是，这个 checkpoint 在 Isaac 训练端自带的 `play.py` 视频渲染里，下盘稳如泰山，毫无任何侧移滑步倾向。这暴露出训练侧与执行部署层在“双腿力学语言”上发生了深度断层。

### 5.1 揭开 `ankleSolver` 踝关节并联力学解算器的干扰黑盒

通过对部署包底层 C++ 运动学模块进行地毯式地肉眼审计，我们终于抓到了藏匿最深的物理内鬼：乐聚原厂的控制仓在 `updateRLcmd()` 刷新底盘电机输出命令时，下半身控制总线上的 12 个腿部电机，在接收到网络下发的 Action 后，**会被系统强制、隐式地送入一个名为 `ankleSolver` 的并联脚踝闭环力学解算器进行二次换算干预！**

* **物理假设断层**：然而，我们在 Isaac Lab 环境里使用 `DelayedPDActuator_S49` 纯关节执行器算子训练大脑时，神经网络是在极其纯净、不带任何外部公式干预的干净关节空间（Joint Space）里直接输出纯 PD 电流力矩的！部署端这个擅自加戏的 `ankleSolver` 解算器，把原本网络计算好的纯正自平衡力矩进行了二次扭曲和截断，导致脚踝出力在侧向产生了解算误差残差，在现实中直接转化为无法抑制的侧向斜向滑步。

## 第 8 章：Sim2Sim 恶性故障速查与抢救断路器指南

### 8.1 F1 诊断项： TensorBoard 里的模仿追踪分高到吓人，但 play 出来直接趴地挺尸刷分

## 第 9 章：`humanoidController.cpp` 状态机三大核心 C++ 代码级魔改备忘录

### 9.1 第一处魔改：时间轴相位滚动与 `episodeLength_` 步数推进约束

在状态机的时钟刷新环路中，我们强行剥离了行走控制器的干扰，重构了纯净的舞蹈控制主时钟：

```cpp
// 🟢 唯有当系统标志位确认 RL 接管控制器完全拉起时，时间轴总嘀答步数才允许累加滚动

    episodeLength_++;  
} else {
    // Stance/WBC 状态下主时钟死死锁定填零，阻断开局错拍子！
    episodeLength_ = 0; 
}
// 严格根据当前配置刷新总期望帧数边界
const int num_frames = std::max(1, (int)std::lround(cycleTime_ * inferenceFrequency_));
```

### 9.2 第二处魔改：`referenceJointPos` 内存提取与逆向打乱重组

```cpp
// 🟢 根据自增时钟，精准提取当前帧的 CSV 原始 Lab 字母序行指针
Eigen::VectorXd raw_lab_frame = danceTrajectoryLoader.getFrame(episodeLength_ % num_frames);

// 调用硬编码死锁的 lab2gym 置换长数组，将学术字母序重拍换算为硬件总线拓扑序
```

### 9.3 第三处魔改：全面旁路（bypass）注销腿部并联 `ankleSolver`

为了彻底对齐 Isaac Lab 训练端的纯关节空间绝对 PD 闭环假设，我们在力矩分发的最下游，无情地斩断了并联脚踝解算器的干预路径：

```cpp
// 🟢 识别到进入舞蹈闭环，强行旁路注销解算器黑盒，直接关节空间纯力矩输出！
if (is_rl_controller_ && current_task_mode == DANCE_PUNCH_TASK) {
    for (int leg_idx = 0; leg_idx < 12; ++leg_idx) {
        motor_effort_output[leg_idx] = joint_space_kp[leg_idx] * (target_q[leg_idx] - current_q[leg_idx]) 
                                       + joint_space_kd[leg_idx] * (target_dq[leg_idx] - current_dq[leg_idx]);
    }
    bypass_ankle_solver_flag = true;
}

```


### 9.4 启动日志里的 `Failed solution:` 与 WBC 起立时序

MuJoCo / 真机 launch 初期，WBC 求解器常刷屏 **`Failed solution: 0.999...`** 一类数值——多数是**名义姿态下 MPC/WBC 迭代未一次收敛**，**不等于节点已崩**。

* **正确时序**：等 **`Run` 变绿**、机器人完成 **WBC 微蹲—起立**、双脚站稳 **≥2 秒**，再在 **§9.5** 的 `joystickSimulator` 窗口内完成 **`R → F → P`** 解锁连招；**切忌 WBC 未站稳就接管**。
* **若 Failed solution 持续且机身明显发散**：检查 `ROBOT_VERSION`、URDF 与 `defaultJointState` 是否与 S49 一致，而非急于改 ONNX。

### 9.5 Sim2Sim 键盘「解锁连招」（与 **15.3** §5.2 / **15.4** 同源，已验证）

`joystick_type:=sim` 时，launch 会弹出 **`joystickSimulator.py` 独立终端**（不是 roslaunch 那个窗口）。等 MuJoCo **Run 变绿**、机器人 WBC 站稳约 **2 秒** 后，**在 joystickSimulator 窗口内**依次短按：

| 顺序 | 键 | 映射 | 作用 |
| --- | --- | --- | --- |
| 1 | **`R`** | `BUTTON_RL`（`joy` 按钮 2） | 激活 RL，加载 `policy_s49.onnx`，`is_rl_controller_=true` |
| 2 | **`F`** | `BUTTON_START`（按钮 7） | 进入准备态，触发 `real_initial_start` |
| 3 | **`P`** | `BUTTON_LB`（按钮 4） | `Walkenable=true`，解除安全锁 |

* **舞蹈任务**：速度指令已在训练侧锁零，**不必**再按 W/A/S/D；完成 **R→F→P** 后应直接原地挥拳。
* **行走任务**（S42 历史链）：R→F→P 后可按 W/A/S/D 给速度。
* **源码链**：`joystickSimulator.py`（`r`/`f`/`p`）→ `/joy` → `humanoidController::joyCallback`；与 **15.3**、**15.4** 部署包一致。
* **roslaunch 主终端备用**：直接按 **`r`** 也可切入 RL（`keyboard_thread_func`，与 R 同名）；Sim2Sim 验收仍以 **joystickSimulator 窗口 R→F→P** 为准。

#### 9.5.1 真机 H12 接管（与 **15.4** §5.2 一致）

| 步骤 | 操作 |
| --- | --- |
| 1 | **`F` 拨杆**拉满 → WBC 托举起立 |
| 2 | 站稳 **≥2 秒**、无啸叫 |
| 3 | 点按 **`B` 键** → INTO RL-Control（对应 `BUTTON_RL`） |

#### 9.5.2 `joystickSimulator` 未弹出 / 进程暴毙（Docker + gnome-terminal）

launch 用 `gnome-terminal` 弹键盘窗；**Docker 内无 X11/DBus** 时常报 `g-spawn-exit-error`，日志出现 **`[joystickSimulator-N] process has died`**，导致 **R→F→P 无法发送**（与 policy 对错无关，官方默认 ONNX 仍可能能走）。

* **验尸**：`rosnode list | grep joystick` 或 launch 日志里是否有 `process has died`。
* **手工补窗**（与 chat / **15.3** 同类 workaround）：
  ```bash
  docker exec -it <你的部署容器> zsh
  cd /root/kuavo_ws && source devel/setup.zsh
  python3 src/humanoid-control/humanoid_controllers/scripts/joystickSimulator.py
  ```
  保持该终端焦点，再按 **R→F→P**。
* **键位映射文件**：Sim 模式读 `launch/joy/sim.json`；其中 **`BUTTON_RL` 必须在按钮 2**（与 `joystickSimulator.py` 的 `r` 键一致，已对齐 `bt2.json`）。
