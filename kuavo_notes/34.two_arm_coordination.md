# 📖 34 双臂协同 — 右手抓瓶 + 左手拧盖（初版方案）

> **文档定位**：在现有单臂抓取栈（`moveit_auto_grasp.py` / LeRobot 采数）之上，新增 **规则式双臂协同** 的可落地方案与实机操作手册。  
> **执行入口**：`bimanual_unscrew.py`（本目录）  
> **关联文档**：[`moveit_grasp.md`](./moveit_grasp.md) · [`28.moveit_grasping.md`](./28.moveit_grasping.md) · [`20.gripper_issue.md`](./20.gripper_issue.md)  
> **版本**：v1.0（2026-06）— 几何推算瓶盖 + 分阶段时序 + 接触修正 + 腕关节旋转

## 第 0 章：一句话结论

| 项目 | 说明 |
|------|------|
| **做什么** | 右手侧夹固定瓶身 → 左手从上往下轻触瓶盖 → 腕关节旋转拧盖 |
| **不做什么** | 不用 YOLO 直接检瓶盖；不让双臂同时前伸；首版不做 LeRobot 训练 |
| **脚本** | `bimanual_unscrew.py` |
| **前提** | 与经典 MoveIt 抓取相同的 **终端 1~7** + **终端 8 换本脚本** |
| **红线** | 首跑必须有人扶、急停就绪；瓶放 **机器人右侧 (Y<0)** 更稳 |

## 第 1 章：为什么需要「初版方案」

单臂抓取（22.4 / `moveit_auto_grasp.py`）的设计前提是：

1. YOLO 只给出 **整瓶** 的 `(x, y)`，深度取 bbox 中心区域中值；
2. 每次只动 **一只手**，另一只手 `_freeze_inactive_arm()` 锁在 init（防重心前倾）；
3. 夹爪 **水平侧夹** 瓶身，闭合度 80~85，适合抓柱体，不适合拧小瓶盖。

「右手抓瓶 + 左手拧盖」需要解决三类问题：

| 问题 | 初版对策 |
|------|----------|
| 双臂互撞 | **分阶段**：仅 A 阶段动右手，B 阶段再动左手；空间上右手右侧、左手上方的分区 |
| 瓶盖太小 YOLO 检不到 | **几何推算**：瓶盖 = 瓶身检测点 + 标定偏移 + 瓶高常量 |
| 夹爪比瓶盖宽 | **改策略**：左手 **垂直接近 + 半闭合 + 低 effort + 绕 z7 旋转**，不侧夹瓶盖 |

## 第 2 章：整体架构

```text
┌─────────────────────────────────────────────────────────────┐
│  上位机 Orin：相机 + YOLO → /vla/yolo_target (瓶身 xy)      │
└───────────────────────────┬─────────────────────────────────┘
                            │ ROS
┌───────────────────────────▼─────────────────────────────────┐
│  下位机 NUC：bimanual_unscrew.py                             │
│    阶段 A  run_right_grasp_hold()   ← 复用 moveit_auto_grasp │
│    阶段 B  compute_cap_target()     ← 几何推算瓶盖           │
│    阶段 C  descend_until_contact()  ← 小步下降 + effort 触顶 │
│    阶段 D  twist_cap()              ← zarm_l7 分步旋转       │
│    收手    松爪 + execute_dual_arm_init_home()               │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 与 `moveit_auto_grasp.py` 的关系

| 模块 | 来源 |
|------|------|
| 视觉 10 帧采点 | `mag._collect_vision_targets_tf2_style()` |

| 双臂 init 归位 | `mag.execute_dual_arm_init_home()` |

**禁止** 与 `kuavo_moveit_bridge.py` 同时运行（同 [`moveit_grasp.md`](./moveit_grasp.md)）。

## 第 3 章：四阶段流水线

```text
阶段 0  双爪张开 → 双臂 init → YOLO 10 帧
    ↓
阶段 A  【仅右手】曲肘 → 预瞄 12cm → 水平插入 → 闭爪抓瓶身 → (可选) 微抬 5cm
    ↓         q_right_hold 锁定
阶段 B  【仅左手】几何算瓶盖 → IK 到瓶盖上方悬停 → 曲肘护胸 → 再至悬停
    ↓
阶段 C  小步下降 (4mm/步) → 左爪 effort 触顶 → 半闭合 (~45) 低 effort
    ↓
阶段 D  zarm_l7 每步 +8° × 15 步 ≈ 120°
    ↓
收手    双爪张开 → 双臂 init
```

### 3.1 阶段 A — 右手抓瓶（`run_right_grasp_hold`）

与 `moveit_auto_grasp.py` 相同，但 **差异**：

- **强制右手**（`is_left_arm=False`），不根据 Y 自动选手；
- **不做** 22cm 抬升与 vla 肩膀外摆收手；
- 抓稳后可选 **微抬 5cm**（`~right_micro_lift_m`），给左手在瓶子上方腾空间。

### 3.2 阶段 B — 瓶盖位置推算（`compute_cap_target`）

YOLO 给出的 `(bottle_x, bottle_y)` 经右手 TCP 补偿后，对应 **瓶身抓握高度** 上的点（`SAFE_LOCKED_Z = 0.385`）。

瓶盖中心：

```text
cap_x = bottle_x + cap_offset_x_m
cap_y = bottle_y + cap_offset_y_m
cap_z = bottle_z + bottle_cap_rise_m    # 默认 +12cm
```

**标定方法**（固定一种水瓶，做一次即可）：

1. 右手抓稳瓶身（或人工扶瓶）；
2. 目视/尺子量「抓握点 → 瓶盖中心」的水平偏移与竖直高度；
3. 写入 ROS param `~cap_offset_x_m`、`~cap_offset_y_m`、`~bottle_cap_rise_m`。

左手姿态：**从上往下**（`get_topdown_left_quat`），不是水平侧夹。

### 3.3 阶段 C — 接触搜索（`descend_until_contact`）

从 `cap_z + cap_hover_m`（默认悬停 3.5cm）开始，每步下降 `contact_step_m`（4mm）：

- 每步 IK + `execute_hold_right()`（右手关节不变）；
- 读 `/leju_claw_state` 左爪 `effort`，超过 `contact_effort_threshold`（0.55）视为触顶；
- 再 **半闭合** 左爪（`left_cap_close_pos=45`，`left_cap_effort=0.28`），防 wedge lock（见 [`20.gripper_issue.md`](./20.gripper_issue.md)）。

### 3.4 阶段 D — 拧盖（`twist_cap`）

只改 **左腕** `zarm_l7_joint`（索引 6），每步 `+twist_deg_per_step`（默认 8°），共 `twist_steps`（默认 15）步。

右手全程 `q_right_hold` 不变。

## 第 4 章：防碰撞设计

### 4.1 时序防撞（首版核心）

| 规则 | 说明 |
|------|------|
| A 阶段完成前 | 左手保持 init，不前伸 |
| B~D 阶段 | 右手锁定 `q_right_hold`，不再规划 |
| 禁止 | 双臂同时从远处插向瓶心 |

### 4.2 空间分区

```text
        左臂区（上/左）          右臂区（右/中）
              ↓                      ↓
         垂直接触瓶盖            水平侧夹瓶身
              \                    /
               \    瓶 + 盖      /
                \      |        /
                 ─── 桌面 ───
```

建议瓶子放在 **机器人右前方**（`Y < 0`），与现有单臂抓取习惯一致。

### 4.3 MoveIt 碰撞（v1 局限）

- 左手 IK 使用 `avoid_collisions=False`（与经典抓取 IK 一致）；
- URDF 手臂 **无 collision mesh**，OMPL 自碰撞不完整；
- v1 **未** attach 瓶体到右手 link；v2 可补 `PlanningScene.attachObject`。

**实机**：低速、有人监护、急停就绪。

### 4.4 WBC 质心能力与局限（必读）

终端 1 `load_kuavo_real.launch` 启动后，底层 **MPC + WBC（约 500Hz）** 在 **stance 站立** 时会跟踪质心，不是「不管腿只挥臂」。

**控制链路（与单臂抓取相同）：**

```text
/arm_traj_change_mode(2)  →  external_control
/kuavo_arm_target_poses   →  humanoid_Arm_time_target_control
                         →  /humanoid_mpc_target_arm → MPC + WBC
```

**WBC 里有什么：**

| 能力 | 说明 |
|------|------|
| 质心跟踪 | `WeightedWbc` 含 `weightComPos_`；可订阅 `/humanoid_controller/com/r` 与 `com/r_des` |
| 躯干微调 | `/humanoid_controller/wbc_planned_body_acc/linear` 等非零表示在调平衡 |
| 轨迹平滑 | WBC 1kHz 样条插值，减轻 jerk（见 `28.moveit_grasping.md`） |

**为什么不能指望「自动补偿双臂前伸」：**

| 局限 | 说明 |
|------|------|
| 补偿有物理上限 | 质心超出脚掌支撑多边形时，踝/躯干无法拉回 → 会前倾（实机已验证） |
| 外部控制关部分保护 | `control_mode=2` 时 **pull-up 保护不生效**（`humanoidController.cpp`） |
| MPC 手臂简化 | v49 `mpcArmsDof=8`，非完整 14 轴，大姿态下预测偏保守 |
| 应用层无补偿 API | `vla_grasp` **未**调用 `/cmd_pose`、`/cmd_vel` 做躯干后移；只能靠规划策略 |

**脚本侧对策（v1 已实现）：**

1. **分阶段** — 仅 A 动右手，B~D 再动左手（见 §4.1）  
2. **工作空间拒绝** — `validate_bottle_workspace()`：`X > bottle_x_max_m`（默认 **0.55m**）或 `Y > 0.05` 直接 abort  
3. **实机监护** — 阶段 B 起有人扶、急停就绪  

**实机观测质心是否在补偿：**

```bash
rostopic echo /humanoid_controller/com/r -n 1
rostopic echo /humanoid_controller/com/r_des -n 1
rostopic echo /humanoid_controller/wbc_planned_body_acc/linear -n 1
```

阶段 A 右手前伸时通常有小幅度调整；若双臂都在前且 `com/r` 的 x 持续偏大，说明 **补偿不够**，应移近瓶子或中止。

**若需更强平衡（v2，未实现）：** `/cmd_pose` 微退躯干、半身固定底座（`only_half_up_body`）、或 `mobile_manipulator_controllers` 整条栈。详见 [`docs/运动控制API.md`](../../docs/运动控制API.md)。

## 第 5 章：可调参数（ROS private param）

启动示例：

```bash
python3 src/demo/vla_grasp/bimanual_unscrew.py \
  _bottle_cap_rise_m:=0.115 \
  _cap_offset_x_m:=0.0 \
  _cap_offset_y_m:=0.01 \
  _twist_steps:=20
```

| 参数 | 默认 | 含义 |
|------|------|------|
| `~bottle_cap_rise_m` | `0.12` | 抓握平面到瓶盖中心的竖直距离 (m) |
| `~cap_offset_x_m` | `0.0` | 瓶盖 X 偏置 (m) |
| `~cap_offset_y_m` | `0.0` | 瓶盖 Y 偏置 (m) |
| `~cap_hover_m` | `0.035` | 接触前悬停高度 (m) |
| `~contact_step_m` | `0.004` | 触顶搜索步长 (m) |
| `~contact_max_steps` | `10` | 最多下降步数 |
| `~contact_effort_threshold` | `0.55` | 左爪触顶 effort 阈值 |
| `~twist_deg_per_step` | `8.0` | 每步腕旋转 (deg) |
| `~twist_steps` | `15` | 拧盖总步数 |
| `~left_cap_close_pos` | `45.0` | 拧盖左爪闭合度 (0~100) |
| `~left_cap_effort` | `0.28` | 拧盖左爪 effort 上限 |
| `~right_hold_after_grasp` | `true` | 抓后是否微抬 |
| `~right_micro_lift_m` | `0.05` | 微抬高度 (m) |
| `~bottle_x_min_m` | `0.30` | 瓶 X 下限，过近拒绝 (m) |
| `~bottle_x_max_m` | `0.55` | 瓶 X 上限，过远拒绝（防前倾）(m) |
| `~bottle_y_max_m` | `0.05` | 瓶 Y 上限，偏左拒绝（右手抓瓶）(m) |

工作空间检查失败时脚本 **不进入 MoveIt 抓取**，直接 `safe_abort()`。单臂抓取 YOLO 采点上限为 0.65m，双臂默认 **0.55m 更严**；若瓶必须在远处，可显式放宽（风险自负）：

```bash
python3 src/demo/vla_grasp/bimanual_unscrew.py _bottle_x_max_m:=0.60
```

### 5.1 调参顺序（建议）

1. **`bottle_cap_rise_m`** — 左手能否到达瓶盖正上方  
2. **`cap_offset_x/y`** — 触顶时是否偏左/偏右  
3. **`left_cap_close_pos` / `left_cap_effort`** — 能否带转瓶盖而不滑盖、不自锁  
4. **`twist_deg_per_step` × `twist_steps`** — 总旋转角是否足够（一般需 360°+ 多圈）

## 第 7 章：误差与精度（三层消化）

```text
层 1  粗定位   YOLO 整瓶 bbox + TCP  →  右手抓瓶身（±1~2cm 可接受）
层 2  中定位   瓶型几何 cap = f(bottle)  →  左手到盖上方（目标 ±5mm）
层 3  细定位   小步下降 + effort 触顶  →  对齐盖顶（±2mm）
```

**不要** 用 TCP 偏移单独修瓶盖 XY——TCP 修的是夹爪相对末端，修不了「bbox 中心 ≠ 瓶盖」。

若触顶仍偏：优先改 **`cap_offset_*`** 和 **`bottle_cap_rise_m`**，不是改 `TCP_OFFSET_Y_RIGHT`。

| 局限 | v2 方向 |

## 第 10 章：FAQ

**Q：能否瓶子放左边、仍用本脚本？**  
A：可以跑，但右手从右侧够左侧瓶身时易与左手工作区重叠；**建议 Y<0**。

**Q：YOLO 要不要改检瓶盖？**  
A：v1 **不需要**。固定瓶型用几何推算 + 触顶更稳。训 YOLO 检 cap 类 ROI 太小，不推荐首版投入。

**Q：拧不动怎么办？**  
A：增大 `twist_steps`；略增 `left_cap_close_pos`（≤55）；检查盖是否过紧；瓶盖加橡胶套增摩擦。

**Q：和 22.4 LeRobot 冲突吗？**  
A：不冲突。本脚本是 **独立终端 8 入口**；采数仍用 `lerobot_data_harvester.py` + `moveit_auto_grasp`。双臂模仿学习属后续 v3。

**Q：左手 IK 无解？**  
A：减小 `bottle_cap_rise_m` 或 `cap_hover_m`；确认右手微抬已开；左手先 `execute_dual_arm_init_home` 再跑全流程。

**Q：报「瓶 X 过远」无法执行？**  
A：故意设计。默认 `_bottle_x_max_m:=0.55` 比单臂 0.65 严，防双臂阶段质心前倾。把瓶移近，或 `_bottle_x_max_m:=0.60`（需有人扶）。

**Q：WBC 不是会自动平衡吗？**  
A：会 **尝试** 跟踪质心，但有物理上限；见 **§4.4**。不能替代分阶段 + 工作空间限制。
