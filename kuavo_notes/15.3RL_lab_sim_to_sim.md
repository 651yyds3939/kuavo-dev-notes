## 第一阶段：Isaac Lab 模型训练与导出 (Train)

### 1.1 源码精准修改 (防止训练报错)
在开始训练前，必须禁用底层关节加速度更新，否则会导致物理引擎结算冲突。
* **文件路径**: `~/IsaacLab/source/extensions/omni.isaac.lab/omni/isaac/lab/assets/articulation/articulation_data.py`
* **修改内容**: 注释掉约 82 行的 `self.joint_acc`（切勿修改 `self._joint_acc`）。
```python
    def update(self, dt: float):
        self._sim_timestamp += dt
        # self.joint_acc  <---- 【在此行前加 # 注释掉】

```

### 1.2 极限性能训练 (16GB 内存生存指南)

在 16GB 内存下，极易遭遇 OOM（内存溢出）导致进程被系统强杀。

* **终极策略**: 关闭视频录制，开启 `--headless`，并发环境设为 8192。

```bash
conda activate isaaclab
cd ~/leju_robot_rl
python scripts/rsl_rl/train.py --task Legged-Isaac-Velocity-Flat-Kuavo-S42-v0 --num_envs 8192 --headless

```

### 1.3 跨越 GUI 死锁导出 ONNX 模型

> ⚠️ **致命踩坑：图形界面死锁 (`GLFW initialization failed`)**
> 默认运行 `play.py` 会因 X11 显示服务冲突卡死在 `Starting the simulation...`。
> **解法**: 强行加入 `--headless` 参数，后台极速导出。

```bash
# 转换 .pt 为 .onnx
python scripts/rsl_rl/play.py --task Legged-Isaac-Velocity-Flat-Kuavo-S42-Play-v0 --num_envs 32 --headless

python scripts/rsl_rl/play.py --task Legged-Isaac-Velocity-Flat-Kuavo-Punch-v0 --num_envs 32 --headless
```

*注：执行后终端看似“卡住”，实则后台已在无限循环执行步态，直接 `Ctrl+C` 退出，即可在 `exported` 目录下获得 `policy_s42.onnx`。*

## 第二阶段：部署环境升级与依赖补全 (Deploy Prep)

强化学习部署代码与经典控制（MPC）彻底分离，必须拉取专属的 `beta` 分支，并补全 C++ 底层字典。

### 2.1 强制同步官方最新架构
强化学习的部署与经典控制不在同一个仓库，必须拉取专属的 `beta` 分支。

> ⚠️ **致命踩坑：C++ 字典与版本落后导致 CMake 报错**
> **表现**：编译时提示找不到 `humanoid_interface`，或者运行 RL 时报 `std::out_of_range (map::at)` 查无此词。
> **真凶**：
> 1. 没有更新 Git 子模块（Submodule），导致底层代码文件夹为空。
> 2. 本地仓库不是最新版，缺少官方最新针对 87 维 Flat 任务注册的输入词汇字典。
> **解决方案**：使用 `stash` 暂存本地冲突，强制拉取官方最新代码并递归更新子模块。
> 
> 

**完整执行指令：**

```bash
# 1. 克隆部署专属仓库
cd ~
git clone -b beta https://gitee.com/leju-robot/kuavo-rl-opensource.git
cd ~/kuavo-rl-opensource

# 2. 如果之前有修改，先打包暂存，防止检出冲突
git stash

# 3. 强制锁定 beta 分支并拉取最新全套代码
git checkout beta
git pull origin beta

# 4. 递归更新底层 C++ 子模块（极其重要，解决找不到包的问题）
git submodule update --init --recursive

```

### 2.2 植入神经网络大脑

将第一阶段生成的 `.onnx` 模型拷贝至部署工程：

```bash
cp ~/leju_robot_rl/logs/rsl_rl/Kuavo/s42/flat/【你的训练日期】/exported/policy_s42.onnx ~/kuavo-rl-opensource/kuavo-robot-deploy/src/humanoid-control/humanoid_controllers/model/networks/

```

## 第三阶段：神经通路对齐 (Config Alignment)

模型输入数据 (`Observation`) 必须与 C++ 控制器发出的数据在维度和顺序上做到 100% 严丝合缝。

* **文件路径**: `~/kuavo-rl-opensource/kuavo-robot-deploy/src/humanoid-control/humanoid_controllers/config/kuavo_v46/rl/flat_csp.info`
*(注：官方最新稳定配置位于 v46 文件夹，向下兼容)*

**核心修改步骤**：

1. **替换模型名**:
```text
networkModelFile          /policy_s42.onnx;

```


2. **确认 87 维输入矩阵 (`singleInputData`)**:
确保文件内的该模块完全按照以下顺序排列，任何错位都会导致机器人“神经错乱”开局平地摔。
```text
singleInputData
{
    bodyAngVel { startIdx 0, numIdx 3, obsScales 1.0 }
    baseEuler { startIdx 0, numIdx 3, obsScales 1.0 }
    command { startIdx 0, numIdx 3, obsScales 1.0 }
    jointPos { startIdx 0, numIdx 26, obsScales 1.0 }
    jointVel { startIdx 0, numIdx 26, obsScales 0.05 }
    action { startIdx 0, numIdx 26, obsScales 1.0 }
}

```



*(核对项：`numSingleObs` 必须为 `87`。`actionScale`、`JointKp`、`JointKd` 必须严格与训练时保持一致。)*

## 第四阶段：Docker 编译神坑终结

完全抛弃官方旧版 `run.sh` 脚本，手动挂载本地 1.3.0 满血镜像。

### 4.1 启动容器

```bash
cd ~/kuavo-rl-opensource/kuavo-robot-deploy

docker run -it --rm \
    --name kuavo_rl_deploy_container \
    --gpus all \
    --privileged \
    --network host \
    --env="DISPLAY" \
    -v /tmp/.X11-unix:/tmp/.X11-unix:rw \
    -v $(pwd):/root/kuavo_ws \
    kuavo_opensource_mpc_wbc_img:1.3.0 \
    zsh

```

### 4.2 清理旧缓存与编译

> ⚠️ **致命踩坑：Bash 与 Zsh 的“跨服聊天”**
> 容器使用 `zsh`，如果强行 `source setup.bash` 会导致路径解析崩溃；同时，必须清理上一次失败的 CMake 缓存，否则 `catkin` 会死记硬背旧的错误路径。

```bash
export ROBOT_VERSION=46
cd /root/kuavo_ws

# 1. 彻底清理旧的 CMake 缓存！
catkin clean -y

# 2. 使用对应的 zsh 脚本加载预编译环境！
source installed/setup.zsh

# 3. 仅编译我们需要的人形控制器
catkin build humanoid_controllers

```

## 第五阶段：Sim2Sim 仿真测试与极限交互

### 5.1 启动 MuJoCo 仿真

```bash
source devel/setup.zsh
roslaunch humanoid_controllers load_kuavo_mujoco_sim.launch joystick_type:=sim

```

### 5.2 键盘“解锁连招”

等夸父掉落站稳后（约 2 秒），**在运行 launch 的终端界面内**，依次按下：

1. **`R` 键 (RL)**: 激活强化学习模式，接管躯体。
2. **`F` 键 (START)**: 进入准备状态。
3. **`P` 键 (Walkenable)**: 解除安全锁。
*完成后按下 `W/A/S/D` 给定指令速度，机器人起步。*

### 5.3 极限抗干扰测试（修正版 · 2026-06）

> **重要结论（与官方 MuJoCo 原版不同）**  
> 本仓库的 MuJoCo 仿真由 `mujoco_node.cc` 的 **独立物理线程** `PhysicsLoop` 驱动 `mj_step`，**不会**在积分前调用 `mjv_applyPerturbForce`。  
> 因此在 GUI 里选中 body、按 Ctrl 拖红色力箭头，**往往只能看到可视化，机器人几乎不受力**。  
> **行走 / 跳舞 RL 的抗推测试，请统一走 ROS 话题 `/external_wrench`**（与训练部署同一套链路，已实测可用）。

#### 5.3.1 前置条件（缺一不可）

1. **MuJoCo 已 Run**：窗口左侧 `Simulation` 面板中 **Run** 为绿色运行状态（暂停时不积分）。
2. **RL 已接管**：按 5.2 节完成 `R → F → P`，机器人处于 RL 控制（行走或跳舞均可）。
3. **ROS 同一 master**：发外力的终端与 `roslaunch` **在同一 Docker 容器内**，且已 source：
   ```bash
   source /root/kuavo_ws/devel/setup.zsh
   ```
4. **确认订阅存在**（Subscribers 至少为 1）：
   ```bash
   rostopic info /external_wrench
   ```
   若 `Subscribers: 0`，说明 MuJoCo 节点未连上当前 roscore，外力发了也没人收。

#### 5.3.2 外力施加机制（实际逻辑）

| 项目 | 说明 |
|------|------|
| 话题名 | `/external_wrench`（`geometry_msgs/Wrench`） |
| 施加位置 | **躯干 base_link**（`mujoco_node.cc` 中 body index = 1） |
| 更新方式 | 每收到一次 `/joint_cmd` 控制周期，物理线程把 `external_wrench_` 写入 `d->xfrc_applied` 后 `mj_step` |
| 持续施力 | 必须 **高频重复发布**（如 `-r 50`），只发一条往往只作用一帧 |
| 坐标系 | `force.x` 前、`force.y` 左、`force.z` 上（与机器人 base 坐标一致） |

可选话题（按 link 名称施力，需持续发布）：

- `/external_wrench/left_hand` → `zarm_l7_link`
- `/external_wrench/right_hand` → `zarm_r7_link`

#### 5.3.3 手动抗推测试（推荐 · 已验证）

在 **新开一个终端**（同一 Docker、已 `source devel/setup.zsh`）执行。

> ⚠️ **zsh 陷阱**：整条 YAML 必须用 **单引号** `'...'` 包裹，且最外层 mapping 需要 **两个 `}`** 闭合（`torque` 一层 + 整段一层）。不要用双引号，否则 zsh 会把 `{}` 当 brace expansion 拆坏。

**向前推 100N（50Hz 持续）：**
```bash
rostopic pub /external_wrench geometry_msgs/Wrench '{force: {x: 100.0, y: 0.0, z: 0.0}, torque: {x: 0.0, y: 0.0, z: 0.0}}' -r 50
```

**侧向推 80N：**
```bash
rostopic pub /external_wrench geometry_msgs/Wrench '{force: {x: 0.0, y: 80.0, z: 0.0}, torque: {x: 0.0, y: 0.0, z: 0.0}}' -r 50
```

**向后推：**
```bash
rostopic pub /external_wrench geometry_msgs/Wrench '{force: {x: -100.0, y: 0.0, z: 0.0}, torque: {x: 0.0, y: 0.0, z: 0.0}}' -r 50
```

**停止外力（发一次零力后退出）：**
```bash
rostopic pub /external_wrench geometry_msgs/Wrench '{force: {x: 0.0, y: 0.0, z: 0.0}, torque: {x: 0.0, y: 0.0, z: 0.0}}' -1
```

**多行 YAML 写法（zsh 同样稳定）：**
```bash
rostopic pub /external_wrench geometry_msgs/Wrench 'force:
  x: 100.0
  y: 0.0
  z: 0.0
torque:
  x: 0.0
  y: 0.0
  z: 0.0' -r 50
```

**观察是否生效：**
```bash
# 终端 A：看话题是否在发
rostopic echo /external_wrench

# 终端 B：MuJoCo 收到外力时可能打印（取决于编译选项）
# in ext wrench callback!
```

**力度建议：** 先从 `50N` 试起，再逐步加到 `100N`、`150N`；跳舞策略对前向力更敏感，过大易倒。

#### 5.3.4 Python 持续施力（避开 shell 引号问题）

若 `rostopic pub` 仍报 YAML 解析错误，用 Python 最稳：

```bash
python3 << 'EOF'
import rospy
from geometry_msgs.msg import Wrench

rospy.init_node("external_wrench_pub")
pub = rospy.Publisher("/external_wrench", Wrench, queue_size=1)
rospy.sleep(1.0)

w = Wrench()
w.force.x = 100.0   # 向前；可改 y/z
rate = rospy.Rate(50)

while not rospy.is_shutdown():
    pub.publish(w)
    rate.sleep()
EOF
```

`Ctrl+C` 停止后，务必发一次零力（见 5.3.3 停止命令）。

#### 5.3.5 自动化极限抗推（可选）

仓库已自带测试节点，向 `/external_wrench` 按方向/力度扫参并记录是否摔倒：

```bash
source /root/kuavo_ws/devel/setup.zsh
rosrun mujoco_cpp force_disturbance_test_node
```

结果 CSV：`/tmp/max_force_thresholds.csv`，断点：`/tmp/force_test_checkpoint.json`。

#### 5.3.6 GUI 操作说明（不推荐用于 RL 抗推）

以下操作在 **原版 MuJoCo Simulate** 中有效，但在 **本仓库定制版** 中因物理线程未合并 GUI 扰动，**不能作为 RL 抗推验收手段**：

| 操作 | 本仓库是否可用 | 说明 |
|------|----------------|------|
| 双击选 body + Ctrl + 拖箭头 | ❌ 基本无效 | 仅可视化，力未进入 `PhysicsLoop` 的 `mj_step` |
| Physics → 改 Gravity Y | ⚠️ 部分有效 | 改的是全局重力，不是“侧风外力”，且与训练域可能不一致 |
| Simulation → Noise scale | ⚠️ 视版本而定 | 噪声注入，非标准 `/external_wrench` 链路 |

若必须用鼠标拖拽测力，需修改 `kuavo-robot-deploy/src/mujoco/src/mujoco_node.cc`：在 `mj_step` 前调用 `mjv_applyPerturbForce(m, d, &sim.pert)` 并与 `/external_wrench` 叠加。**日常测试请直接用 5.3.3 的 rostopic 方案。**

#### 5.3.7 抗推验收参考（行走 / 跳舞通用）

1. RL 模式下机器人稳定执行策略（走或舞）。
2. 施加 `100N` 前向力 1–2 秒，观察是否恢复或明显失衡。
3. 侧向 `80N` 各测一次。
4. `Ctrl+C` 停止发布，并发零力，确认不再持续受力。
5. 记录最大不摔倒力度，便于对比不同 checkpoint / 不同 `skw_rl_param.info`。
