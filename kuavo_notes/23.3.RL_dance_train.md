## 第 0 章：训练架构总览与开训前红线 Checklist

从 `master` 主分支切出的独立 `feat/punch` 分支，其核心工程逻辑是**将原本成熟的行走 baseline 彻底锁死代码，而将所有大范围的舞蹈/特技改造全量剥离在平行的虚拟宇宙中进行**。在开炉点火训练前，如果以下三项基础建设没有对齐，后续不管 TensorBoard 曲线收敛得多么漂亮，都纯粹是欺骗性的假高分。

### 0.1 开训前全量核对清单

* **Gym 任务注册状态**：旧链路对应 `Legged-Isaac-Velocity-Flat-Kuavo-Punch-v0`（服务于 S42/S46 阶段资产）；全新链路对应 `Legged-Isaac-Velocity-Flat-Kuavo-S49-Punch-v0`（统一 S49 阶段）。其中带有 `-Play-v0` 后缀的专属环境仅用于 ONNX 大脑模型的离线导出。

* **日志归置目录**：全量采样轨迹和权重数据固定存放在 `logs/rsl_rl/Kuavo/s49/dance/<timestamp>/` 文件夹下。其中转储生成的 `params/env.yaml` 是后续部署端进行数据对齐的唯一绝对金标准。

* **三大前置核心确认**：网络输出 Action 的 shape 必须严格对齐 `26`；策略网络的 policy obs 输入必须对齐 `115` 维；必须确认底层的 `DelayedPDActuator_S49` 执行器算子已全面生效，严禁误调用 S42 版本的踝关节解算器。



* **部署侧切换与训练无关**：Sim2Sim 验收用 **joystickSimulator R→F→P**；真机用 **H12 F→B**——见 **23.4** §9.5、**23.6** 第四/五章。

* **坚决禁止继承（resume）7461权重**：在我们彻底完成了下述的“四块核心物理修复”后，系统的奖励结构、执行器动力学以及 URDF 网格密度已发生质的变化，旧的 checkpoint 产生了严重的语义不兼容，如果强行引入 resume，模型只会继续挺尸趴地。

> ### 🛑 恶性 Bug 档案：TensorBoard 模仿分狂飙，play.py 现场挺尸
> 
> 
> * **灾难表象**：模型高频训练至 `iter 7461` 阶段，从 TensorBoard 看，上肢出拳追踪得分 `track_punch_arms` 高达 `+6.1`，下肢姿态复现得分 `track_punch_legs` 达到了华丽的 `+7.3`。但在 `play.py` 离线渲染中，机器人一诞生便瞬间摔死，随后整过回合就像一滩软泥一样平躺在地面上，极为精准地根据 CSV 时间戳挥舞四肢。
> 
> 
> * **底层因果解剖**：这是经典的**模仿分高过生存惩罚**引发的智能体欺骗。策略网络偷懒发现了漏洞：比起辛辛苦苦对抗重力去维持 ZMP 平衡站立跳舞，直接开局躺平在绝对安全的地板上，完全避开倾倒惩罚，反而能以极低能耗把 15.0 权重的手臂模仿奖励给全额刷满。这导致整体姿态偏离项 `flat_orientation` 恶化至 `-0.79`，且高达 `0.29` 频率的骨盆躯干撞地终止率（`base_contact`）被网络完全无视。
> 
> 
> * **终极破局手段**：强行引入 `upright` 躯干竖直检测门控 + 物理防摔重拳镇压 `flat_orientation_l2` 权重拉满至 `-8.0` + 底盘离地垂向高度限制 `base_height` 惩罚拉满至 `-4.0`。在完成全量重构后，彻底清理旧权重，从第 0 代（iter 0）重新开炉重训。
>

## 第 1 章：`feat/punch` 分支隔离与 `punch_env_cfg` 中枢调配

`punch_env_cfg.py` 文件是整个原地舞蹈/打拳训练任务的控制大本营，发令枪速度范围、奖励惩罚系数、外部 `DANCE_CSV` 动作源寻址以及高维观测项全量挂载在内部进行编排。

### 1.1 与 master 行走任务的物理隔离规范

严禁直接在 master 行走配置代码上修改任何属于打拳模仿的奖励函数，更不要尝试用行走的 checkpoint 去初始化高肢体协作的舞蹈策略。

* **古老资产封存规程**：在每一次大幅度调整权重前，必须将当前的配置文件和函数副本另存重命名为 `punch_env_cfg_old.py` 和 `rewards_old.py`，确保随时能够通过 `diff` 查数并无损回滚。

* **另存路径纪律（chat 踩坑）**：写整份 `punch_env_cfg.py` 时，必须保存到 **`config/s42/` 或 `config/s49/` 下且文件名为 `punch_env_cfg.py`**——**严禁覆盖** `rough_env_cfg.py` 等 master 行走配置；文件头注释若仍写 `rough_env_cfg.py` 只是模板残留，以**磁盘路径**为准。



### 1.2 速度发令枪死锁（原地控制假设）

在 `CommandsCfg` 类中，我们将线速度与角速度指令范围全部强制重写为 `0`：

```python
# 🔴 舞蹈/打拳模式：锁死全向运动范围，禁止任何位移导致的吊绳拉紧
ranges=mdp.UniformVelocityCommandCfg.Ranges(
    lin_vel_x=(0.0, 0.0),
    lin_vel_y=(0.0, 0.0),
    ang_vel_z=(0.0, 0.0),
    ...
)

```

这与基础行走任务中考官高频对速度 `vx/vy/ω` 进行大范围随机采样的特征形成了鲜明的对比。

### 1.3 奖惩系数矩阵的进化历程对比

从早期的过渡调练阶段，到最终的 S49 满血闭环，奖励函数的权重配比经历了大幅度的强度加码：

| 核心控制项名称 | S42 第二阶段过渡期权重 | S49 满血闭环最终推荐权重 | 物理级调参因果直觉 |
| --- | --- | --- | --- |
| **base_lin_vel_xy_stationary** | `weight = -1.3` | `weight = -1.3` | 严厉惩罚根节点 XY 轴线速度，强行压制出拳位移导致的真机向前平移蹭步。 |
| **base_ang_vel_yaw_stationary** | `weight = -0.3` | `weight = -0.3` | 惩罚原地自旋偏航，对抗编舞不对称动量引起的偏航平移漂移。 |
| **track_punch_arms** | `weight = 15.0` | 升级为 **Upright 门控版本** | 原版直接计算 $L2$ 误差会诱发躺平；S49 强制唯有躯干直立才发放此正分。 |
| **track_punch_legs** | `weight = 9.0` | 升级为 **Upright 门控版本** | 逼迫腿部配合上肢律动，同样与躯干是否直立进行硬性挂钩。 |
| **flat_orientation_l2** | `weight = -1.0` | `weight = -8.0` | ❌ 旧版惩罚太轻导致机器人直接躺下；新版加大 8 倍，将其列为生命第一红线。 |
| **base_height** | 未配置（缺失状态） | `weight = -4.0` | S49 专属新增：以 `0.85m` 为目标高度，一旦机器人过度下蹲或塌陷，扣分雪崩式暴涨。 |

* **DANCE_CSV 寻址路径映射**：S42 阶段读取 `FULLBODY_INPLACE` 或 `PERFECT_LIMIT` 的简短文件；统一 S49 阶段后由变量全量指向长达 1565 帧的五代机跨代平滑重采样波形文件 `kuavo_action_S49_FROM_S54_INPLACE_RAD.csv`。

* **Action 动作网络输出缩放增益 (`actionScale`)**：参数卡与网络必须死锁在统一的 **`0.30`** 弧度限幅尺度上，两端错位将导致电机出力异常。

### 1.4 从行走 baseline fork 时必须清零的「反向锁臂」惩罚（chat 经典坑）

从 master / 行走 `rough_env_cfg.py` 改出舞蹈任务时，以下 walking 专用项若未关闭，TensorBoard 可能仍好看，但 **`play.py` 里手臂几乎不动或只会微抖**：

| 项 | 推荐处理 | 原因 |
| --- | --- | --- |
| **`stand_still_without_cmd`** | **`weight = 0.0`** | 无速度指令时惩罚一切关节动——与挥拳矛盾 |
| **`joint_deviation_arms`** | **注释掉或 `weight = 0.0`** | 惩罚 `zarm_*` 偏离默认角，直接锁死大臂 |
| **`fft_dof_symmetry` 等对称惩罚** | 舞蹈链通常 **关闭** | 左右交替出拳天然不对称 |

* **`preserve_order=True`**：凡 `joint_pos` / `joint_vel` ObsTerm 与 Action 白名单必须加此参数，否则 CSV 列与 Lab 序错位（详见 **23.2** §2.1.1）。

## 第 2 章：`rewards.py` 显式模仿跟踪奖励体系与时间相位绑定

### 2.1 动作相位的 sin/cos 高频时间轴绑定

在模仿学习中，机器人绝对不能使用固定 1.2 秒的经典行走循环 gait 周期。程序通过当前环境机器人各自独特的仿真更新步数累加器 `episode_length_buf` 与 CSV 文件内部包含的总行数 `num_frames` 进行取余计算，将其转换为处于 $[0,1]$ 之间的连续时间相位 `phase`，并使用 sin/cos 算子映射为 2 维的连续周期性输入：

```python
# 🟢 严格的时间相位查表绑定公式
frame_indices = (env.episode_length_buf % num_frames).long()
phase = frame_indices.float() / num_frames
return [torch.sin(2 * np.pi * phase), torch.cos(2 * np.pi * phase)]

```

* **帧数与时间维度的严格绑定**：642 帧版本的完美限制文件其时间轴长为 $642 \div 50 = 12.84\text{s}$；而 S49 满血舞蹈 CSV 包含 1565 帧，长为 $1565 \div 50 = 31.30\text{s}$。**变量 `num_frames` 必须雷打不动地严格等于你加载的 CSV 文件的实际行数**。

* **空间索引红线**：在标准的 89 维网络架构中，`action_phase` 被硬性限定为**最后多出来的 2 维（即处于第 87 和 88 位索引）**。在导出端 `play.py` 中如果因为代码逻辑错误将其误插到了网络中间，将构成毁灭性的 P0 级灾难 Bug。

### 2.2 核心指数核模仿函数与 TensorBoard 读数诊断

双肢追踪的核心数学核函数为：$R = \exp\left(-\frac{\|\mathbf{q} - \mathbf{q}_{\text{ref}}\|^2}{\sigma^2}\right)$，其中标准差系数 $\sigma$ 决定了网络对动作误差的物理容忍度。

* **曲线读数在 0.3~0.5 之间震荡的物理含义**：这代表智能体在环境中正在进行极其保守、小幅度的局部安全跟踪。这绝不是因为磁盘 I/O 读取速度慢造成的瓶颈，而是因为网络在生存惩罚与打拳得分之间进行了理性的数学妥协（trade-off）。

* **曲线读数逼近 1.0 满分的物理含义**：代表机器人的物理关节角度几乎完美无瑕地与 CSV 参考轨迹达成了空间重合。但注意：**绝对不能盲目单看这一项高分**，必须同时核对 `flat_orientation` 是否接近于 0。

* **确诊“假高分”诈骗**：如果 TensorBoard 面板上模仿分极高（$>0.8$），但同时倾倒惩罚 `flat_orientation` 极差、且躯干撞地终止率 `base_contact` 居高不下，100% 确诊机器人已经陷入了“躺倒挺尸、在地上挥舞四肢”的局部最优解泥潭。

### 2.3 降维打击“躺平漏洞”的 Upright 姿态门控大坝

为了彻底切断智能体通过躺下摆动四肢来刷分作弊的捷径，我们在 S49 环境中实装了强制的物理拦截门控：

```python
def _upright_factor(asset, min_upright=0.75):
    # 提取机器人体坐标系下的投影重力 Z 轴分量（直立时应接近 -1）
    z = -asset.data.projected_gravity_b[:, 2]
    return torch.clamp((z - min_upright) / (1.0 - min_upright), min=0.0, max=1.0)

def track_punch_arms_trajectory_upright_exp(...):
    # 唯有当躯干倾角在安全门限内，upright_factor 大于 0 时，才发放模仿追踪正分
    track = _track_joint_names_exp(...)
    return track * _upright_factor(asset, min_upright)

```

* **参数微调红线**：`min_upright` 默认阈值标定为 `0.75`。如果在实际重训中发现由于上肢出力过猛机身依然出现大面积倾倒，可将该门限死锁抬升至极度严格的 `0.85`，并配合 `-8.0` 的重型防摔权重联合出手镇压。

## 第 3 章：突破“记忆瓶颈”，从 89 维跨越至 115 维高维观测空间扩展

### 3.1 原始 89 维基础特征向量的极限与局限性

在早期的开发中，输入网络的 89 维长向量由以下感官数据连续拼接而成：

$$\text{base\_ang\_vel}(3) + \text{projected\_gravity}(3) + \text{velocity\_commands}(3) + \text{joint\_pos\_rel}(26) + \text{joint\_vel\_rel}(26) + \text{last\_action}(26) + \text{action\_phase}(2) = 89$$

* **致命的局限性**：在这种架构下，神经网络的大脑等于只依靠最后 2 维区区两个 sin/cos 数字，就必须强行在脑回路里硬生生“死记硬背”住一整段长达几十秒、包含了 26 个关节空间错综复杂的动作乐谱。这在简单的上肢单兵出拳任务中勉强可行，但面对全尺寸的全身长编舞，探索难度无异于大海捞针，网络会迅速崩溃。

### 3.2 引入“开卷抄答案”机制：新增 26 维 `reference_joint_pos`

为了给网络减负，我们实施了重大的扩维手术：我们在观测空间里强行挂载了多达 26 维的**目标期望关节角度相对差（`reference_joint_pos`）**。

* **物理逻辑变更**：通过这行代码，我们在每一个控制帧，把当前时间戳下 CSV 表格里写死的机器人各关节目标期望角度，减去默认的静态站立角度，直接当作特征组塞入网络的嘴里。智能体从原本痛苦的“死记硬背时间轴”，降维进化为了“直接看着答案抄当前帧的最优动作”，网络探索难度瞬间呈指数级断崖式暴降！

```python
# 🟢 115维满血打拳观测核心注入项
reference_joint_pos = ObsTerm(
    func=local_rewards.reference_joint_pos_rel,
    params={"csv_path": DANCE_CSV},
)

```

* **张量大闭环校验**：扩展后的总维度精确核算为：$3 + 3 + 3 + 26 + 26 + 26 + 2 + 26 = 115$ 维。**配置参数卡里的 `numSingleObs` 必须无条件与导出的 ONNX 模型输入的第一个 Dimension 进行严格相等的硬件死锁**！

### 3.3 115 维全量观测特征空间切片顺序与对齐寻址主表

在重新编译并拉起 MuJoCo 仿真后，如果系统发生畸变，请使用命令 `rostopic echo /rl_controller/singleInputData -n 1` 打印出下位机组装的真实数据，并对照本表进行逐块、逐维的物理位置查数排错：

| 特征块名称 | 占据空间维度 | 数组绝对起始偏移量 | 训练侧缩放系数 (Scale) | C++ 部署端映射对应的 Info 核心参数键名 |
| --- | --- | --- | --- | --- |
| **1. base_ang_vel** | 3 维 | `Offset: 0` | `scale = 1.0` | 严格匹配并查找 `bodyAngVel` 字典结构体 |
| **2. projected_gravity** | 3 维 | `Offset: 3` | `scale = 1.0` | 严格匹配并查找 `gravity_body` 字典结构体 |

### 4.2 终结 `dof_pos_illegal 57.79`：重构 `DelayedPDActuator_S49` 纯空间阻尼执行器

这是导致新旧环境交替时开局第 1 步闪退的头号杀手。旧环境自带的 S42 执行器算子内部，包含了一套高度定制的 `joint_to_motor_position` 连杆并联脚踝力学解算器，且其逆向转换参数是死死针对老旧的 S46 USD 模型进行硬编码标定的。

* **发头发病根因**：当我们把这一套属于 S42 的连杆并联矩阵公式直接套在几何外形和碰撞厚度发生跨代异变的新一代 S49 模型双脚上时，底层的数学推导直接发生除零发散，计算出了一串全身乱码和全是空的 **NaN 电机位置期望**。这导致物理引擎在运行的第 0 个时间步（iter 0）就会瞬间拉响警报，抛出 `dof_pos_illegal 57.79` 并强行中断仿真。

* **解药修复**：我们彻底废掉了 `actuators/actuator_pd.py` 里复杂的连杆踝解算组件，将其强行退化改造为与 MuJoCo 部署端旁路（bypass） `ankleSolver` 思想完全一致的**纯关节空间绝对 PD 控制器**：

$$\text{computed\_effort} = K_p \cdot e_{\text{pos}} + K_d \cdot e_{\text{vel}} + \tau_{\text{ff}} - \text{friction} \text{[cite: 18]}$$

* **力矩上限的等效置换**：为了在大军开局探索时给予机器人强大的下盘抗震能力，我们在 `kuavo.py` 内部的 `Kuavos49_CFG` 类中暂时上调了电机的发力上限（例如将侧摆髋关节 `leg_l2` 的力矩极限从真实的 22.2 N·m 强行暂时放宽到 **`100 N·m`**），以此换取开局不会因为剧烈出拳而瞬间散架的稳定性；而真实的出力限制和过载保护，全量留给部署端的参数卡去严格把控，达成完美解耦。

## 第 6 章：PPO 并行训练参数矩阵与安全防爆盾点火

在扫清了上述所有 import 崩溃和维度发散地雷后，我们终于可以在 Bash 环境下开炉训练专属于 Kuavo 4 Pro 的原地安全打拳大模型策略了。

### 6.1 黄金超参训练矩阵卡对齐

为了防止由于超参设置不当导致模型陷入“躺平模仿”的局部最优解泥潭，我们在 `config/s49/agents/rsl_rl_ppo_cfg.py` 中对 PPO 算法核心权重进行了针对打拳任务的定向微调：

* **`num_steps_per_env = 24`**：每个环境在每轮迭代更新前，连续采集 24 步样本（在 50Hz 频率下折合单次连续收集 0.48 秒的运动轨迹）。这能确保大军在每次 rollout 更新时，其视线能够精准覆盖并内化一段完整的出拳加速和收拳片段。

* **`learning_rate = 1e-3` 自适应调度**：网络开局使用较大步长在空间中疯狂探索，如果在 iter 500 后发现策略出现剧烈共振晃动，可将其手动降频锁定在更平稳的 `5e-4` 稳健中线。

* **`clip_param = 0.2` 策略裁切约束**：严格限制新旧策略之间分布的更新幅度。因为机器人大幅度挥臂出拳时，网络动作输出（Action）的分布极其宽广，极小的力矩突变都会在倒立摆模型上产生巨大的重心震荡，0.2 的裁剪线能死死阻断由于网络突然更新导致双腿直接踩入劈叉死区的悲剧。

* **`entropy_coef = 0.01` 探索熵系数**：这是防止网络“过早躺平”的核心保险丝。当打拳的 `mimic` 奖励在后期占据绝对主导地位时，由于其具有强大的诱惑性，网络的动作熵会自杀式暴降并快速收敛。保持 0.01 的熵奖励，能强制策略网络保持在原地“多想多试”，防止其因为一时的贪婪直接陷入“趴在地上摆臂拿高分”的垃圾局部最优解黑洞。

* **`gamma = 0.99` / `lam = 0.95`（GAE广义优势估计）值**：由于打拳和全身舞属于长达几百个时间步（约 860 步至 1565 步）的长周期 Episode 任务，设置高打折因子（0.99）能迫使当前的优势函数（Advantage）在计算回报时，必须深度、长远地把远期可能发生前栽或侧倒的姿态大惩罚分给严格地 discount 融合折算计算进来，让它具备真正的“居安思危”平衡意识。

### 6.2 训练用力矩上限与 URDF 真实硬件上限的等效置换矩阵

为了在仿真世界里保护极其脆弱的电机，并实现极高的平衡探索成功率，我们对 26 个关节的力矩做了一套**训练放大、部署限幅**的置换对齐矩阵。这能防止网络因为开局扭矩过小站不稳而直接全量放弃：

| 关节空间英文官方定名 | 机器人 URDF 资产标注的真实额体力矩上限 | 我们的 `kuavo.py` 训练包中暂用等效力矩上限 | 深度动力学因果调参直觉（为什么必须放宽） |
| --- | --- | --- | --- |
| **1. leg_l1_joint / leg_r1_joint** | 真实额体：`127 N·m` | 训练放宽至：`180 N·m` | 负责盆骨髋关节 Pitch 轴。在机器人挥拳手臂探出的一瞬间，此处需要倾注极大的扭矩向后拉扯机身，放宽到 180 能极大换取探索初期的稳定性。 |
| **2. leg_l2_joint / leg_r2_joint** | ❌ 真实额体：**`22.2 N·m`** | 训练放宽至：**`100 N·m`** | **重型防炸机死穴点**：原厂硬件为了控制下肢质量，将髋 Roll 轴电机设得极其孱弱（22.2）。这导致在仿真 iter 0 刚出生时，双腿由于无法抗衡侧向重力直接向两侧劈叉瘫软。我们在训练包里强行放宽 5 倍给它塞入 100 刚性，让它先站稳学会跳舞；而在真机下发参数卡 `skw_rl_param_dance.info` 里，我们再改回真实限幅，走小幅度的柔顺自平衡，实现 Sim2Real 软硬件安全解耦。 |
| **3. leg_l4_joint / leg_r4_joint** | 真实额体：`135 N·m` | 训练放宽至：`180 N·m` | 负责最沉重的膝关节驱动。舞蹈和打拳需要高频进行“屈膝-直立”的全身律动，必须放宽上限确保力矩在运动中不会发生恶性饱和截断。 |
| **4. zarm_l1_joint / zarm_r1_joint** | 真实额体：`50 N·m` | 训练放宽至：`80 N·m` | 负责大肢体肩关节 Pitch 轴。手臂向前猛烈爆发冲拳时，力矩如果发生过早饱和，挥拳轨迹在物理层面上会被无情“削平”，导致模型学不会出拳，必须在此处放开探索上限。 |

### 7.3 Checkpoint 黄金窗口选取（避免 6000/9000 梯度黄昏期）

结合 S42 长训与 S49 重训经验，**最大 iter 的 `.pt` 不一定是最好 `.pt`**：

| 阶段 | 典型 iter | 特征 | 导出建议 |
| --- | --- | --- | --- |
| **黄金窗口** | **2500–3000** | mimic 高、`flat_orientation` 已回升、`base_contact≈0` | **优先 `play.py` 验收** |
| **稳定平台** | 4000–6000 | 下肢极稳，动作可能变「卡死」 | 与 3000 对比后再定 |
| **黄昏/污染** | 接近 **inf/NaN 爆炸** 前 | 动作饱和高频抽搐 | **禁用**；回退 3000 |
| **S49 参考** | 按 TensorBoard 实际峰值 | 勿默认 `model_9000.pt` | 见 **23.6** §2.2 |

* **肉眼比对**：对 2500 / 3000 / 6000 等候选 checkpoint 各跑一轮 **`play.py --video`**（**23.6** §1.4.2），站立挥拳正常再导出 ONNX。

* **早停策略**：`Episode_Reward` 平台期 + `entropy` 骤降 + play 变僵硬 → 直接取前期 checkpoint 导出，不必跑满 `max_iterations`。
* **梯度爆炸**：终端出现 **inf/NaN** 时，**爆炸前最后一个正常 `model_*.pt`** 往往仍可用（chat 中 6395 iter 奖励 ~19 分仍站立的案例）。
