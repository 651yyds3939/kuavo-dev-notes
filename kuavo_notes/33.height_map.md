## 1. 系统架构

### 1.1 整体数据流

```
Livox Mid-360 雷达
       │
       ▼
┌──────────────────┐    /livox/lidar     ┌─────────────────────┐    /livox/pointcloud2    ┌────────────────────────┐
│ livox_ros_driver2 │ ──────────────────▶ │  livox2pointcloud   │ ──────────────────────▶ │   elevation_mapping    │
│ (livox_frame)     │    (CustomMsg)      │  (livox_frame)      │    (sensor_msgs::PC2)   │   (map frame)          │
└──────────────────┘                      └─────────────────────┘                         │                        │
         │                                                                                │  输出:                 │
         │ /livox/imu                                                                     │  /elevation_mapping/   │
         ▼                                                                                │    elevation_map_raw   │
┌──────────────────┐    /Odometry          ┌─────────────────────┐   /localization        │  (grid_map_msgs/       │
│     FAST-LIO     │ ────────────────────▶ │  transform_fusion   │ ──────────────────────▶│    GridMap)            │
│  (laserMapping)  │   (camera_init→body)  │  + global_localization                      └────────────────────────┘
└──────────────────┘                       └─────────────────────┘
```

### 1.2 依赖关系

| 包名 | 来源 | 用途 |
|------|------|------|
| `elevation_mapping` | [ANYbotics/elevation_mapping](https://github.com/ANYbotics/elevation_mapping) | 核心高程图算法 |
| `grid_map` | [leggedrobotics/grid_map](https://github.com/leggedrobotics/grid_map) | 网格地图底层库 |
| `kindr` | [ANYbotics/kindr](https://github.com/ANYbotics/kindr) | 运动学/旋转库 |
| `kindr_ros` | [ANYbotics/kindr_ros](https://github.com/ANYbotics/kindr_ros) | kindr 的 ROS 接口 |
| `message_logger` | [ANYbotics/message_logger](https://github.com/ANYbotics/message_logger) | 日志工具 |
| `livox_ros_driver2` | Livox SDK | Livox 雷达驱动 |
| `FAST_LIO` | [hku-mars/FAST_LIO](https://github.com/hku-mars/FAST_LIO) | LiDAR-惯性里程计 |
| `livox2pointcloud` | 本地开发 | Livox 自定义格式 → 标准 PointCloud2 转换 |

## 2. 初始化工作（第一阶段）

### 2.1 已创建的配置文件

在 `kuavo_nav_demo` 功能包中创建了 elevation_mapping 所需的完整配置。

#### 文件目录结构

```
src/navigation_opensource/src/kuavo_nav_demo/
├── launch/
│   └── elevation_mapping.launch          # 主 Launch 文件
├── config/
│   ├── robots/
│   │   └── kuavo.yaml                    # 机器人参数（坐标系、输入源）
│   ├── elevation_maps/
│   │   └── map.yaml                      # 高程图网格参数
│   └── sensor_processors/
│       └── livox.yaml                    # Livox 噪声模型
└── scripts/
    └── odometry_to_pose_with_covariance.py  # 里程计格式转换
```

#### 各文件说明

##### 2.1.1 `launch/elevation_mapping.launch`（原始版本）

- 启动 ANYbotics `elevation_mapping` 节点（可执行文件名为 `elevation_mapping`）
- 加载三个 YAML 配置
- 支持 Launch 参数：
  - `odom_topic`（默认 `/localization`）：全局定位
  - `pointcloud_topic`（默认 `/livox/pointcloud2`）
  - `enable_rviz`（默认 `false`）

##### 2.1.2 `config/robots/kuavo.yaml`（原始版本）

```yaml
map_frame_id: map
robot_base_frame_id: body
input_sources:
  lidar:
    type: pointcloud
    topic: /livox/pointcloud2
    queue_size: 10
track_point_frame_id: body
```

##### 2.1.3 `config/elevation_maps/map.yaml`（原始版本）

```yaml
length_in_x: 4.0
length_in_y: 4.0
resolution: 0.04         # 4cm 分辨率
min_update_rate: 2.0
sensor_processor/ignore_points_above: 1.5   # 过滤头顶以上 1.5m
sensor_processor/ignore_points_below: -1.0  # 过滤脚下 1m
```

##### 2.1.4 `config/sensor_processors/livox.yaml`（原始版本）

```yaml
sensor_processor:
  type: laser
  min_radius: 0.05
  beam_angle: 0.003
  beam_constant: 0.002
```

##### 2.1.5 `scripts/odometry_to_pose_with_covariance.py`

ETH `elevation_mapping`（ROS 1）订阅的是 **`geometry_msgs/PoseWithCovarianceStamped`**，不能直接使用 `nav_msgs/Odometry` 类型的 `/localization` 或 `/Odometry`。此脚本完成格式转换。

### 2.2 依赖包的预置

将以下依赖 clone 到工作空间内：

```bash
cd ~/kuavo_ros_application/src/navigation_opensource/src
git clone https://github.com/ANYbotics/elevation_mapping.git
git clone https://github.com/ANYbotics/kindr.git
git clone https://github.com/ANYbotics/kindr_ros.git
git clone https://github.com/ANYbotics/message_logger.git
git clone https://github.com/leggedrobotics/grid_map.git
```

## 3. 问题排查与定位（第二阶段）

### 3.1 逐层排查过程

#### 排查 1：elevation_mapping 是否已编译？

```bash
$ ls ~/kuavo_ros_application/devel/lib/elevation_mapping/
# 结果：不存在

$ rosmsg show grid_map_msgs/GridMap
# 结果：unknown package [grid_map_msgs]
```

**诊断**：❌ **elevation_mapping 从未编译。** 依赖包虽然已 clone，但未执行 `catkin build`。

---

#### 排查 2：工作空间是否正确 source？

```bash
$ echo $ROS_PACKAGE_PATH | tr ':' '\n' | grep kuavo_nav
# 结果：空

$ roslaunch kuavo_nav_demo kuavo_nav_test.launch
# 结果：RLException: [kuavo_nav_test.launch] is neither a launch file in package [kuavo_nav_demo] ...
```

**诊断**：❌ **`navigation_opensource` 是独立 catkin 工作空间**，需要单独 source 其 `devel/setup.bash`。

工作空间层级结构：
```
~/kuavo_ros_application/                    ← 主工作空间（机器人控制包）
└── src/navigation_opensource/              ← 独立子工作空间（导航/雷达/mapping 包）
    └── src/
        ├── kuavo_nav_demo/
        ├── elevation_mapping/
        ├── grid_map/
        ├── FAST_LIO/
        └── ...
```

---

#### 排查 3：点云数据是否到达？

```bash
$ rostopic hz /livox/pointcloud2
# 结果：no new messages

$ rostopic list | grep livox
/livox/lidar     # 自定义格式 CustomMsg，有数据 (10Hz)
/livox/imu       # 有数据 (10Hz)
/livox/pointcloud2  # 话题存在但无数据！
```

**诊断**：❌ **`livox2pointcloud` 转换节点未启动。** Livox 驱动发布的是自定义 `livox_ros_driver2/CustomMsg` 格式，而 elevation_mapping 需要标准 `sensor_msgs/PointCloud2`。`livox2pointcloud` 节点负责此转换，但没有被启动。

---

#### 排查 4：坐标系是否正确？

**当前运行的 TF 树**（仅机器人控制栈）：
```
odom
├── mm/world (static, Z=0.81)
├── base_link (dynamic, from EKF/robot_state_publisher)
│   └── (URDF 关节连杆: leg_l*, leg_r*, zarm_l*, zarm_r*, zhead_*)
└── dummy_link (dynamic)
    └── base_link (static, identity)
```

**elevation_mapping 配置期望的 TF 树**（完整导航栈）：
```
map
└── camera_init (dynamic, from transform_fusion.py)
    └── body (dynamic, from FAST-LIO /Odometry)
        └── livox_frame (static, identity, from kuavo_localize.launch)
```

**诊断**：⚠️ **坐标系不匹配。**

| 配置项 | 配置值 | 当前实际值 | 状态 |
|--------|--------|-----------|------|
| `map_frame_id` | `map` | `odom` (无 `map` 帧) | ❌ |
| `robot_base_frame_id` | `body` | `base_link` (无 `body` 帧) | ❌ |
| `track_point_frame_id` | `body` | 不存在 | ❌ |
| 点云 frame_id | `livox_frame` | `livox_frame` (无 TF 连接) | ⚠️ |

**重要发现**：当前只有机器人控制栈在运行。完整的导航感知栈（Livox + FAST-LIO + transform_fusion）需要启动 `kuavo_nav_test.launch` 才会发布 `map → body → livox_frame` 的 TF 链和 `/localization` 话题。

---

### 3.2 根因汇总

| # | 问题 | 严重程度 | 影响 |
|---|------|---------|------|
| 1 | elevation_mapping 未编译 | 🔴 致命 | 节点无法启动 |

## 4. 修复与优化（第三阶段）

### 4.1 Step 1：编译

```bash
cd ~/kuavo_ros_application/src/navigation_opensource
catkin build elevation_mapping grid_map kindr kindr_ros message_logger kuavo_nav_demo -j2
```

> ⚠️ 注意：此机器内存有限，编译大型 C++ 包（尤其是 grid_map）时需加 `-j2` 限制并行数，否则会 OOM。

编译产物验证：
```bash
$ ls devel/lib/elevation_mapping/
elevation_mapping    # ✅

$ rosmsg show grid_map_msgs/GridMap
grid_map_msgs/GridMapInfo info    # ✅
```

### 4.2 Step 2：修正坐标系配置

#### 4.2.1 修改 `kuavo.yaml`

**改动**：移除了硬编码的 `map_frame_id`、`robot_base_frame_id`、`track_point_frame_id` 和 `input_sources/lidar/topic`。这些参数改为由 launch 文件通过 `<arg>` 动态注入，实现多场景适配。

**修改后的 `config/robots/kuavo.yaml`**：
```yaml
# Kuavo humanoid robot parameters for ETH elevation_mapping.
#
# Frame IDs and input source topic are set via launch args
# in elevation_mapping.launch, NOT hardcoded here.

robot_pose_with_covariance_topic: /elevation_mapping/pose
robot_pose_cache_size: 200

track_point_x: 0.0
track_point_y: 0.0
track_point_z: 0.0

input_sources:
  lidar:
    type: pointcloud
    queue_size: 10
    publish_on_update: true
```

#### 4.2.2 传感器配置归位

**改动**：`ignore_points_above` 和 `ignore_points_below` 从 `map.yaml`（高程图参数）移至 `livox.yaml`（传感器参数）。点云过滤属于传感器属性，应放在传感器配置命名空间下。

**修改后的 `config/sensor_processors/livox.yaml`**：
```yaml
sensor_processor:
  type: laser
  min_radius: 0.05
  beam_angle: 0.003
  beam_constant: 0.002

  # Filter points far above/below the robot torso (body/base_link frame).
  # Above: suppress ceiling, head, and arm returns.
  # Below: keep ground points down to ~1m below the LiDAR.
  ignore_points_above: 1.5
  ignore_points_below: -1.0
```

### 4.3 Step 3：自包含化 Launch 文件

**改动**：重写 `elevation_mapping.launch`，实现以下功能：

1. **内置静态 TF 发布**：自动发布 `<robot_frame> → livox_frame`（单位变换），不再依赖 `kuavo_localize.launch`
2. **Launch args 覆盖机制**：`<param>` 放在 `<rosparam load>` **之后**，确保 args 覆盖 YAML 默认值
3. **支持 4 种运行模式**：

| 模式 | 命令 | map_frame | robot_frame | odom_topic |
|------|------|-----------|-------------|------------|
| A) 完整导航栈 | `roslaunch kuavo_nav_demo elevation_mapping.launch` | `map` | `body` | `/localization` |
| B) FAST-LIO 里程计 | `...odom_topic:=/Odometry map_frame:=camera_init` | `camera_init` | `body` | `/Odometry` |
| C) EKF 里程计 | `...odom_topic:=/odometry/filtered map_frame:=odom robot_frame:=base_link` | `odom` | `base_link` | `/odometry/filtered` |
| D) FAST-LIO 配准点云 | `...pointcloud_topic:=/cloud_registered_body` | `map` | `body` | `/localization` |

**最终 `launch/elevation_mapping.launch`**：参见 [附录 A](#附录-a-elevation_mappinglaunch-完整源码)。

### 4.4 Step 4：验证与修复

#### 4.4.1 启动 livox2pointcloud 转换

```bash
roslaunch livox2pointcloud livox2pointcloud.launch
```

验证：

## 6. 使用指南

### 6.1 环境准备

**⚠️ 每次新终端必须先 source 两个工作空间**：

```bash
source ~/kuavo_ros_application/devel/setup.bash
source ~/kuavo_ros_application/src/navigation_opensource/devel/setup.bash
```

> 建议将上述命令添加到 `~/.bashrc` 末尾，避免每次手动执行。

### 6.2 启动方式

#### 模式 A：完整导航栈（推荐）

> **注意**：`kuavo_nav_test.launch` **不包含** `livox2pointcloud` 转换节点。默认点云话题 `/livox/pointcloud2` 需额外启动转换，否则 elevation_mapping 收不到点云。

```bash
# 终端 0：每个新终端都要 source
source ~/kuavo_ros_application/devel/setup.bash
source ~/kuavo_ros_application/src/navigation_opensource/devel/setup.bash

# 终端 1：雷达 + FAST-LIO 定位 + move_base
roslaunch kuavo_nav_demo kuavo_nav_test.launch

# 终端 1.5：Livox CustomMsg → PointCloud2 转换（模式 A 必须）
roslaunch livox2pointcloud livox2pointcloud.launch

# 确认点云有数据
rostopic hz /livox/pointcloud2          # 应 ~10 Hz
rostopic hz /localization               # 应有输出

# 终端 2：高程图
roslaunch kuavo_nav_demo elevation_mapping.launch

# 终端 3：确认高程图输出（比 RViz 更直接）
rostopic hz /elevation_mapping/elevation_map_raw   # 应 ~5 Hz
```

#### 模式 B：仅 FAST-LIO（无全局定位）

```bash
roslaunch kuavo_nav_demo elevation_mapping.launch \
    odom_topic:=/Odometry \
    map_frame:=camera_init
```

#### 模式 C：仅 EKF 里程计（无 FAST-LIO）

```bash
roslaunch kuavo_nav_demo elevation_mapping.launch \
    odom_topic:=/odometry/filtered \
    map_frame:=odom \
    robot_frame:=base_link
```

#### 模式 D：使用 FAST-LIO 配准点云（body 系，已去畸变）

> **⚠️ 前置条件（缺一不可）**
>
> 1. **必须先运行** `kuavo_nav_test.launch`（或至少 FAST-LIO + 定位），单独起 `elevation_mapping.launch` 无效
> 2. **`/cloud_registered_body` 必须有数据**：由 FAST-LIO 的 `laserMapping` 节点发布，频率约 10 Hz
> 3. **不需要** `livox2pointcloud`（点云已由 FAST-LIO 输出为标准 PointCloud2）
> 4. RViz 需手动 Add **GridMap** 显示，**不能**使用 `localization.rviz`（该配置不含高程图）

```bash
# 终端 1：雷达 + FAST-LIO + 定位（必须）
roslaunch kuavo_nav_demo kuavo_nav_test.launch

# 确认 FAST-LIO 配准点云有数据（没数据就不要起 elevation_mapping）
rostopic hz /cloud_registered_body        # 应 ~10 Hz，若 no new messages 说明 FAST-LIO 未运行
rostopic hz /Odometry                     # FAST-LIO 里程计
rostopic hz /localization                 # 全局定位

# 终端 2：高程图（模式 D）
roslaunch kuavo_nav_demo elevation_mapping.launch \
    pointcloud_topic:=/cloud_registered_body

# 终端 3：确认高程图输出
rostopic hz /elevation_mapping/elevation_map_raw   # 应 ~5 Hz
```

**模式 A vs 模式 D 对比**：

| 对比项 | 模式 A（默认） | 模式 D |
|--------|---------------|--------|
| 点云话题 | `/livox/pointcloud2` | `/cloud_registered_body` |
| 点云坐标系 | `livox_frame` | `body`（已去畸变） |
| 需 livox2pointcloud | ✅ 必须单独启动 | ❌ 不需要 |
| 需 FAST-LIO | ✅ | ✅ |
| 推荐场景 | 通用、链路简单 | 与导航 2D 激光同源，运动补偿更好 |

### 6.3 在 RViz 中查看高程图

> **常见误区**：打开 `localization.rviz` 或 FAST-LIO 自带的 RViz 配置**看不到高程图**——那些配置只订阅定位/点云话题，没有 GridMap 显示项。高程图必须手动添加 GridMap 插件。

1. 确保 RViz 所在终端已 source 两个工作空间（见 §6.1）
2. 先用命令确认有数据，再开 RViz：
   ```bash
   rostopic hz /elevation_mapping/elevation_map_raw   # 应有 ~5 Hz
   ```
3. 启动 RViz（不要用 `localization.rviz`）：
   ```bash
   rviz
   ```
4. 点击左下角 **「Add」** 按钮
5. 切换至 **「By display type」** 标签页
6. 在列表中寻找 **`GridMap`**（若找不到，说明未 source `navigation_opensource` 或未编译 `grid_map_rviz_plugin`）
7. 选中，点击 **OK**
8. 展开左侧 Displays 面板中的 `GridMap`：
   - **Topic**：选择 `/elevation_mapping/elevation_map_raw`
   - **Layer**：选择 `elevation`
9. 高程图将以彩色显示在 3D 视图中

> 便捷方式：如果 elevation_mapping 节点正在发布，可直接在 「Add → By topic」标签下选择 `/elevation_mapping/elevation_map_raw`，RViz 会自动创建 GridMap 显示。

> **`enable_rviz:=true` 暂不可用**：launch 引用的 `rviz/elevation_mapping.rviz` 尚未创建，请按上述步骤手动配置。

### 6.4 Launch 参数参考

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `odom_topic` | `/localization` | 里程计话题 (nav_msgs/Odometry) |
| `pointcloud_topic` | `/livox/pointcloud2` | 点云话题 (sensor_msgs/PointCloud2) |
| `map_frame` | `map` | 全局参考坐标系 |
| `robot_frame` | `body` | 机器人本体坐标系 |
| `enable_rviz` | `false` | 是否同时启动 RViz |

## 7. 坐标系与 TF 树

### 7.1 完整 TF 树（导航栈运行时）

```
map                              ← 全局地图坐标系（由 transform_fusion 发布）
│
├── camera_init                  ← FAST-LIO 里程计原点（由 transform_fusion 动态 TF 发布）
│   │
│   └── body                     ← FAST-LIO 跟踪的机器人本体（由 FAST-LIO /Odometry + TF 发布）
│       │
│       └── livox_frame          ← Livox 雷达坐标系（静态 TF，identity，由 kuavo_localize.launch
│                                    和 elevation_mapping.launch 双重发布）
│
└── robot_foot_init              ← （静态 TF，identity）
    └── odom                     ← 机器人 URDF 里程计根（静态 TF，identity）
        └── base_link            ← URDF 根连杆（由 robot_state_publisher + EKF 发布）
            └── (所有关节连杆)
```

### 7.2 关键坐标系说明

| 坐标系 | 发布者 | 类型 | 说明 |
|--------|--------|------|------|
| `map` | transform_fusion.py | dynamic TF | 全局固定坐标系，由全局定位与地图对齐 |
| `camera_init` | transform_fusion.py | dynamic TF | FAST-LIO 里程计原点，首次启动时的 body 位置 |
| `body` | FAST-LIO | dynamic TF | 机器人本体参考系（FAST-LIO 跟踪对象） |
| `livox_frame` | static_transform_publisher | static TF | LiDAR 坐标系，与 body 共原点（identity） |
| `odom` | static_transform_publisher | static TF | URDF 里程计帧 |
| `base_link` | robot_state_publisher | dynamic TF | URDF 定义的机器人根连杆 |

### 7.3 elevation_mapping 的 TF 查找链

elevation_mapping 需要将点云从 `livox_frame` 变换到 `map` 坐标系：

```
livox_frame ──(static, identity)──▶ body ──(FAST-LIO TF)──▶ camera_init ──(transform_fusion TF)──▶ map
```

**两个 `body → livox_frame` 发布者**：
1. `kuavo_localize.launch` 中的 `tf_pub_1`（导航栈内）
2. `elevation_mapping.launch` 中的 `tf_elevation_robot_to_livox`（自包含）

二者都发布 identity 变换，不冲突。

## 8. 话题与数据流

### 8.1 输入话题

| 话题 | 类型 | 频率 | 发布者 | 说明 |
|------|------|------|--------|------|
| `/livox/lidar` | `livox_ros_driver2/CustomMsg` | 10 Hz | `livox_lidar_publisher2` | Livox 原始自定义点云 |
| `/livox/imu` | `sensor_msgs/Imu` | 10 Hz | `livox_lidar_publisher2` | Livox 内置 IMU |
| `/livox/pointcloud2` | `sensor_msgs/PointCloud2` | 10 Hz | `livox2pointcloud` | 转换后的标准点云 |
| `/cloud_registered_body` | `sensor_msgs/PointCloud2` | 10 Hz | FAST-LIO (`laserMapping`) | body 系配准点云（已去畸变） |
| `/Odometry` | `nav_msgs/Odometry` | 10 Hz | FAST-LIO | camera_init → body 里程计 |
| `/localization` | `nav_msgs/Odometry` | varies | `transform_fusion.py` | map → body 全局定位 |

### 8.2 输出话题（elevation_mapping 发布）

| 话题 | 类型 | 频率 | 说明 |
|------|------|------|------|
| `/elevation_mapping/elevation_map_raw` | `grid_map_msgs/GridMap` | 5 Hz | 原始高程图（含所有图层） |
| `/elevation_mapping/elevation_map` | `grid_map_msgs/GridMap` | 5 Hz | 融合后高程图 |
| `/elevation_mapping/pose` | `geometry_msgs/PoseWithCovarianceStamped` | 实时 | 机器人当前位姿 |
| `/elevation_mapping/visibility_cleanup_map` | `grid_map_msgs/GridMap` | 1 Hz | 可见性清理图 |

### 8.3 GridMap 图层说明

| 图层名 | 说明 |
|--------|------|
| `elevation` | 高程值（m），主要显示图层 |
| `variance` | 高程方差 |
| `horizontal_variance_x` | 水平方差 X |
| `horizontal_variance_y` | 水平方差 Y |
| `horizontal_variance_xy` | 水平方差 XY |
| `color` | 颜色 |
| `time` | 时间戳 |
| `dynamic_time` | 动态时间 |
| `lowest_scan_point` | 最低扫描点 |
| `sensor_x/y/z_at_lowest_scan` | 最低扫描点处传感器坐标 |

## 9. 常见问题排查

### 9.1 看不到高程图 - 完整排查流程

```bash
# Step 1: 检查 elevation_mapping 是否已编译
ls ~/kuavo_ros_application/src/navigation_opensource/devel/lib/elevation_mapping/
# 应有 elevation_mapping 可执行文件

# Step 2: 检查 grid_map_msgs 是否可用
rosmsg show grid_map_msgs/GridMap
# 应正常显示消息定义

# Step 3: 检查工作空间 source
echo $ROS_PACKAGE_PATH | grep kuavo_nav_demo
# 应有结果（包含 navigation_opensource 路径）

# Step 4: 检查 elevation_mapping 节点是否运行
rosnode list | grep elevation_mapping
# 应有 /elevation_mapping

# Step 5: 检查点云是否到达（按当前模式选择话题）
rostopic hz /livox/pointcloud2          # 模式 A：应 ~10 Hz
rostopic hz /cloud_registered_body      # 模式 D：应 ~10 Hz

# Step 6: 检查点云有无数据（如果没有，启动转换）
rosnode list | grep livox2pointcloud
# 如果没有，执行: roslaunch livox2pointcloud livox2pointcloud.launch

# Step 7: 检查高程图话题
rostopic hz /elevation_mapping/elevation_map_raw
# 应有 5Hz 左右的输出

# Step 8: 检查高程图是否有实际数值（非全 NaN）
rostopic echo /elevation_mapping/elevation_map_raw -n 1 | grep -E "nan|-?[0-9]+\.[0-9]+" | head -20
# 如果全为 nan，说明没有点云到达或 TF 查找失败

# Step 9: 检查 TF 树
rosrun tf view_frames
# 确认 map → body → livox_frame 链完整
```

### 9.2 常见问题

**Q: 编译时 OOM？**
```bash
catkin build elevation_mapping grid_map kindr kindr_ros message_logger -j2
```

**Q: `roslaunch` 找不到 `kuavo_nav_demo` 包？**
```bash
source ~/kuavo_ros_application/src/navigation_opensource/devel/setup.bash
```

**Q: 高程图全为 NaN？**
- 检查 `/livox/pointcloud2` 是否有数据（启动 `livox2pointcloud`）
- 检查 TF 链 `map → camera_init → body → livox_frame` 是否完整
- 检查 `ignore_points_above`/`ignore_points_below` 是否过滤了所有点

**Q: RViz 中找不到 GridMap 显示类型？**
- RViz 所在终端需 source `navigation_opensource` 工作空间
- 确认 `grid_map_rviz_plugin` 已编译

**Q: 高程图位置不对/偏移？**
- 检查机器人位姿话题是否正常
- 检查 `map_frame_id` 和 `robot_base_frame_id` 是否匹配当前 TF 树

### 9.3 模式 D 启动后 RViz 无信号（实测案例）

**现象**：
```bash
roslaunch kuavo_nav_demo elevation_mapping.launch \
    pointcloud_topic:=/cloud_registered_body enable_rviz:=true

rostopic hz /cloud_registered_body
# → no new messages

rviz -d .../rviz/localization.rviz
# → 看不到高程图
```

**根因与对策**：

| 现象 | 根因 | 对策 |
|------|------|------|
| `/cloud_registered_body` 无数据 | 未运行 `kuavo_nav_test.launch`，FAST-LIO 未启动 | 先起导航栈，确认 `rostopic hz /cloud_registered_body` ~10 Hz |
| RViz 无任何高程图 | 使用了 `localization.rviz`，不含 GridMap 显示 | 手动 Add → GridMap，Topic 选 `/elevation_mapping/elevation_map_raw` |
| 节点日志仍显示订阅 `/livox/pointcloud2` | 使用了旧版 launch（param 在 rosparam load 之前被覆盖） | 重新 `catkin build kuavo_nav_demo` 并 source，当前 launch 已在 load 之后覆盖 topic |
| `enable_rviz:=true` 无效 | `rviz/elevation_mapping.rviz` 文件不存在 | 手动配置 RViz（见 §6.3） |
| 只运行了机器人控制栈 | TF 树为 `odom → base_link`，无 `map`/`body` | 必须运行完整导航感知栈 |

**正确启动顺序（模式 D）**：
```bash
# 1. source（每个终端）
source ~/kuavo_ros_application/devel/setup.bash
source ~/kuavo_ros_application/src/navigation_opensource/devel/setup.bash

# 2. 导航栈
roslaunch kuavo_nav_demo kuavo_nav_test.launch

# 3. 确认前置话题
rostopic hz /cloud_registered_body        # ~10 Hz ✅
rostopic hz /localization                 # 有数据 ✅

# 4. 高程图
roslaunch kuavo_nav_demo elevation_mapping.launch \
    pointcloud_topic:=/cloud_registered_body

# 5. 确认输出后再开 RViz
rostopic hz /elevation_mapping/elevation_map_raw   # ~5 Hz ✅
rviz   # 手动 Add GridMap
```

### 9.4 调试命令速查

## 10. 已知局限与后续工作

### 10.1 当前局限

| 局限 | 说明 | 影响 |
|------|------|------|
| 局部窗口 4m×4m | 只能看到脚下区域 | 足端规划够用，路径规划需结合全局地图 |
| 依赖完整导航栈 | 需要 FAST-LIO + transform_fusion 提供完整 TF 链 | 启动步骤较多 |
| 雷达噪声模型为经验值 | `beam_angle: 0.003` 等参数基于 Mid-360 经验 | 可能需要根据实际场景微调 |
| 无 RViz 配置文件 | `enable_rviz` 引用的 rviz 文件不存在 | 需手动配置 RViz |
| OOM 风险 | grid_map 等包编译时内存占用高 | 需加 `-j2` 编译 |

### 10.2 后续工作建议

- [ ] 创建 `rviz/elevation_mapping.rviz` 配置文件（高优先级，`enable_rviz:=true` 依赖此文件）
- [ ] 将 `livox2pointcloud.launch` 集成到 `kuavo_nav_test.launch`，避免模式 A 漏启动转换节点
- [ ] 将 elevation_mapping.launch 集成到 `kuavo_nav_test.launch` 中一键启动
- [ ] 根据实际使用场景优化 `ignore_points_above`/`ignore_points_below` 参数
- [ ] 添加高程图 → 足端碰撞检测节点
- [ ] 将 `source navigation_opensource/devel/setup.bash` 添加到用户的 `~/.bashrc`

## 附录 A：`elevation_mapping.launch` 完整源码

```xml
<launch>
  <!--
    ETH elevation_mapping for Kuavo.

    Provides real-time local terrain elevation map (4m×4m, 4cm resolution)
    from Livox Mid-360 point clouds.

    ── Prerequisites ──────────────────────────────────────────────
    The full nav stack (kuavo_nav_test.launch) provides:
      - Livox point cloud:    /livox/pointcloud2   (frame_id: livox_frame)
      - Global localization:  /localization        (nav_msgs/Odometry, map→body)
      - TF tree:              map → camera_init → body → livox_frame

    Minimum requirement (elevation_mapping can run standalone):
      - Any odometry source publishing nav_msgs/Odometry
      - Livox driver publishing /livox/pointcloud2
      - Static TF: <robot_frame> → livox_frame (auto-published by this launch)

    ── Quick Start ────────────────────────────────────────────────
    # A) On top of full nav stack (default):
    roslaunch kuavo_nav_demo elevation_mapping.launch

    # B) With FAST-LIO odometry (no global localization):
    roslaunch kuavo_nav_demo elevation_mapping.launch odom_topic:=/Odometry map_frame:=camera_init

    # C) With EKF odometry (no FAST-LIO, no global localization):
    roslaunch kuavo_nav_demo elevation_mapping.launch odom_topic:=/odometry/filtered map_frame:=odom robot_frame:=base_link

    # D) Using FAST-LIO registered point cloud (body frame, deskewed):
    roslaunch kuavo_nav_demo elevation_mapping.launch pointcloud_topic:=/cloud_registered_body
  -->

  <arg name="odom_topic" default="/localization" />
  <arg name="pointcloud_topic" default="/livox/pointcloud2" />
  <arg name="map_frame" default="map" />
  <arg name="robot_frame" default="body" />
  <arg name="enable_rviz" default="false" />

  <!-- Static TF: robot_frame → livox_frame (identity) -->
  <node pkg="tf2_ros" type="static_transform_publisher"
        name="tf_elevation_robot_to_livox"
        args="0 0 0 0 0 0 $(arg robot_frame) livox_frame"
        if="$(eval arg('robot_frame') != 'livox_frame')" />

  <!-- Odometry: Odometry → PoseWithCovarianceStamped -->
  <node pkg="kuavo_nav_demo" type="odometry_to_pose_with_covariance.py"
        name="odom_to_pose_with_covariance" output="screen">
    <param name="odom_topic" value="$(arg odom_topic)" />
    <remap from="pose" to="/elevation_mapping/pose" />
  </node>

  <!-- Elevation Mapping node -->
  <node pkg="elevation_mapping" type="elevation_mapping" name="elevation_mapping" output="screen">
    <!-- Load YAML configs first -->
    <rosparam command="load" file="$(find kuavo_nav_demo)/config/robots/kuavo.yaml" />
    <rosparam command="load" file="$(find kuavo_nav_demo)/config/elevation_maps/map.yaml" />
    <rosparam command="load" file="$(find kuavo_nav_demo)/config/sensor_processors/livox.yaml"
              ns="input_sources/lidar" />

    <!-- THEN set frame IDs + topic from launch args (overrides YAML) -->
    <param name="map_frame_id" value="$(arg map_frame)" />
    <param name="robot_base_frame_id" value="$(arg robot_frame)" />
    <param name="track_point_frame_id" value="$(arg robot_frame)" />
    <param name="input_sources/lidar/topic" value="$(arg pointcloud_topic)" />
  </node>

  <group if="$(arg enable_rviz)">
    <node pkg="rviz" type="rviz" name="elevation_mapping_rviz" output="screen"
          args="-d $(find kuavo_nav_demo)/rviz/elevation_mapping.rviz" />
  </group>
</launch>
```
